Staff-Level Insight
The four-domain split is deliberate: safety-critical control loops stay on-device (Jetson), queue routing stays warehouse-local (sub-10ms decisions), bulk ML training lives in cloud (GPU elastic scaling), and operator endpoints connect via WebRTC peer-to-peer with TURN relay fallback. This prevents any single domain failure from cascading.

Staff-Level Insight
Shared Control (robot proposes, human confirms) is the highest-value mode for data collection — the robot's proposal becomes a training signal whether accepted or corrected. Queue-aware thresholds can nudge borderline cases toward this mode when operators are idle.

Staff-Level Insight — Epistemic-Only Escalation
The key innovation: only escalate on high epistemic uncertainty (model hasn't seen this before — human can help). High aleatoric uncertainty (inherent sensor noise) means humans can't resolve it either. This preserves operator capacity for genuinely novel situations and avoids wasting human time on noisy-but-fundamentally-random outcomes.

Batch Matching Algorithm — Operator Assignment

batch_matching_algorithm.arch 
    BIPARTITE BATCH MATCHING — HUNGARIAN ALGORITHM (every 2-5 seconds)

    Step 1: Collect pending requests and available operators over window

    PENDING REQUESTS                          AVAILABLE OPERATORS

    ┌──────────────────────┐                   ┌──────────────────────┐
    │ R3:  P0, skill=grasp │                   │ Op1: grasp=95,nav=60 │
    │      SLA: 3s remain  │                   │      load: 1 active  │
    ├──────────────────────┤                   ├──────────────────────┤
    │ R7:  P1, skill=class │                   │ Op2: class=90, QA=80 │
    │      SLA: 22s remain │                   │      load: 0 active  │
    ├──────────────────────┤                   ├──────────────────────┤
    │ R12: P2, skill=nav   │                   │ Op3: nav=85, all=70  │
    │      SLA: 2m remain  │                   │      load: 2 active  │
    └──────────────────────┘                   └──────────────────────┘

    Step 2: Compute cost matrix

                       Op1          Op2          Op3
                 ┌────────────┬────────────┬────────────┐
        R3:grasp │   0.08     │   0.82     │   0.55     │   cost = f(skill_gap,
                 ├────────────┼────────────┼────────────┤       sla_urgency,
        R7:class │   0.71     │   0.12     │   0.48     │       operator_load,
                 ├────────────┼────────────┼────────────┤       recency_bonus)
        R12:nav  │   0.65     │   0.73     │   0.22     │
                 └────────────┴────────────┴────────────┘

    Step 3: Solve minimum-cost bipartite matching (Hungarian, O(n³))

        R3  ═══════▸ Op1   (cost 0.08, best grasp skill, SLA-critical)
        R7  ═══════▸ Op2   (cost 0.12, best classify skill, idle)
        R12 ═══════▸ Op3   (cost 0.22, best nav skill, despite load)

    Total cost: 0.42  (vs greedy: 0.42 same here, but 50% better at scale)

    Cost function breakdown:

    cost(request_i, operator_j) =
        w₁ · (1 - skill_match(req.skill, op.skills))      // 0-1, skill gap
      + w₂ · (1 - remaining_sla / max_sla)                // 0-1, urgency
      + w₃ · (op.active_tasks / op.max_concurrent)        // 0-1, load
      + w₄ · (1 - recency_bonus(op, req.robot_id))        // 0-1, familiarity

    Typical weights: w₁=0.35, w₂=0.30, w₃=0.20, w₄=0.15

Staff-Level Design Tradeoffs

tradeoff_matching.md
BATCH (Hungarian)                       GREEDY (First Available)

+ 50% lower wait times at scale         + Simpler implementation
+ Globally optimal assignment           + Zero batch delay
+ Fair workload distribution            + Works with 1 operator

- 2-5s batch window adds delay          - Suboptimal skill matching
- O(n³) complexity per batch            - Head-of-line blocking
- Need ≥3 operators to benefit          - Operator fatigue imbalance

VERDICT: Use greedy for <5 operators, batch for 5+.
Hybrid: greedy for P0 (bypass batch), batch for P1-P3.
tradeoff_uncertainty.md
EPISTEMIC-ONLY ESCALATION               ALL-UNCERTAINTY ESCALATION

+ Only escalates when humans             + Simpler to implement
  CAN actually help                      + Catches all low-conf cases
+ Preserves operator capacity            + No MC Dropout overhead
+ Better data quality (novel cases)

- Requires MC Dropout (10 passes)        - Wastes operator time on noise
- ~10× more compute for uncertainty      - Humans can't fix aleatoric
- Harder to explain to operators         - Floods queue during bad lighting

VERDICT: Epistemic-only for mature systems. Start with all-uncertainty,
add decomposition when you have enough data to calibrate.
tradeoff_thresholds.md
DYNAMIC (Queue-Aware)                    STATIC THRESHOLDS

+ Adapts to operational load             + Predictable behavior
+ Maximizes data collection              + Easy to audit/explain
  when operators are idle                + Consistent error rate
+ Self-regulates throughput
+ Prevents queue overflow               - Can't adapt to surges
                                          - Wastes idle operators
- Risk of oscillation                    - Misses data collection
- Harder to guarantee SLAs                 opportunities
- Need dampening / hysteresis

VERDICT: Dynamic with guardrails. Static P0 threshold (never adjustable),
dynamic P2-P3 thresholds with ±5% max swing and 60s cooldown.
tradeoff_streaming.md
WebRTC                    WebTransport              SRT

+ Universal browser       + Simpler server          + Lowest latency
+ P2P capable             + Unreliable datagrams    + FEC built-in
+ Mature ecosystem       + QUIC multiplexing       + Cellular-optimized
+ NAT traversal           + No STUN/TURN needed

- Complex (ICE/STUN)      - No iOS/Safari           - No browser support
- SFU costs at scale     - Experimental            - Custom client needed
                          - No media negotiation    - No P2P

VERDICT: WebRTC today (proven, cross-platform). Plan WebTransport
migration for controlled operator workstations. SRT for dedicated
cellular links only.
tradeoff_safety_locality.md
CRITICAL DESIGN PRINCIPLE: Safety enforcement NEVER leaves the robot.

Why not cloud-side safety?

Scenario: Network drops for 2 seconds during operator teleop.

Cloud-side safety:
  Cloud doesn't receive heartbeat → Cloud sends e-stop → NETWORK IS DOWN
