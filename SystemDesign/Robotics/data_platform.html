Robotics Data Platform - System Design

A comprehensive system design for ingesting, processing, cataloging, and selecting robotics training data at scale. This document covers the full architecture from customer data upload to model training experiments.

💡 Key Insight

In robotics ML, data selection often matters more than model architecture. This platform treats data curation as a first-class, tunable hyperparameter rather than a fixed preprocessing step.

What This System Does

  • Ingest - Accept robot demonstration data from multiple customers in various formats (MCAP, ROS Bags, HDF5)
  • Process - Synchronize multi-rate sensors, normalize actions to canonical space, segment into episodes
  • Catalog - Store all episodes with rich metadata (labels, quality scores, embeddings) for queryable selection
  • Select - Use declarative "recipes" to dynamically select training datasets based on quality, diversity, and task criteria
  • Experiment - Integrate with MLflow/Optuna to optimize both model hyperparameters AND data selection jointly

Business Context

Why This Matters

Foundation model companies for robotics (like TorqueAGI, Skild AI, Physical Intelligence) need to:

  1. Onboard diverse customers - Each customer has different robots, formats, action spaces
  2. Pre-train on cross-embodiment data - Leverage Open X-Embodiment's 1M+ trajectories
  3. Fine-tune on customer-specific data - 100-500 demonstrations per task
  4. Iterate rapidly - Data strategy is a competitive advantage
⚠️ Market Reality

Only ~5-10 companies globally need cross-robot ingestion at scale. This is internal infrastructure, not a product. The addressable market is too small for a standalone offering, but it's a critical operational moat for horizontal players.

Requirements

Functional Requirements

  • Accept robot data in MCAP, ROS1 Bag, HDF5, and CSV+MP4 formats
  • Support arbitrary robot kinematics via URDF
  • Normalize actions to canonical 8-dimensional space
  • Segment continuous recordings into discrete episodes
  • Label episodes (success/failure) via heuristics and VLM
  • Score episode quality across multiple dimensions
  • Compute embeddings for diversity-based sampling
  • Support declarative recipe-based dataset selection
  • Track experiments with full lineage (recipe → dataset → model → metrics)

class BaseAdapter(ABC): """All format adapters implement this interface""" @abstractmethod def read(self, file_path: Path) -> Iterator[RawFrame]: """Yield raw frames from file""" pass @abstractmethod def get_channels(self, file_path: Path) -> List[ChannelInfo]: """List available channels/topics in file""" pass @abstractmethod def get_time_range(self, file_path: Path) -> Tuple[float, float]: """Return (start_time, end_time) of recording""" pass class MCAPAdapter(BaseAdapter): """MCAP format adapter using mcap library""" def read(self, file_path: Path) -> Iterator[RawFrame]: with open(file_path, "rb") as f: reader = mcap.reader.make_reader(f) for schema, channel, message in reader.iter_messages(): yield RawFrame( timestamp=message.log_time / 1e9, channel=channel.topic, msg_type=schema.name, data=message.data )

💡 Design Decision: MCAP as Container

MCAP is a container format (like MP4 for video), not a data format. It provides O(1) seeking, per-channel compression, and schema preservation. It's ideal for the first layer of ingestion before converting to training formats.

Robot Manifest

The manifest is the key to handling diverse robots without code changes. It answers: "How do I READ this robot's data?"

┌─────────────────────────────────────────────────────────────────────────────────┐ │ MANIFEST vs PIPELINE CONFIG │ ├─────────────────────────────────────────────────────────────────────────────────┤ │ │ │ MANIFEST (per robot, static) PIPELINE CONFIG (per run, tunable) │ │ ───────────────────────────── ───────────────────────────────── │ │ │ │ • Schema mapping (topic names) • Target Hz (output frequency) │ │ • Robot kinematics (URDF, joints) • Sync strategy (interpolation) │ │ • Sensor config (rates, intrinsics) • Segmentation method │ │ • Action space (control mode, dims) • Quality thresholds │ │ • Gripper config (type, range) • Output format (LeRobot, RLDS) │ │ • Coordinate frames • Compression settings │ │ │ │ Answers: "How do I READ?" Manifest Components

  • Robot definition - URDF path, joint names, limits, base/EE frames
  • Schema mapping - Topic name → field extraction for each data type
  • Camera config - Intrinsics, extrinsics, encoding, expected FPS
  • Action space - Control mode (joint position, EE pose, velocity), dimensionality
  • Gripper config - Type, raw range, normalization target
  • Expected rates - For validation and sync strategy selection

Deep Dive: Processing Layer

Synchronization

The synchronization module aligns multi-rate sensor streams to a common timeline.

┌─────────────────────────────────────────────────────────────────────────────────┐ │ MULTI-RATE SYNCHRONIZATION │ ├─────────────────────────────────────────────────────────────────────────────────┤ │ │ │ Input (raw rates): Output (synchronized @ 30 Hz): │ │ │ │ joints @ 1000 Hz ┌─────────────────────────┐ │ │ ││││││││││││││││││││││││ │ All streams @ 30 Hz │ │ │ │ │ │ │ camera @ 30 Hz ─────────▶ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ │ Aligned to camera │ │ │ gripper @ 10 Hz │ timestamps │ │ │ │ │ │ └─────────────────────────┘ │ │ │ ├─────────────────────────────────────────────────────────────────────────────────┤ │ │ │ SYNC STRATEGIES: │ │ │ │ 1. nearest_neighbor - Use closest reading within max_drift_ms │ │ + Fast, simple │ │ - Potential jitter │ │ │ │ 2. interpolate - Linear interpolation between readings │ │ + Smooth values │ │ - Can hallucinate values in gaps │ │ │ │ 3. camera_trigger - Emit frame only when camera fires │ │ + Perfect image alignment │ │ - Depends on camera timing │ │ │ └─────────────────────────────────────────────────────────────────────────────────┘

Sync Algorithm Details

▼
def synchronize_streams( streams: Dict[str, Iterator[TimestampedData]], target_hz: int = 30, strategy: str = "nearest_neighbor", max_drift_ms: float = 50 ) -> Iterator[SyncedFrame]: """ Synchronize multiple data streams to common timeline. Args: streams: Dict mapping stream name to iterator of timestamped data target_hz: Output frequency strategy: One of "nearest_neighbor", "interpolate", "camera_trigger" max_drift_ms: Maximum allowed timestamp drift Yields: SyncedFrame with all streams aligned to same timestamp """ target_dt = 1.0 / target_hz buffers = {name: deque(maxlen=100) for name in streams} # Fill initial buffers for name, stream in streams.items(): for _ in range(10): buffers[name].append(next(stream)) # Find common start time start_time = max(buf[0].timestamp for buf in buffers.values()) current_time = start_time while True: synced_data = {} for name, buf in buffers.items(): if strategy == "nearest_neighbor": nearest = min(buf, key=lambda x: abs(x.timestamp - current_time)) if abs(nearest.timestamp - current_time) > max_drift_ms / 1000: raise SyncError(f"Drift exceeded for {name}") synced_data[name] = nearest.data elif strategy == "interpolate": # Find bracketing samples before = max((x for x in buf if x.timestamp <= current_time), key=lambda x: x.timestamp) after = min((x for x in buf if x.timestamp >= current_time), key=lambda x: x.timestamp) alpha = (current_time - before.timestamp) / (after.timestamp - before.timestamp) synced_data[name] = lerp(before.data, after.data, alpha) yield SyncedFrame(timestamp=current_time, data=synced_data) current_time += target_dt

🚨 Critical: Forward Kinematics

When normalizing joint-space robots to EE delta space, you need accurate forward kinematics. This requires the URDF from the manifest. Any mismatch between the URDF and actual robot causes systematic errors in the training data.

Example: Joint Position to EE Delta

def normalize_joint_to_ee_delta( joint_positions: List[np.ndarray], # Sequence of (7,) joint configs urdf_path: str, base_frame: str, ee_frame: str ) -> List[np.ndarray]: """Convert joint position sequence to EE delta sequence.""" # Load robot kinematics robot = urdfpy.URDF.load(urdf_path) ee_poses = [] for q in joint_positions: # Forward kinematics fk = robot.link_fk(cfg={joint.name: q[i] for i, joint in enumerate(robot.joints)}) ee_pose = fk[ee_frame] # 4x4 homogeneous transform ee_poses.append(ee_pose) # Compute deltas actions = [] for i in range(1, len(ee_poses)): prev_pose = ee_poses[i-1] curr_pose = ee_poses[i] # Position delta delta_pos = curr_pose[:3, 3] - prev_pose[:3, 3] # Rotation delta (as euler) prev_rot = Rotation.from_matrix(prev_pose[:3, :3]) curr_rot = Rotation.from_matrix(curr_pose[:3, :3]) delta_rot = (curr_rot * prev_rot.inv()).as_euler('xyz') actions.append(np.concatenate([delta_pos, delta_rot])) return actions

Quality Scoring

def compute_quality_scores(episode: Episode) -> QualityScores: """Compute multi-dimensional quality scores for an episode.""" # Completeness: Are all expected sensors present? expected_streams = {"joints", "gripper", "wrist_cam", "overhead_cam"} present_streams = set(episode.available_streams) completeness = len(present_streams & expected_streams) / len(expected_streams) # Smoothness: Low jerk is better actions = np.array([f.action for f in episode.frames]) jerk = np.diff(actions, n=2, axis=0) jerk_magnitude = np.linalg.norm(jerk, axis=1).mean() smoothness = 1.0 - np.clip(jerk_magnitude / MAX_JERK, 0, 1) # Sync quality: Low timestamp drift max_drift = max(f.sync_drift_ms for f in episode.frames) sync_quality = 1.0 - np.clip(max_drift / 100, 0, 1) # Efficiency: Short path to goal is better path_length = sum(np.linalg.norm(actions[i, :3]) for i in range(len(actions))) optimal_length = np.linalg.norm(episode.goal_position - episode.start_position) efficiency = optimal_length / max(path_length, optimal_length) # Overall: Weighted combination overall = ( 0.30 * completeness + 0.25 * smoothness + 0.25 * sync_quality + 0.20 * efficiency ) return QualityScores( overall=overall, completeness=completeness, smoothness=smoothness, sync_quality=sync_quality, efficiency=efficiency )

Embedding for Diversity

def compute_embeddings(episode: Episode) -> EpisodeEmbeddings: """Compute embeddings for diversity-based sampling.""" # Initial state embedding: What does the scene look like? first_frame_image = episode.frames[0].observation.images["overhead"] initial_state_emb = clip_model.encode_image(first_frame_image) # (256,) # Trajectory embedding: What motion pattern is this? action_sequence = np.array([f.action[:6] for f in episode.frames]) # Exclude gripper # Simple approach: PCA on action sequence statistics traj_features = np.concatenate([ action_sequence.mean(axis=0), action_sequence.std(axis=0), action_sequence.max(axis=0), action_sequence.min(axis=0), ]) trajectory_emb = trajectory_encoder(traj_features) # (256,) # Pre-compute cluster assignment for fast diversity sampling cluster_id = kmeans_model.predict(initial_state_emb.reshape(1, -1))[0] return EpisodeEmbeddings( initial_state=initial_state_emb, trajectory=trajectory_emb, cluster_id=cluster_id )

Interview Questions

Q: Why normalize actions to a canonical 8-dim space instead of keeping robot-specific actions?

Cross-robot transfer learning. The foundation model pre-trains on Open X-Embodiment (22+ robot types). If each robot uses different action representations, the model can't transfer knowledge between them. The canonical 8-dim space (Δx, Δy, Δz, Δroll, Δpitch, Δyaw, gripper, terminate) provides a common language that enables:

  • Pre-training on diverse robots with consistent action semantics
  • Fine-tuning on customer data without architectural changes
  • Potential zero-shot transfer to similar tasks

The tradeoff is loss of robot-specific control nuances, but for manipulation tasks, EE pose delta is sufficiently expressive.

Q: Why separate Episode Store (S3) and Episode Catalog (PostgreSQL)?

Different access patterns require different storage solutions.

  • Episode data (video, actions): Large (~10MB/episode), accessed sequentially during training, rarely queried by attributes → S3 is ideal
  • Episode metadata (labels, scores, embeddings): Small (~2KB/episode), frequently queried with complex filters, vector similarity search → PostgreSQL + pgvector is ideal

Combining them would either make queries slow (scanning S3) or storage expensive (duplicating data in DB). The s3_path foreign key links them efficiently.

Q: Why use recipes instead of just querying the database directly?

Reproducibility and experimentation. A recipe is a versioned, declarative specification that captures exactly how a training dataset was selected. Benefits:

  • Reproducibility: Recipe + checksum guarantees identical dataset reconstruction
  • Experimentation: Recipes can be parameters in hyperparameter search (Optuna)
  • Audit trail: Every trained model links to a specific recipe version
  • Collaboration: ML engineers can share recipes, iterate on them like code

Raw SQL queries provide no versioning, no structured sampling methods, and no integration with experiment tracking.

Q: How would you handle a robot that uses a completely different action space (e.g., a mobile manipulator with base movement)?

Extend the canonical space while maintaining backward compatibility.

  1. Define a superset action space: [Δx, Δy, Δz, Δroll, Δpitch, Δyaw, gripper, terminate, base_vx, base_vy, base_ωz]
  2. For arm-only robots, pad with zeros for base actions
  3. For mobile manipulators, populate all fields
  4. Train with attention masking so model learns to ignore irrelevant action dims per robot

This is exactly what Open X-Embodiment does. The key is consistent semantics per dimension, with zeros for unsupported DOFs.
