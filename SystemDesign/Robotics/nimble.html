Requirements & Scale Estimation

Functional Requirements

  1. Ingest orders from Shopify (webhooks) and NetSuite (REST/polling) with idempotent processing
  2. Decompose multi-item orders into robot-executable task DAGs (pick → consolidate → pack → label → ship)
  3. Manage distributed transactions across pick-pack-ship with compensating actions (saga pattern)
  4. Maintain real-time inventory at the bin level with strong write consistency and eventual read consistency (CQRS)
  5. Provide immutable audit trail for every inventory movement (event sourcing)
  6. Orchestrate long-running fulfillment workflows with durable execution (Temporal.io)
  7. Assign tasks to 200+ robots per FC using optimal multi-robot task allocation (MRTA)
  8. Generate shipping labels via multi-carrier rate shopping and write back tracking to sales channels
  9. Handle inbound receiving, QC inspection, and velocity-based putaway
  10. Support partial fulfillment, split shipments, and backorders

Shopify Webhook Security

Every Shopify webhook includes an X-Shopify-Hmac-Sha256 header — the HMAC-SHA256 of the raw request body computed with the app's API secret. The ingestion gateway verifies this before processing. The X-Shopify-Event-Id header provides a unique event identifier for deduplication. Shopify retries failed deliveries up to 8 times over ~48 hours with exponential backoff; 8 consecutive failures auto-delete the subscription.

NetSuite Integration

NetSuite's SuiteTalk REST API exposes sales orders at /services/rest/record/v1/salesOrder. For complex queries, SuiteQL provides SQL-like syntax with JOINs, subqueries, and ORDER BY — capabilities missing from the basic REST record API. Authentication uses OAuth 1.0 TBA (Token-Based Authentication). RESTlets offer custom server-side logic for specialized patterns.

Staff-Level InsightWebhooks should never be the sole source of truth. The reconciliation cron catches missed deliveries, ordering issues, and de-registered subscriptions. This is a common failure mode that most candidates miss.

Cartonization (3D Bin Packing)

Selecting the optimal box size is NP-hard. The First Fit Decreasing (FFD) heuristic sorts items largest-first, tries each available box size, and picks the smallest that fits all items. FFD achieves ≤ 11/9·OPT + 6/9 approximation ratio. Modern systems like Paccurate add cost-aware cartonization considering carrier DIM weight thresholds and zone-based pricing.

Waveless vs Wave Picking

Waveless (continuous) picking is strongly preferred for robotic warehouses. Unlike wave picking (batching orders into time-bounded groups), waveless continuously releases tasks and assigns robots in real-time via the MRTA auctioneer. This eliminates inter-wave idle time and maximizes robot utilization.

Staff-Level Signal: Forward RecoveryPrefer forward recovery over full compensation. If a pick fails at Bin A-12, query for alternate bin locations and create a new pick task before resorting to full rollback. ~1 in 22 checkouts enters a failure path — this is steady-state, not edge-case.

Partial Fulfillment — Parent/Child Sagas

When only some items are available, the parent saga spawns child sagas per shipment. Business rules are configurable per brand: ship available items and backorder the rest, hold for complete fulfillment with timeout, or cancel unavailable lines.

Deep Dive — CQRS Inventory Architecture

Inventory exhibits extreme read/write asymmetry: ~1M+ reads/sec for availability vs ~10K writes/sec for reservations. CQRS separates the command side (strong consistency) from the query side (eventually consistent materialized views).

TWO-PHASE RESERVATION
available = on_hand - soft_reserved - hard_reserved
Soft reserve: tentative, TTL 15min, auto-expire if not confirmed
Hard reserve: committed at pick start, subtracted from available

Redis DECR — The Atomic Inventory Guard

Redis DECR is a single atomic operation that decrements a key and returns the new value. Redis is single-threaded — all commands serialize at the server level. No locks needed. If DECR inventory:SKU-A:available returns ≥ 0, the reservation succeeded. If negative (oversold), immediately INCR back and return out-of-stock. ~0.1ms per operation vs ~5-15ms for PostgreSQL SELECT ... FOR UPDATE.

The Hybrid Approach

Redis as "fast gate" (handles burst, rejects oversells in sub-ms), PostgreSQL as "durable commit" (optimistic lock, event store append). Brief consistency window where Redis has decremented but PG hasn't committed — mitigated by TTL-based soft reservations and periodic reconciliation.

Point-in-time queries are the killer feature: "What was inventory of SKU-X at warehouse Y on December 15 at 3pm?" — answered by replaying events up to that timestamp. Impossible with traditional CRUD.

Tamper-proof logging uses cryptographic hash chaining — each event includes the SHA-256 hash of the previous event. Any modification breaks the chain. Combined with WORM storage (S3 Object Lock), this satisfies FDA 21 CFR Part 11 audit trail requirements.

Deep Dive — MRTA Auction Algorithm

Multi-Robot Task Allocation treats task assignment as a market: each robot bids on tasks, and the fleet manager awards to the lowest-cost bidder.

mrta_auction.archMID LEVEL
AUCTION-BASED MRTA — per pick task

  Task arrives: "Pick 2× SKU-A from Bin A-12"
       │
       ▼ broadcast to all idle robots
  ┌─────────────────────────────────────────────────────┐
  │  BID FORMULA (per robot)                              │
  │                                                       │
  │  bid = w1 × distance_to_bin                            │
  │       + w2 × (1 / battery_pct)                        │
  │       + w3 × gripper_compat_penalty                    │
  │       + w4 × current_load_factor                       │
  │       + w5 × zone_affinity_penalty                     │
  │                                                       │
  │  zone_penalty = BASE × (1 - overflow_ratio)           │
  │  // when zone is overloaded, penalty shrinks          │
  │  // → cross-zone bids become affordable               │
  └─────────────────────────────────────────────────────┘

  R-042  dist:12m bat:78% grip:OK  load:0 zone:A → bid: 6.7 ★ WINNER
  R-017  dist:34m bat:92% grip:OK  load:1 zone:B → bid: 14.9
  R-088  dist:8m  bat:22% grip:OK  load:0 zone:A → bid: 28.4 (low battery)
  R-063  dist:45m bat:85% grip:FAIL load:2 zone:C → bid: 999 (disqualified)

RUNS ON EDGE (fog cluster), NOT CLOUD
Reason: needs sub-10ms access to robot state (in-memory, 100ms refresh)
Cloud round-trip (40-80ms) would make bids stale and collision avoidance dangerous

Zone affinity uses soft penalties rather than hard assignments. Each robot has a home zone that biases its bid downward. When a zone is overloaded (more pending tasks than idle robots), the penalty shrinks, automatically pulling in help from neighboring zones. Steady state: ~80% intra-zone, 20% cross-zone.

Key Architectural Tradeoffs

Orchestration vs Choreography Saga

Chose: Orchestration (Temporal). Fulfillment is strictly sequential with physical dependencies. Central workflow provides full visibility ("where is this order?"), simplifies debugging (single execution history), and handles complex compensation logic. Choreography would scatter state across 6+ services.

Waveless vs Wave Picking

Chose: Waveless (continuous). Robotic fleets are expensive — idle time is waste. Waveless continuously assigns tasks via MRTA, eliminating inter-wave idle. Support both modes: waveless for e-commerce, wave for B2B wholesale with scheduled carrier departures.

Redis DECR + PG vs PG-Only Locking

Chose: Hybrid. Redis handles 100K+ concurrent checkouts per hot SKU in sub-ms. PG-only optimistic locking fails at ~500 reservations/min/SKU due to version conflicts. Tradeoff: brief Redis-PG divergence window, mitigated by TTL-based soft reservations and reconciliation cron.

Temporal.io vs AWS Step Functions

Chose: Temporal.io. Nimble already uses it. Full programming language vs JSON. Unlimited duration. Native Signals for scan events. Heartbeating for physical picks. ~80% cost reduction self-hosted on EKS. Step Functions wins only for simple, low-volume AWS-native workflows.

Event Sourcing Everywhere vs Selective

Chose: Selective. Event sourcing for inventory (audit, point-in-time, reconciliation) and order lifecycle (chain of custody). NOT for robot telemetry (ScyllaDB time-series) or user sessions. Apply where audit/replay value justifies the complexity.

MRTA on Edge vs Cloud

Chose: Edge (fog cluster). MRTA needs sub-10ms access to robot state (in-memory, 100ms refresh). Cloud round-trip 40-80ms makes bids stale. Edge provides offline resilience (30 min autonomy). MRTA is tightly coupled to path planning, which must also be local.

Hard Zone Assignment vs Free Roaming vs Soft Affinity

Chose: Soft zone affinity. Hard assignment (60-70% utilization, SPOF on zone overload). Free roaming (congestion, long travel). Soft affinity (85-92% utilization, demand-driven rebalancing, zone penalty shrinks when overloaded).
