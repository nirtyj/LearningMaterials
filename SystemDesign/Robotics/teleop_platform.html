System Design Study Guide Robotics Teleoperation Platform

End-to-end design for a multi-warehouse, multi-operator remote assistance platform. Informed by production architectures at Waymo, Cruise, Zoox, Nimble Robotics, Phantom Auto, Ottopia, DriveU.auto, Boston Dynamics, and academic research from CoRL, IEEE, and TUM.

Problem Statement

Design a platform where warehouse robots request human judgment on perception tasks (pick setpoint marking, drop confirmation). Operators are in Southeast Asia, robots in North America. Normal operation is 8 seconds — every millisecond of teleoperation latency directly impacts throughput. The platform must handle assignment, session lifecycle, real-time streaming, and graceful degradation.

Teleoperation Taxonomy

The industry distinguishes three modes of teleoperation, each with fundamentally different latency requirements and system architectures. Understanding which mode your system implements is the first thing to clarify in an interview.

Mode Latency Req Who Drives Industry Examples
Direct Remote Driving <100ms glass-to-glass Human has full vehicle control (steering, throttle, brake) Vay, Phantom Auto, Nuro (low-speed)
Supervisory Assistance <500ms E2E Robot drives; human provides waypoints, path guidance, or object classification Waymo, Cruise, Zoox, Aurora
Exception-Based Intervention <2s E2E (per-task) Robot operates autonomously 90-95%; human resolves perception failures Nimble Robotics, Covariant, Cognicept
Interview Insight

Our target system is exception-based intervention — the robot asks for help only when its vision model fails. This is NOT direct remote driving. The distinction matters because it means: (1) we don't need continuous video streaming during normal operation, (2) the latency budget is per-request not per-frame, and (3) we can tolerate higher assignment latency since the robot is safely stopped while waiting.

Latency Budget Analysis READ FIRST

Interview Lead with this. Decomposing the latency budget should be your opening frame in any teleoperation system design interview. It forces every downstream decision to be justified against a concrete number.

The 8-Second Constraint

Normal pick operation takes 8 seconds. When intervention is needed, the robot is paused. Every second of intervention latency is a second of lost throughput. If average intervention takes 10 seconds and 5% of picks need intervention, that's a 6.25% throughput loss. Our goal: keep total intervention time under 5 seconds (2.5% throughput loss at 5% intervention rate).

TOTAL INTERVENTION BUDGET: ≤5,000ms
─────────────────────────────────────────────────────────────

Phase 1: Assignment         ≤500ms
  ├─ Request reaches cloud        ~50ms  (edge→cloud)
  ├─ Assignment decision           ~20ms  (priority queue lookup)
  ├─ Notification to operator      ~80ms  (cloud→SEA WebSocket)
  └─ Operator accepts              ~350ms (UI render + click)

Phase 2: Session Setup       ≤800ms
  ├─ WebRTC signaling              ~200ms (SDP offer/answer via cloud)
  ├─ ICE connectivity check        ~300ms (STUN/TURN)
  ├─ First keyframe delivery       ~200ms (I-frame at session start)
  └─ UI context loading            ~100ms (robot state, task context)

Phase 3: Human Decision      ≤2,500ms
  ├─ Operator orients to scene     ~1,000ms (visual comprehension)
  ├─ Operator marks setpoint       ~1,000ms (mouse click on image)
  └─ Confirmation / review         ~500ms

Phase 4: Command Delivery    ≤400ms
  ├─ Command encode + transmit     ~150ms (SEA→cloud→edge)
  ├─ Edge→robot WiFi               ~20ms
  ├─ Robot processes command        ~30ms  (validates, plans)
  └─ Margin                        ~200ms

Total worst-case: ~4,200ms → within 5s budget ✓
Key Insight from Research

Network transmission is only ~10% of total glass-to-glass latency (2025 research finding). The real bottlenecks are: camera capture/exposure (5-33ms), encoding (5-30ms), decoding (2-10ms), and human reaction time (1-3s). This is why Phantom Auto, Ottopia, and DriveU invest heavily in codec optimization, not just network improvements. arXiv 2602.17381

Latency Optimization Strategies

StrategyLatency SavedHow
Pre-warm WebRTC connections ~500ms from Phase 2 Maintain idle DataChannels between edge servers and operator pool. On assignment, promote to active — skip ICE negotiation entirely.
Snapshot-first, stream-second ~300ms from Phase 2 Send a high-res JPEG snapshot with the assignment notification. Operator can start deciding while video stream establishes.
Predictive assignment ~200ms from Phase 1 ML model predicts which robots are about to need help (low confidence scores trending down). Pre-assign operator before request fires.
Edge-local video encoding ~15ms from Phase 2 Encode video on edge server (NVIDIA T4/L4) instead of robot Jetson. Better encoder, lower latency.
Operator UI pre-loading ~100ms from Phase 2 When operator pool is warm, pre-load robot context (SKU info, recent failures) in background tabs.

Architecture Rationale

Why not direct robot-to-operator? Three reasons: (1) robots are behind NAT on warehouse WiFi — you need a relay, (2) operators need to be assigned dynamically — you need a broker, (3) you need audit trails and access control — you need a control plane. The cloud sits in the middle as orchestrator, but the data path can be optimized to minimize hops.

Why edge servers? Warehouse internet is limited bandwidth. The edge server aggregates robot streams, transcodes/compresses, buffers during outages, and provides local recording. Without it, 50 robots each pushing 5 Mbps would require 250 Mbps uplink — unrealistic for warehouse internet.

Key design decisions:

Why Zenoh over DDS for robot↔edge? Standard ROS 2 DDS middleware suffers significant performance degradation over WiFi due to IP fragmentation from large UDP payloads. Zenoh uses TCP with built-in fragmentation handling, achieving 3-10x better throughput over lossy wireless links. arXiv 2508.11366

Why H.264 on-robot? Jetson Orin has hardware H.264/H.265 encoders (NVENC). Encoding on-robot reduces WiFi bandwidth by 10-50x vs raw frames. Use ultrafast preset + zerolatency tune for minimum encode latency (~5ms).

Edge Server Architecture & Failure Modes

The edge server is the most critical single point of failure in the warehouse. It must be designed for high availability.

Edge Failure Scenarios

Failure 1: Edge loses internet (WAN down)
  → Robots continue autonomous operation
  → Intervention requests buffer locally (SQLite queue)
  → When connectivity restores, drain queue LIFO
  → If buffer > 5min, robots fall back to skip-and-continue

Failure 2: Edge server crashes (hardware)
  → Robots detect via heartbeat timeout (5s)
  → Robot falls back to DIRECT CLOUD mode:
    • Robot has pre-provisioned cloud endpoint
    • Connects via warehouse internet directly
    • Higher latency but still functional
  → Alert ops team for hardware replacement

Failure 3: WiFi degradation
  → Edge monitors per-robot link quality (RSSI, packet loss)
  → Adaptive: reduce video quality (720p→480p→JPEG snapshots)
  → If link drops entirely, robot stops and waits
  → Consider: mesh WiFi (Rajant) for large warehouses

Industry reference: Boston Dynamics' Spot supports WiFi, LTE, 5G, Rajant mesh, and Starlink as connectivity options through its Orbit platform. Boston Dynamics Blog

Deep Dive: Control Plane

Assignment Service

Priority Scoring, Fairness, and Elastic Scaling

Industry The operator scheduling problem is NP-Hard (proven by Patel et al., IEEE IROS 2022). In practice, greedy priority-queue approaches work well. The Fleet-DAgger framework from CoRL 2022 formalizes the Return on Human Effort (ROHE) metric for scheduling. Fleet-DAgger Paper

Priority Score Formula

For each eligible operator op and request req:

score(op, req) =
    0.30 × availability(op)           // 1.0 if idle, 0.5 if queue_depth=1, 0.0 if full
  + 0.25 × success_rate(op, req.type) // rolling 100-request window
  + 0.20 × speed(op, req.type)        // 1.0 - (avg_time / max_time)
  + 0.15 × fairness(op)               // time_since_last_request / max_idle_time
  + 0.10 × familiarity(op, req.wh_id) // requests_at_warehouse / total_requests

Elastic Scaling Strategy

Operator pool scales based on request queue depth and time-of-day patterns:

Tier 1: Cross-warehouse rebalancing
  If warehouse_A queue_depth > threshold AND warehouse_B idle operators exist:
    Expand operator ACLs temporarily to cover both warehouses
    (requires pre-training on warehouse_A layout)

Tier 2: Shift extension
  If global queue_depth > 2× baseline for 5min:
    Alert shift managers, request overtime operators

Tier 3: Degraded mode
  If operator pool exhausted:
    Robots fall back to skip-and-requeue behavior
    Non-critical requests deprioritized
    Critical path items (high-value SKUs) prioritized

Session Manager

State Machine, TTL, Reconnection, Episode Curation

The session manager is the orchestrator that binds an operator to a robot for the duration of a single intervention. It owns the state machine shown in the Low-Level Architecture section.

Implementation: Temporal.io Workflow

Each session is a Temporal workflow with durable timers, automatic retries, and audit trails. Why Temporal over a custom state machine? Because session state must survive service restarts, and Temporal gives you exactly-once execution semantics and built-in timeout handling.

SessionWorkflow (Temporal)
───────────────────────────────
workflow SessionWorkflow(request):
  // Phase 1: Assignment
  operator = await activity.AssignOperator(request)
  if operator is null:
    return SessionResult.NO_OPERATOR_AVAILABLE

  // Phase 2: Wait for acceptance (30s timeout)
  accepted = await signal.WaitForAcceptance(timeout=30s)
  if not accepted:
    await activity.ReassignToNext(request)
    continue  // retry with next candidate

  // Phase 3: WebRTC session setup
  session = await activity.EstablishWebRTC(operator, request.edge_id)
  timer.Start("session_ttl", duration=120s)

  // Phase 4: Wait for command or timeout
  select:
    case command = await signal.OperatorCommand():
      await activity.DeliverCommand(command, request.robot_id)
      await activity.WriteEpisode(session, command)
      return SessionResult.COMPLETED

    case await signal.OperatorDisconnect():
      reconnected = await signal.WaitForReconnect(timeout=15s)
      if reconnected: goto Phase 4
      else: await activity.ReassignToNext(request)

    case await timer.Fired("session_ttl"):
      await activity.TimeoutSession(session)
      return SessionResult.TIMEOUT

Freshness Validation

Pattern Frame-timestamped commands — a critical safety mechanism. Every command references the video frame the operator was looking at when they made their decision. If the command arrives at the robot and the referenced frame is older than 500ms, the robot drops it and requests a fresh decision.

// On robot side:
func validateCommand(cmd Command) bool {
  currentTime := now()
  frameAge := currentTime - cmd.FrameTimestamp
  if frameAge > 500ms {
    log.Warn("Stale command", "age", frameAge)
    return false  // reject — operator saw old data
  }
  return true
}

Industry reference: Transitive Robotics implements this exact pattern — control commands are timestamped against the last received video frame, and stale commands are dropped. Transitive Robotics

Queue Management: Max 3 Concurrent

Each operator has a queue depth of max 3. This is based on CMU research showing that open queues (operators freely choose) achieve the best resolution rates. The operator sees all pending requests for their authorized warehouses and can accept in any order — not forced FIFO.

Inspection Mode (Requirement C.2)

Operators with VIEWER or OPERATOR role can connect to any authorized robot for read-only inspection without an active assistance request.

INSPECTION MODE — ON-DEMAND STREAMING
─────────────────────────────────────────────────

  Key design: streams are NOT always-on.
  Bandwidth budget: 50 robots × 5 Mbps = 250 Mbps
  Warehouse uplink: ~50-100 Mbps
  → Cannot afford always-on streaming for all robots

  Solution: Tiered approach
  Level 1 (always): Telemetry heartbeats only (50 bytes/5s)
  Level 2 (dashboard): Periodic JPEG thumbnails (1 per 10s, ~50KB each)
  Level 3 (inspect): Full video stream on-demand (operator clicks "Inspect")

  Inspection flow:
  1. Operator clicks robot in Manage tab
  2. Cloud sends START_STREAM to edge for that robot
  3. Edge begins forwarding robot's camera feed to SFU
  4. WebRTC subscription established to operator
  5. Operator gets read-only view (no command channel)
  6. When operator closes, STOP_STREAM sent → edge stops forwarding

User Flow: Assistance Request

The primary flow — a robot needs help and an operator provides it.

1
Robot detects perception failure

Vision model confidence drops below threshold (e.g., 0.3). Robot transitions to REQUESTING_HELP state. Captures high-res snapshot. Publishes request to /teleop/request topic on Zenoh.

2
Edge server receives and forwards

Edge's Robot Communication Layer receives the request. Uploads snapshot to S3 (pre-signed URL). Forwards enriched request to cloud via WebSocket: {robot_id, warehouse_id, request_type, priority, snapshot_url, context}.

3
Assignment Service matches operator

Filters eligible operators (ACL + availability + queue depth). Scores candidates. Atomically assigns top candidate in Redis. Pushes assignment notification via WebSocket to operator's browser.

4
Operator sees request + snapshot

Browser receives assignment with snapshot URL. Renders snapshot immediately (operator can start forming intent). Audio chime alerts operator. Request appears in queue tab with [ACCEPT] button.

5
Operator accepts → WebRTC session starts

Operator clicks ACCEPT. Session Manager initiates WebRTC signaling. SDP offer/answer exchange via cloud signaling server. ICE connectivity established (STUN first, TURN fallback). First I-frame delivered — live video begins.

6
Operator makes decision

Operator views live video feed + context sidebar. Clicks on image to set pick point, or selects action (retry/skip/flag). Command created with frame_id and timestamp for freshness validation.

7
Command delivered to robot

Command sent via WebRTC DataChannel → cloud → edge → robot. Robot validates freshness (frame age < 500ms). Robot executes command (moves to setpoint, retries pick, etc.). Robot ACKs completion.

8
Session closes, episode recorded

Session Manager transitions to COMPLETING. Episode MCAP file finalized on edge. Metrics emitted (intervention duration, decision time). Episode uploaded to S3 for ML training pipeline. Robot resumes autonomous operation.

User Flow: Inspection Mode

1
Operator opens Manage Robots tab

Sees list of authorized warehouses and robots. Each robot shows: status (online/offline/assisting), battery level, last heartbeat time. Data comes from telemetry heartbeats (no video streaming yet).

2
Operator clicks [INSPECT] on a robot

Cloud sends START_STREAM command to edge server for that robot. Edge begins encoding and forwarding robot's camera feed to SFU.

3
Read-only video stream established

WebRTC subscription established between SFU and operator browser. Operator sees live camera feed + telemetry overlay. No command channel — inspection is strictly read-only.

4
Operator closes inspection

Cloud sends STOP_STREAM to edge. Edge stops forwarding. Bandwidth freed.

User Flow: Failover Scenarios

Operator Disconnects Mid-Session
1
WebRTC connection drops

Browser detects via RTCPeerConnection.oniceconnectionstatechange → "disconnected".

2
Session enters RECONNECTING (15s grace)

Session Manager starts reconnect timer. If ICE restart succeeds within 15s, session resumes from ACTIVE state.

3
Grace period expires → reassign

Assignment Service picks next eligible operator. New operator receives full context (snapshot + previous operator's partial actions if any). Robot remains in WAITING_INPUT state throughout.

Edge Server Loses Internet
1
Cloud detects via heartbeat timeout (10s)

Edge's WebSocket to cloud drops. Cloud marks all robots at that warehouse as EDGE_DISCONNECTED.

2
Edge buffers requests locally

New intervention requests from robots are queued in SQLite on edge. Edge continues local MCAP recording.

3
Connectivity restores

Edge reconnects WebSocket. Buffered requests drained LIFO (most recent first — older requests may be stale). Active sessions that were interrupted attempt reconnection.

4
Extended outage (>5 min)

Robots fall back to skip-and-continue for non-critical items. Critical items queued for manual handling. Alert sent to warehouse ops team.

Edge Server Hardware Failure
1
Robots detect edge down via heartbeat (5s)

Zenoh connection to edge fails.

2
Robots switch to DIRECT CLOUD mode

Each robot has pre-provisioned cloud endpoint (TURN server credentials baked into firmware). Robot connects directly via warehouse internet. Higher latency (~100ms more) but functional.

3
Bandwidth throttling

50 robots × 5 Mbps = 250 Mbps would saturate warehouse internet. Cloud-side admission control limits to 5 concurrent streams. Remaining robots fall back to snapshot-based intervention.

Reliability & Fault Tolerance

Design Principle

Graceful degradation hierarchy: Full teleoperation → reduced quality → keyframes only → JPEG snapshots → local autonomy → safe stop. As connectivity degrades, the system progressively reduces reliance on the network. The robot must always have a safe local fallback. This mirrors BSI Flex 1886's requirement that autonomous systems must be able to perform a Minimal Risk Manoeuvre when connectivity is lost. DriveU Network Redundancy

Traps to Avoid

Common Mistakes

1. Designing for direct remote driving. The problem says "mark a pick setpoint on an image" — that's supervisory, not driving. Don't over-engineer for sub-100ms latency you don't need.

2. Ignoring the bandwidth constraint. "Warehouse internet is limited" is in the problem statement. If you propose always-on video for 50 robots, you're immediately at 250 Mbps — way too much. Must design for on-demand streaming.

3. Skipping the latency budget. The single biggest differentiator at L6. Open with it.

4. No reliability discussion. The problem explicitly lists three failure scenarios. Name them and sketch mitigations.

5. Auction-based assignment. Adds latency and complexity without clear benefit over priority scoring for this use case. Auctions make sense when operators bid on prices — not when you're optimizing for speed.

Staff-Level Q&A

Q: Why WebRTC and not a custom protocol?

A: For exception-based intervention with a ~5s budget, WebRTC's 200-400ms glass-to-glass is well within our latency envelope. The benefit is zero-plugin browser support, built-in DTLS-SRTP encryption, NAT traversal, and adaptive congestion control — all out of the box. Phantom Auto, Ottopia, and DriveU built custom protocols because they target direct remote driving at <100ms, which WebRTC can't reliably achieve. We don't need that. The pragmatic engineering choice is WebRTC with SFU for fleet scale, and invest our engineering effort in the harder problems: assignment optimization, edge reliability, and the ML data flywheel.

Q: How do you handle the bandwidth constraint at the warehouse?

A: Three-layer approach. First, on-demand streaming only — no robot streams video until an intervention is requested or an operator opens inspection mode. Second, edge-local encoding with adaptive bitrate — the edge server uses NVENC to compress streams and dynamically adjusts quality based on available WAN bandwidth. Third, snapshot-first pattern — when a request arrives, we send a JPEG snapshot (~100KB) immediately and start the video stream in parallel. The operator can begin forming intent from the snapshot while the stream starts. At 5% intervention rate with 50 robots, we average ~1.5 concurrent streams per warehouse — well within even a 50 Mbps uplink.

Q: What happens when the edge server goes down?

A: Robots detect via heartbeat timeout (5s) and switch to direct cloud mode. Each robot has pre-provisioned TURN server credentials. The problem: 50 robots trying to stream directly would saturate the WAN. Solution: cloud-side admission control limits to ~5 concurrent video streams. Remaining robots fall back to snapshot-based intervention (upload JPEG, get command back). For HA, deploy dual edge servers with leader election (etcd). But honestly, a single edge with hardware redundancy (RAID, dual PSU, dual NIC) and 4-hour replacement SLA is more cost-effective for a warehouse deployment.

Q: How do you ensure operators aren't making decisions on stale video?

A: Frame-timestamped commands. Every command the operator sends references the video frame they were looking at. The robot validates that the referenced frame is less than 500ms old. If the command is stale (e.g., network hiccup delayed delivery), the robot drops it and the operator is prompted to re-confirm. This is the same pattern used by Transitive Robotics in their production teleoperation system. Additionally, the operator UI shows real-time latency and bandwidth indicators — if latency exceeds a threshold, the UI shows a "STALE" warning overlay on the video.

Q: How does the system get smarter over time?

A: Every intervention session is recorded as an MCAP episode containing video, telemetry, and the operator's command as ground truth. This feeds an imitation learning pipeline — the operator's setpoint annotation becomes a training label for the vision model. Over time, the model learns to handle cases it previously failed on, reducing the intervention rate. Nimble Robotics uses exactly this flywheel to push toward a fully autonomous "dark warehouse." Covariant achieved <0.1% intervention rate through cloud-aggregated fleet learning across continents. The key architectural support is: (1) MCAP recording at the edge with S3 offload, (2) structured episode metadata for filtering/querying, (3) Airflow pipeline for batch retraining, (4) canary deployment of updated models to subset of robots.

Q: Why not always-on video for monitoring?

A: Math kills it. 50 robots × 5 Mbps = 250 Mbps from a single warehouse — that's datacenter-grade bandwidth, and the problem says warehouse internet is limited. Instead, we use tiered monitoring: heartbeat telemetry (always on, 50 bytes/5s), periodic JPEG thumbnails (1/10s, ~50KB each = 250 KB/s total for 50 robots), and full video on-demand when an operator opens inspection mode. This is the same pattern DroneDeploy uses — streams start when a client requests them and stop when the client disconnects.
