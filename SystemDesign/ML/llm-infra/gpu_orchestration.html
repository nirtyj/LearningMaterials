Building a GPU Cluster Orchestration Platform from Scratch

Staff-level system design — from bare metal to production-scale training & inference

System Overview

This platform solves a deceptively complex problem: given a pool of bare-metal GPU servers, build a service that accepts training jobs (identify idle clusters, schedule distributed training) and inference workloads (deploy models on separate clusters with autoscaling). Every component must handle continuous hardware failures — at 10K+ GPUs, something breaks every 1-3 hours.

Core Insight: At scale, hardware failures are not exceptions but continuous background noise. Meta experienced 419 failures in 54 days across 16K H100s — one every ~3 hours. Every design decision must account for this reality.

The fundamental compute unit is a GPU server containing 8 GPUs interconnected via NVSwitch, with multiple high-speed NICs for inter-node communication, local NVMe storage, and dual CPUs for host operations.

Mid-Level: Intra-Node GPU Interconnect

Inside a server, NVSwitch creates a full-mesh between all 8 GPUs. Every GPU can talk to every other GPU at the full NVLink bandwidth — no hop penalty, no congestion from shared links. This is why tensor parallelism (the most communication-intensive) stays intra-node.

Volcano Deep Dive

High-Level: What is Volcano?

Volcano is a batch scheduling system for Kubernetes built specifically for HPC, AI/ML, and big data workloads. It replaces the default kube-scheduler with a GPU-aware scheduler that understands gang scheduling, fair-share queues, and topology. It's the most widely deployed GPU scheduler in production K8s clusters.

Mid-Level: Volcano Scheduling Cycle (Actions + Plugins)

Each scheduling cycle is a Session. Within a session, Volcano runs a pipeline of Actions, and each action consults Plugins for scoring, filtering, and ordering decisions. This is fundamentally different from kube-scheduler's per-pod approach — Volcano reasons about groups of pods simultaneously.

Mid Level 
   Volcano Scheduling Session (one cycle)
   ═══════════════════════════════════════

   ┌─────────────────────────────────────────────────────────────────┐
   │  SESSION START (triggered every 1 second by default)            │
   │                                                                  │
   │  Input state:                                                    │
   │  ├─ Pending PodGroups (from all Queues)                          │
   │  ├─ Node status (GPU capacity, allocatable, taints)              │
   │  └─ Running jobs (current allocations)                           │
   │                                                                  │
   │  ═══════════════════════════════════════════════════════════════  │
   │                                                                  │
   │  ACTION 1: ENQUEUE                                               │
   │  ┌──────────────────────────────────────────────────────────┐   │
   │  │  For each Queue, sort PodGroups by priority               │   │
   │  │  Move "ready" PodGroups to scheduling pipeline            │   │
   │  │                                                           │   │
   │  │  Plugins consulted:                                       │   │
   │  │  ├─ proportion: check queue hasn't exceeded its share     │   │
   │  │  └─ overcommit: allow queue to exceed share if others idle│   │
   │  └──────────────────────────────────────────────────────────┘   │
   │                          │                                       │
   │                          ▼                                       │
   │  ACTION 2: ALLOCATE                                              │
   │  ┌──────────────────────────────────────────────────────────┐   │
   │  │  For each PodGroup (highest priority first):              │   │
   │  │                                                           │   │

   │  │  ┌─────────────────────────────────────────────────────┐  │   │
   │  │  │ Plugin: GANG                                         │  │   │
   │  │  │ if podgroup.spec.minMember > schedulable_count:      │  │   │

   │  │  │ else:                                                │  │   │
   │  │  │   proceed to node selection                          │  │   │
   │  │  └─────────────────────────────────────────────────────┘  │   │
   │  │                                                           │   │
   │  │  Step 2b: For each pod in group, select best node        │   │

   │  │                                                           │   │
   │  │  Step 2c: BIND all pods atomically                       │   │

   │  │  │ → kubelet starts containers on all nodes             │  │   │
   │  │  │ → PodGroup status → "Running"                        │  │   │
   │  │  └─────────────────────────────────────────────────────┘  │   │
   │  └──────────────────────────────────────────────────────────┘   │
   │                          │                                       │
   │                          ▼                                       │
   │  ACTION 3: PREEMPT                                               │
   │  ┌──────────────────────────────────────────────────────────┐   │
   │  │  If high-priority PodGroup couldn't be allocated:         │   │
   │  │  ├─ Find lowest-priority running PodGroups                │   │
   │  │  ├─ Check if evicting them frees enough GPUs              │   │
   │  │  ├─ Evict ENTIRE PodGroup (gang-aware preemption)         │   │
   │  │  └─ Re-try allocation for high-priority job               │   │
   │  │                                                           │   │
   │  │  Plugin: GANG (ensures whole group evicted, not partial)  │   │
   │  │  Plugin: PRIORITY (determines eviction order)             │   │
   │  └──────────────────────────────────────────────────────────┘   │
   │                          │                                       │
   │                          ▼                                       │
   │  ACTION 4: RECLAIM                                               │
   │  ┌──────────────────────────────────────────────────────────┐   │
   │  │  If a Queue is under its guaranteed share:                │   │
   │  │  ├─ Find PodGroups in OTHER queues exceeding their share  │   │
   │  │  ├─ Preempt "borrowing" jobs to return resources          │   │
   │  │  └─ Queue gets back its guaranteed allocation             │   │
   │  │                                                           │   │
   │  │  Plugin: PROPORTION (calculates per-queue fair share)     │   │
   │  └──────────────────────────────────────────────────────────┘   │
   │                          │                                       │
   │                          ▼                                       │
   │  ACTION 5: BACKFILL                                              │
   │  ┌──────────────────────────────────────────────────────────┐   │
   │  │  Fill remaining gaps with small jobs that:                │   │
   │  │  ├─ Have lower priority                                   │   │
   │  │  ├─ Don't block pending large gang jobs                   │   │
   │  │  └─ Can start immediately without gang constraints        │   │
   │  │                                                           │   │
   │  │  Critical for utilization: without backfill, gaps left    │   │
   │  │  by gang scheduling waste 10-15% of cluster GPUs          │   │
   │  └──────────────────────────────────────────────────────────┘   │
   │                                                                  │
   │  SESSION END → repeat in 1 second                                │
   └─────────────────────────────────────────────────────────────────┘

Kueue Deep Dive

High-Level: Kueue vs Volcano — Different Philosophies

Kueue is a job queueing system that works WITH the default kube-scheduler rather than replacing it. It controls admission — deciding WHEN a job enters the cluster — while leaving pod-to-node binding to kube-scheduler. Volcano replaces the scheduler entirely. Kueue is now the recommended approach for GKE and is maturing rapidly.

   Kueue Admission Algorithm (per scheduling cycle)

   ┌─────────────────────────────────────────────────────────────┐

   │  first):                                                       │
   │                                                                │

   │                                                                │

   │  │    │  (within own quota)                                │   │

   │  │    ├─ borrowingLimit allows borrowing from cohort       │   │
   │  │    │  AND cohort has idle capacity                      │   │
   │  │    └─ If no flavor fits → Workload is "Inadmissible"   │   │
   │  └────────────────────────────────────────────────────────┘   │
   │                         │                                      │
   │                         ▼                                      │

   │                         │                                      │

   Bubble = 3/11 = 27.3%

   ├─ Increase micro-batches (m >> p)

   └─ Llama 3 custom schedule: removed m % PP == 0 constraint

NCCL Communication

Mid Level 
   NCCL Ring All-Reduce (4 GPUs simplified)
   ═════════════════════════════════════════

   Phase 1: Reduce-Scatter (each GPU gets 1/4 of the final sum)

   GPU 0: [A0|A1|A2|A3]  GPU 1: [B0|B1|B2|B3]
   GPU 2: [C0|C1|C2|C3]  GPU 3: [D0|D1|D2|D3]

   Step 1: GPU 0→1 sends A3, GPU 1→2 sends B0, GPU 2→3 sends C1, GPU 3→0 sends D2
   Step 2: Accumulated chunks rotate...
   Step 3: After 3 steps, each GPU has 1/4 of the total sum

   GPU 0: [  |  |  |S3]  GPU 1: [S0|  |  |  ]
   GPU 2: [  |S1|  |  ]  GPU 3: [  |  |S2|  ]
   (S = sum of all A+B+C+D for that chunk)

   Phase 2: All-Gather (broadcast the reduced chunks)

   3 more steps → every GPU has the complete [S0|S1|S2|S3]

   Total data transferred per GPU: 2 × (N-1)/N × data_size
   For N=8 GPUs, 1GB gradient: 2 × 7/8 × 1GB = 1.75 GB per GPU
   At 900 GB/s NVLink: ~2ms

   Algorithm selection by NCCL:
   ┌─────────────────────────────────────────────────────┐
   │  Message Size    │  Algorithm        │  Why          │
   │  < 256 KB        │  Tree (LL)        │  Low latency  │
   │  256 KB - 4 MB   │  Tree (Simple)    │  Log(N) hops  │
   │  > 4 MB          │  Ring (Simple)    │  BW-optimal   │
   │  > 4 MB + NVLink │  NVLink SHARP     │  In-switch     │
   └─────────────────────────────────────────────────────┘

Eight Design Principles

Summary 
   ┌──────────────────────────────────────────────────────────────┐
   │  1. TREAT FAILURES AS CONTINUOUS                             │
   │     MTTF at 10K GPUs = hours. Every component must be       │
   │     failure-aware by default, not as an exception handler.   │
   │                                                              │
   │  2. TOPOLOGY AWARENESS PERMEATES SCHEDULING                  │
   │     28× BW difference (NVLink vs PCIe) means topology-      │
   │     ignorant placement wastes enormous compute.              │
   │                                                              │
   │  3. SEPARATE TRAINING AND INFERENCE CLUSTERS                 │
   │     Training: throughput + fault tolerance                    │
   │     Inference: latency + autoscaling + multi-tenancy         │
   │                                                              │
   │  4. ASYNC DISTRIBUTED CHECKPOINTING FROM DAY ONE             │
   │     Google's in-memory ckpt: 85% → 97% goodput.             │
   │     Millions of GPU-hours saved at scale.                    │
   │                                                              │
   │  5. USE KUEUE FOR K8S-BASED SCHEDULING                       │
   │     Gang scheduling + cohort fair-sharing + topology         │
   │     awareness + workload-level preemption in one system.     │
   │                                                              │
   │  6. ADOPT DISAGGREGATED PREFILL/DECODE                       │
   │     Now standard across every major serving framework.       │
   │     2-4× goodput improvement.                                │
   │                                                              │
   │  7. INVEST HEAVILY IN HEALTH MONITORING                      │
   │     3-tier system: continuous passive + active checks +      │
   │     deep stress tests with automatic replacement.            │
   │                                                              │
   │  8. PLAN NETWORKING FOR NEXT GENERATION                      │
   │     UEC 1.0 + Spectrum-X + NVLink 5.0 domains are           │
   │     actively changing optimal fabric architecture.           │
   └──────────────────────────────────────────────────────────────┘
The Gap: The difference between naive deployment and production-grade orchestration at 10K+ GPUs is not incremental — it is the difference between 50% and 95% effective training time, representing hundreds of millions of dollars in compute efficiency.
