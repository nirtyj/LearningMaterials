5. Distributed Training Infrastructure

Scaling model training across hundreds of GPUs — data, tensor, and pipeline parallelism, communication primitives, and fault tolerance.

5.1

Why Distribute?

Modern LLMs are too large for a single GPU — in memory, compute, and time.

The Memory Wall

A 70B parameter model requires ~140 GB just to store weights in BF16 (2 bytes per parameter). During training, you also need optimizer states (2× for Adam momentum + variance in FP32 = 8 bytes/param) plus gradients (2 bytes/param in BF16). Total: ~12 bytes/param × 70B ≈ 840 GB. Even the flagship H100 has only 80 GB HBM3.

Memory Breakdown for Training (BF16 weights + FP32 optimizer)

ComponentBytes / Param70B Model405B Model
Weights (BF16)2140 GB810 GB
Gradients (BF16)2140 GB810 GB
Optimizer States (FP32)8560 GB3,240 GB
Total12840 GB4,860 GB

Activations add even more — scaling with batch size and sequence length. For long contexts, activation memory can rival parameter memory.

COMPUTE

Llama 3 405B required ~30.84 million GPU-hours on H100s. At 1 GPU, that is 3,520 years. With 16,384 GPUs running in parallel, it took approximately 54 days.

SOLUTION

Split the model, the data, or both across hundreds or thousands of GPUs. Each parallelism strategy makes different trade-offs between memory, communication, and compute efficiency.

Key Metrics for Distributed Training
  • Throughput — tokens processed per second across all GPUs
  • MFU (Model FLOPs Utilization) — fraction of theoretical peak FLOPs actually used. State-of-the-art systems achieve 40–55% MFU
  • Training Goodput — fraction of wall-clock time spent on useful training vs. overhead (communication, checkpointing, recovery from failures)
  • Scaling Efficiency — speedup achieved vs. linear ideal as you add more GPUs
5.2

Data Parallelism (DP)

The simplest strategy — replicate the model, split the data.

Every GPU holds a complete copy of the model. The global training batch is split evenly across replicas. Each GPU runs forward + backward on its local micro-batch, producing local gradients. An all-reduce operation averages the gradients across all replicas, then every GPU applies the same optimizer step — keeping parameters in sync.

Data Parallelism (4 GPUs) ======================== Global Batch: [D0 | D1 | D2 | D3] | | | | ┌─────┘ | | └─────┐ v v v v ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐ | GPU 0 | | GPU 1 | | GPU 2 | | GPU 3 | | Full | | Full | | Full | | Full | | Model | | Model | | Model | | Model | | Copy | | Copy | | Copy | | Copy | └────┬─────┘ └────┬─────┘ └────┬─────┘ └────┬─────┘ | | | | └──────┬──────┴─────┬──────┘ | | | | v v v ┌─────────────────────────────────────────┐ | ALL-REDUCE GRADIENTS | └─────────────────────────────────────────┘ | | | | v v v v [Synced Params] [Synced] [Synced] [Synced]
Effective Batch Size = per_GPU_batch × num_GPUs
Gradient Accumulation

To simulate larger batch sizes without more GPUs, accumulate gradients over K micro-batches before running all-reduce. This reduces communication frequency by K× while achieving effective_batch = per_GPU × K × num_GPUs. Essential for hitting the optimal batch size when GPU memory is limited.

Limitation

The entire model (weights + optimizer + gradients) must fit in each GPU’s memory. For models beyond ~10B parameters, standard DP is insufficient on 80 GB GPUs. This is exactly the problem ZeRO solves.

5.3

ZeRO Optimizer

Eliminate redundancy in data parallelism — shard optimizer states, gradients, and parameters.

ZeRO (Zero Redundancy Optimizer), developed by Microsoft’s DeepSpeed team, keeps the communication pattern of data parallelism but removes the memory redundancy of replicating everything on every GPU. It operates in three progressive stages:

ZeRO Stages

StageWhat’s ShardedMemory / GPU (70B)Communication Overhead
Baseline (DP)Nothing — full replica~840 GB1× (all-reduce)
ZeRO-1Optimizer states~210 GB (4× reduction)~1×
ZeRO-2+ Gradients~105 GB (8× reduction)~1×
ZeRO-3+ Parameters~840/N GB (linear scaling)~1.5× (all-gather params)
ZeRO-3 / FSDP

At Stage 3, no GPU holds complete parameters. Before each forward/backward, an all-gather collects the needed parameters on-the-fly, then discards them after use. Memory scales linearly: 64 GPUs → each holds only 1/64th of all state.

FSDP (PyTorch Native)

Fully Sharded Data Parallelism is PyTorch’s built-in implementation of ZeRO Stage 3. Tighter integration with PyTorch autograd, simpler API (FullyShardedDataParallel wrapper), and native torch.compile support.

DeepSpeed ZeRO vs PyTorch FSDP
  • DeepSpeed: more features — CPU/NVMe offloading, ZeRO++ (quantized communication, hierarchical partitioning), elastic training, ZeRO-Infinity
  • FSDP: better PyTorch integration, works with torch.compile, simpler debugging, first-class Meta/PyTorch support
  • Trend: FSDP2 (PyTorch 2.x) is closing the gap. Many teams start with FSDP and add DeepSpeed only for advanced offloading features

5.4

Tensor Parallelism (TP)

Split individual layers across GPUs — the Megatron-LM approach.

Instead of replicating entire layers, tensor parallelism partitions each weight matrix across multiple GPUs. Each GPU computes a slice of the output and they synchronize via collective operations. This is the primary intra-node parallelism strategy because it requires high-bandwidth interconnects.

COLUMN-PARALLEL

Split the weight matrix by columns. Input X is broadcast to all GPUs. Each GPU computes Y_i = X @ W_i (partial output). Results are concatenated — no all-reduce needed for this half. Used for the first linear in MLP and Q/K/V projections.

ROW-PARALLEL

Split the weight matrix by rows. Each GPU holds partial input and computes Y_i = X_i @ W_i. An all-reduce sums the partial outputs to produce the full result. Used for the second linear in MLP and output projection.

Tensor Parallelism in a Transformer MLP (2-way TP) =================================================== Input X (all GPUs have full copy) | ┌────┴────┐ v v GPU 0 GPU 1 Column-Parallel (1st Linear) W1_col0 W1_col1 Split weight columns | | GeLU GeLU Activation (no communication) | | W2_row0 W2_row1 Row-Parallel (2nd Linear) | | Split weight rows └────┬────┘ v ALL-REDUCE Sum partial outputs | v Output Y
Attention Heads — Natural Parallelism

Multi-head attention is naturally suited to TP: split heads across GPUs. With 8-way TP on a model with 64 heads, each GPU handles 8 heads. For GQA (e.g., 8 KV heads), the KV heads divide evenly across 8 GPUs — one KV head per GPU.

Communication Requirement

TP requires two all-reduce operations per transformer layer (one for MLP, one for attention). This happens at every layer for every micro-batch — demanding ultra-high bandwidth. That is why TP is used within a single node where GPUs are connected by NVLink (900 GB/s on H100), not across nodes with slower InfiniBand links.

5.5

Pipeline Parallelism (PP)

Split model layers into stages — pipeline micro-batches through them.

Pipeline parallelism assigns contiguous groups of layers to different GPUs (or groups of GPUs). Data flows through the pipeline stage by stage. The key challenge is the pipeline bubble — idle time when stages are waiting for data.

Pipeline Schedules Compared

Reducing the Bubble
  • More micro-batches: increasing m shrinks the bubble fraction. With p=8 and m=64, bubble is ~10%
  • Interleaved schedules: each GPU handles multiple non-contiguous layer groups, reducing effective pipeline depth
  • 1F1B scheduling: once the pipeline fills, alternate forward and backward passes to reduce peak memory and start freeing activations earlier
  • Communication: only point-to-point sends between adjacent stages — far less bandwidth than TP. Works well across nodes
5.6

3D Parallelism

Combining TP + PP + DP to train 100B+ parameter frontier models.

No single parallelism dimension can efficiently scale to thousands of GPUs. 3D parallelism composes all three strategies, placing each on the network topology where it works best: TP on the fastest links, PP across nodes in a rack, and DP across racks.

Why This Layout?
  • TP within node: requires all-reduce every layer → needs NVLink bandwidth (900 GB/s bidirectional)
  • PP across adjacent nodes: only point-to-point sends between stages → moderate bandwidth sufficient
  • DP across racks: all-reduce gradients once per global step → tolerates higher latency
  • This is the standard approach used by Megatron-LM, NeMo, and all frontier model training (GPT-4, Llama 3, Claude, Gemini)

5.7

Sequence Parallelism

For extremely long contexts (128K+), even one sequence’s activations may not fit in memory.

As context windows grow from 4K to 128K to 1M+ tokens, the memory for activations and the KV cache grows quadratically with sequence length (for standard attention) or linearly (for FlashAttention, but still large). Sequence parallelism splits a single sequence across multiple GPUs.

RING ATTENTION

Each GPU holds a chunk of the sequence. During attention, KV blocks are passed around in a ring topology. Each GPU computes attention with its local Q against the current KV block, then forwards KV to the next GPU. After one full ring rotation, every GPU has attended to the full sequence.

ULYSSES (DeepSpeed)

Splits the attention computation along the sequence dimension using all-to-all communication. Each GPU computes full attention heads but only on its sequence chunk. More communication-efficient than Ring Attention for moderate parallelism degrees.

Ring Attention (4 GPUs, sequence split into 4 chunks) ===================================================== GPU 0: Q[0:L/4] GPU 1: Q[L/4:L/2] GPU 2: Q[L/2:3L/4] GPU 3: Q[3L/4:L] KV[0:L/4] KV[L/4:L/2] KV[L/2:3L/4] KV[3L/4:L] Step 1: Each GPU computes local Q × local KV Step 2: Pass KV to next GPU in ring GPU0 → GPU1 → GPU2 → GPU3 → GPU0 Step 3: Compute Q × received KV, accumulate Step 4: Repeat until full rotation (3 more steps) Result: Each GPU has full attention output for its Q chunk
Context Parallelism in Practice
  • Megatron-LM Context Parallelism (CP): combines with TP for efficient long-context training. TP handles intra-layer sharding, CP handles sequence-dimension sharding
  • Enables training on 128K–1M+ token sequences without OOM
  • Communication is overlapped with computation: while computing attention for one KV block, the next is being transferred
  • With 4D parallelism (TP + PP + DP + CP), teams train 128K context models on thousands of GPUs
Trend: As long-context models become standard (Gemini 1M, Claude 200K, GPT-4 128K), sequence parallelism is transitioning from a niche technique to a core component of the training stack.
5.8

Mixed Precision Training

Train in lower precision for speed and memory savings without sacrificing model quality.

Modern GPUs have dedicated hardware for low-precision arithmetic. Using FP16/BF16/FP8 for compute-heavy operations (matrix multiplies) while keeping a master copy of weights in FP32 gives near-full-precision training quality at 2–4× the speed.

Numerical Formats Compared

FormatBitsExponentMantissaRangeTypical Use
FP3232823±3.4 × 1038Master weights, optimizer states
BF161687±3.4 × 1038Default training dtype (same range as FP32)
FP1616510±6.5 × 104Legacy; needs loss scaling
FP8 (E4M3)843±448Forward pass on H100/H200
FP8 (E5M2)852±57,344Backward pass (gradients need range)
BF16 — THE DEFAULT

Brain Float16 has the same exponent range as FP32 (8-bit exponent), so it can represent very large and very small numbers without overflow/underflow. Reduced mantissa (7 bits vs 23) means lower precision, but this is acceptable for training. No loss scaling needed.

FP8 — THE FRONTIER

Available on H100/H200 Tensor Cores. ~2× speedup over BF16 for matrix multiplications. Uses E4M3 for forward pass (more precision) and E5M2 for backward pass (more range for gradients). Requires per-tensor dynamic scaling to avoid overflow.

Mixed Precision Training Flow ============================== Master Weights (FP32) ──cast──> BF16/FP8 Weights | Forward Pass (BF16/FP8) | Loss (FP32) | Backward Pass (BF16/FP8) | Gradients (BF16) | ──cast──> FP32 Gradients | Optimizer Step (FP32) | Update Master Weights (FP32)
Why Not Train Entirely in FP32?

On H100, BF16 Tensor Cores deliver 989 TFLOPS vs 67 TFLOPS for FP32 — a 15× difference. FP8 delivers 1,979 TFLOPS. Memory savings (2× for BF16, 4× for FP8) also allow larger batch sizes. The FP32 master copy is only needed for the optimizer step, which is a small fraction of total compute.

Communication & Hardware

The network fabric that makes distributed training possible.

Distributed training performance is bottlenecked by communication speed between GPUs. The hardware interconnect hierarchy determines which parallelism strategies are feasible at each level.

Collective Operations

All-Reduce: every GPU contributes a tensor, all GPUs receive the sum. Used in DP gradient synchronization.
All-Gather: each GPU has a shard; all GPUs receive the full tensor. Used in ZeRO-3/FSDP to reconstruct parameters.
Reduce-Scatter: reduce + scatter in one operation. Each GPU gets a different reduced shard.

All-to-All: each GPU sends a unique piece to every other GPU. Used in expert parallelism (MoE) and Ulysses sequence parallelism.
Broadcast: one GPU sends to all others.
Point-to-Point: direct send/recv between two GPUs. Used in pipeline parallelism.

NCCL — NVIDIA Collective Communication Library

The de facto standard for multi-GPU communication. NCCL automatically selects optimal algorithms (ring, tree, or direct) based on topology detection. It handles NVLink, NVSwitch, PCIe, and InfiniBand transparently. All major frameworks (PyTorch, DeepSpeed, Megatron-LM) use NCCL under the hood.

Communication-Computation Overlap

Key to high MFU: overlap communication with useful computation. While GPU 0 is computing the forward pass for layer k+1, it can simultaneously all-gather parameters for layer k+2 in the background. FSDP and Megatron-LM support this via CUDA streams — achieving 30–50% reduction in effective communication cost.

5.10

Fault Tolerance

At scale, hardware failures are the norm — not the exception.

A cluster of 16,384 H100 GPUs running for weeks will experience multiple failures per day — GPU hardware faults, ECC memory errors, NVLink degradation, InfiniBand link drops, NIC failures, and node crashes. Without fault tolerance, each failure restarts training from scratch.

Fault Tolerance Strategies

StrategyMechanismRecovery TimeGoodput Impact
Periodic CheckpointingSave full state to shared storage every N steps10–30 min85–90% goodput
Async CheckpointingWrite checkpoint in background while training continues10–30 min90–95% goodput
In-Memory RedundancyReplicate state across peer GPUs for fast recovery1–5 min95–98% goodput
Elastic TrainingAutomatically resize DP dimension when nodes fail/join< 1 min96–99% goodput
CHECKPOINT-BASED

Traditional approach: periodically save model weights, optimizer states, data loader state, and RNG state to distributed storage (S3, FSx Lustre). On failure, replace the bad node and resume from the last checkpoint. Cost: checkpoint write time (minutes for large models) + rollback of lost training steps.

CHECKPOINTLESS RECOVERY

AWS HyperPod approach: maintain redundant copies of state across healthy nodes via peer-to-peer transfer. On failure, a spare node reconstructs state from peers in seconds. No rollback, no storage I/O bottleneck. Achieves >95% training goodput even with frequent failures.

Health Monitoring & Proactive Replacement
  • ECC error tracking: rising correctable ECC errors predict imminent uncorrectable failures — preemptively drain the node
  • GPU temperature monitoring: thermal throttling degrades throughput before causing failure
  • NVLink/InfiniBand CRC errors: link degradation detected via counters before total failure
  • NCCL watchdog: detect hung collective operations (timeout-based) and trigger automatic recovery
  • Training loss spikes: anomalous loss values can indicate silent data corruption from faulty hardware
At Meta’s scale (Llama 3 training), they reported ~466 job interruptions over 54 days of training. Robust fault tolerance was essential for maintaining >90% effective training time.

Training Frameworks

The software stack that orchestrates distributed training.

Several frameworks provide out-of-the-box support for the parallelism strategies described above. Choosing the right one depends on your model scale, hardware, and team expertise.

Framework Comparison

FrameworkKey FeaturesBest ForMaintained By
DeepSpeed ZeRO (Stages 1–3), 3D parallelism, ZeRO-Infinity (NVMe offload), elastic training, MoE support Maximum flexibility; CPU/NVMe offloading; research experimentation Microsoft
Megatron-LM TP/PP/DP, sequence parallelism, context parallelism, optimized kernels Training large models on NVIDIA hardware; highest MFU NVIDIA
FSDP / FSDP2 ZeRO Stage 3, native PyTorch, torch.compile, composable APIs Teams wanting pure PyTorch; simpler debugging; good up to ~70B Meta / PyTorch
NeMo Built on Megatron-LM, data processing pipelines, model hub, fine-tuning recipes End-to-end training workflows; enterprise deployment NVIDIA
Megatron-DeepSpeed Combines Megatron TP/PP with DeepSpeed ZeRO for DP dimension Best of both worlds; used for very large models (100B+) Microsoft + NVIDIA

Decision Guide: Which Framework?

  • Model ≤ 13B, PyTorch shop → FSDP (simplest setup)
  • Model 13B–70B, need offloading → DeepSpeed ZeRO-3
  • Model 70B+, NVIDIA cluster → Megatron-LM (best MFU)
  • Model 100B+, mixed strategies → Megatron-DeepSpeed
  • End-to-end pipeline with data processing → NeMo
  • AWS SageMaker / HyperPod → SMP (SageMaker Model Parallelism) or NeMo

Key Considerations

  • MFU: Megatron-LM achieves highest MFU (~55%) due to hand-optimized CUDA kernels
  • Ease of use: FSDP > DeepSpeed > NeMo > Megatron-LM (from easiest to hardest)
  • Flexibility: DeepSpeed offers most knobs (offloading, quantized comms, elastic)
  • Long context: Megatron-LM context parallelism is most mature
  • Community: FSDP has broadest community; DeepSpeed has most GitHub stars in the space
Emerging: torchtitan & PyTorch Native 4D Parallelism

Meta’s torchtitan project demonstrates native PyTorch 4D parallelism (TP + PP + FSDP + CP) without requiring external frameworks. Built on PyTorch 2.x composable APIs (DTensor, DeviceMesh), it aims to make Megatron-level performance accessible within pure PyTorch. Early results show competitive MFU on Llama 3 training.

Industry Trend: The gap between frameworks is narrowing. PyTorch native capabilities (FSDP2, torch.compile, DTensor) are absorbing features that previously required DeepSpeed or Megatron-LM. Long term, expect convergence toward a single PyTorch-native stack for most use cases, with Megatron-LM remaining dominant for maximum-performance frontier model training.
