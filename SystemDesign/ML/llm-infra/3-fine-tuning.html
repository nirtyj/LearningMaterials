Chapter 3: Supervised Fine-Tuning & PEFT

Teaching a pre-trained LLM how to behave — from full-parameter SFT to parameter-efficient adaptation.

3.1

SFT Overview

Pre-training teaches a model what language is. SFT teaches it how to respond.

Supervised Fine-Tuning (SFT) is the process of adapting a pre-trained base model on labeled instruction–response pairs (or multi-turn conversations) so it produces helpful, well-formatted outputs. It is the bridge between a raw next-token predictor and a useful assistant.

Core Principle: Quality > Quantity

A few thousand high-quality, diverse instruction–response examples can outperform millions of low-quality ones. The LIMA paper showed that just 1,000 carefully curated samples can produce a strong instruction-following model. Focus your effort on data curation, not volume.

When to Use SFT vs CPT vs RAG

TechniquePurposeUse When
SFTBehavioral alignment — format, style, toneModel has the knowledge but responds incorrectly
CPTKnowledge injectionModel lacks domain-specific knowledge
RAGDynamic/changing informationKnowledge changes frequently; no training needed

Typical SFT Recipe

  • Epochs: 1–5 (often 2–3)
  • Learning Rate: 1e-5 to 5e-5
  • Schedule: Cosine decay with linear warmup
  • Warmup: 3–10% of total steps
  • Batch Size: 32–128 (effective, via accumulation)
  • Precision: BF16 mixed-precision
SFT does not add new factual knowledge effectively. If the base model does not already know something, SFT will not reliably teach it — it will just make the model more confidently wrong. Use CPT or RAG for knowledge.

3.2

Full-Parameter SFT

Maximum expressivity — update every weight in the model.

In full-parameter SFT, all model weights are unfrozen and updated via backpropagation. This gives the optimizer maximum freedom to reshape the model's behavior and internal representations, but comes at significant memory and compute cost.

Memory Footprint with AdamW (BF16)

Memory ≈ P × (2 + 2 + 4 + 4 + 4) = P × 16 bytes   (params + grads + optimizer m + v + master weights)

For practical estimates, budget 16–20 bytes per parameter (including activations). A 4B-parameter model requires roughly 64–80 GB of GPU memory for full-parameter SFT with AdamW in BF16.

Risk: Catastrophic Forgetting

Aggressively fine-tuning on a narrow dataset can cause the model to lose general capabilities it learned during pre-training. Mitigation strategies:

  • Mix 5–20% general-purpose data into your SFT dataset
  • Use a low learning rate (1e-5 or lower)
  • Limit to 2–3 epochs — more passes increase forgetting risk
  • Evaluate on held-out general benchmarks throughout training
Risk: Overfitting on Small Datasets

Full-parameter SFT with fewer than ~5K examples often overfits. Symptoms include declining validation loss but increasing repetition and reduced diversity in generations. If your dataset is small, consider LoRA (Section 3.3) instead.

Full-parameter SFT is best suited for scenarios with large, high-quality datasets (10K+ examples) where maximum behavioral change is required, such as a CPT → SFT pipeline for a domain-specific model.
3.3

LoRA & PEFT Methods

Adapt large models by training a tiny fraction of parameters.

Parameter-Efficient Fine-Tuning (PEFT) methods freeze the original pre-trained weights and inject a small number of trainable parameters. The dominant approach is LoRA (Low-Rank Adaptation), which adds low-rank decomposition matrices to attention layers.

LoRA: Low-Rank Adaptation

Freeze all original weights W0. For each target weight matrix, add a trainable low-rank update:

W = W₀ + BA   where  B ∈ ℝd×r, A ∈ ℝr×k,  rank r << min(d, k)

Typical rank values: r = 8, 16, 32, or 64. Applied to Q, K, V, and O projection matrices in attention layers. B is initialized to zeros and A to random Gaussian, so the initial adaptation is zero (no disruption to the base model at start).

Trainable Parameters

  • Original: d × k per layer
  • LoRA: (d × r) + (r × k) per layer
  • Savings: With d=k=4096, r=16: 0.78% of original
  • Typical total: 0.1–1% of all model params

Key Hyperparameters

  • Rank (r): Expressivity of the update (8–64)
  • Alpha (α): Scaling factor, usually α = 2r
  • Target modules: q_proj, v_proj minimum; all linear for best results
  • Dropout: 0.05–0.1 on adapter outputs

QLoRA: Quantized LoRA

Quantize the base model to 4-bit (NF4 data type) and train LoRA adapters in BF16 on top. Uses double quantization and paged optimizers to handle memory spikes. Enables fine-tuning a 65B-parameter model on a single 48 GB GPU.

QLoRA is the go-to method for fine-tuning large models when GPU budget is limited. Quality is within 1–2% of full-precision LoRA for most tasks.

DoRA (Weight-Decomposed LoRA)

Decomposes each weight matrix into magnitude and direction components. Applies LoRA only to the direction component. Consistently outperforms standard LoRA across tasks with minimal additional overhead.

LoRA+ (Differential LR)

Uses different learning rates for the A and B matrices — B gets a higher LR (typically 2–8x). A simple one-line change that yields measurable improvements in convergence speed and final quality.

Other Adapter Variants

  • PREFIX TUNING  Add learnable “virtual tokens” to each layer’s key/value representations. Effective but harder to tune than LoRA.
  • PROMPT TUNING  Learn a set of soft prompt embeddings prepended to the input. Extremely parameter-efficient; best for large models (10B+).
  • IA³  Infused Adapter by Inhibiting and Amplifying Inner Activations — learned scaling vectors on keys, values, and FFN. Fewer params than LoRA.

3.4

Knowledge Distillation

Compress a large teacher model into a smaller, faster student.

Knowledge distillation transfers the learned behavior of a large “teacher” model into a smaller “student” model. Instead of training on hard labels (0 or 1), the student learns from the teacher’s output probability distribution — soft targets that carry far more information about inter-class relationships.

Logit Distillation

The student is trained to minimize the KL divergence between its own output distribution and the teacher’s softened distribution:

L = α · KL(softmax(zT / τ) ‖ softmax(zS / τ)) + (1 − α) · CE(y, zS)

Where τ (temperature) controls distribution softness and α balances distillation loss vs. standard cross-entropy loss on ground-truth labels.

Temperature Effect

  • τ = 1: Standard softmax (sharp, peaky)
  • τ = 2–5: Softer distributions — more information transfer about “dark knowledge”
  • τ > 10: Nearly uniform — diminishing returns
  • Sweet spot: Typically τ = 2–4 for LLMs

Common Distillation Recipes

  • 70B → 7B: Retains 85–92% of teacher quality
  • Pro → Lite: Production-grade compression
  • Ensemble → Single: Combine multiple teachers
  • Typical LR: 2e-5 to 1e-4
Offline Distillation

Pre-generate teacher outputs and cache them. Student trains on saved logits/tokens. Cheaper (teacher runs once), easier to parallelize, but frozen teacher quality.

Online Distillation

Teacher and student train jointly. Teacher generates outputs on-the-fly for each batch. Higher quality (teacher can adapt), but requires both models in memory simultaneously.

Distillation is how most production “small” models are trained. It is more data-efficient than training from scratch because the teacher provides richer supervision signals than hard labels alone.

3.5

SFT Best Practices

Practical techniques that improve fine-tuning quality and efficiency.

NEFTune FREE IMPROVEMENT

Add uniform noise to embedding vectors during fine-tuning: emb = emb + α · U(-1,1) where α scales with sequence length. Improves conversational quality 10–15% on MT-Bench with zero computational overhead. Works because it acts as a regularizer, preventing the model from overfitting to exact token positions.

Sample Packing 2–5x THROUGHPUT

Concatenate multiple short examples into a single sequence (up to max context length) with separator tokens and attention masking to prevent cross-contamination. Eliminates padding waste — typical SFT datasets have highly variable sequence lengths, and naive batching wastes 40–70% of compute on padding tokens.

Naive batching: [Ex1|pad|pad|pad] [Ex2|pad|pad|pad|pad] [Ex3|pad] Sample packing: [Ex1|Ex2|Ex3|Ex4] [Ex5|Ex6|Ex7|Ex8|Ex9] [Ex10|Ex11]

Chat Template Standardization

Use consistent templates with proper special tokens. Common formats: ChatML (<|im_start|>), Llama ([INST]), or provider-specific. Mismatched templates between training and inference cause severe quality degradation.

Loss Masking on Prompts

Only compute loss on assistant response tokens. Mask system prompts, user instructions, and all template tokens. Standard practice — prevents the model from learning to parrot instructions and focuses learning on generation quality.

Curriculum Learning

Order training data from easy to hard for stability. Start with shorter, simpler examples (single-turn, factual QA) and gradually introduce complex multi-turn conversations. Stabilizes early training and can improve final quality by 2–5%.

Gradient Accumulation

Simulate larger effective batch sizes when GPU memory is limited. Accumulate gradients over N micro-batches before performing an optimizer step. effective_batch = micro_batch × accumulation_steps × num_gpus. Allows using optimal batch sizes (32–128) even on a single GPU.

SFT Checklist
  • Enable NEFTune noise (noise_alpha = 5–15)
  • Enable sample packing with proper attention masks
  • Verify chat template matches inference template exactly
  • Enable loss masking on non-assistant tokens
  • Set gradient checkpointing if memory-constrained
  • Log eval metrics every 50–100 steps; watch for overfitting
  • Mix 5–20% general data to preserve base capabilities
3.6

Data Preparation

Garbage in, garbage out — data quality is the single biggest lever in SFT.

Standard JSONL Format

{"messages": [ {"role": "system", "content": "You are a helpful coding assistant."}, {"role": "user", "content": "Write a Python function to reverse a string."}, {"role": "assistant", "content": "def reverse_string(s):\n return s[::-1]"} ]}

One JSON object per line. Multi-turn conversations include alternating user/assistant messages. System message is optional but recommended for consistency.

Quality Filtering Pipeline

StepTechniquePurpose
1. DeduplicationMinHash / exact-match on instructionsRemove redundant examples that waste compute
2. Length FilteringRemove very short (<10 tokens) or very long outliersReduce noise and training instability
3. Format ValidationVerify JSON schema, role alternation, special tokensPrevent training on malformed data
4. Quality ScoringUse a strong LLM to rate response quality (1–5 scale)Filter out low-quality responses
5. Human ReviewSample 200–500 examples for manual auditCatch systematic issues automated checks miss
6. Decontaminationn-gram overlap check against eval benchmarksEnsure no leakage from MMLU, HumanEval, etc.
Synthetic Data Generation

Use a stronger model (e.g., GPT-4, Claude) to generate training data for a weaker model. Techniques include:

  • Self-Instruct: Model generates its own instruction-response pairs
  • Evol-Instruct: Iteratively evolve instructions for more complexity
  • Self-Play: Model debates itself to generate preference data
Data Augmentation

Increase diversity without collecting new data:

  • Rephrase instructions: Same task, different wording
  • Vary difficulty: Easy / medium / hard versions
  • Multilingual: Translate high-quality examples
  • Role variations: Different system prompts
Balance Across Tasks and Topics

An imbalanced dataset causes mode collapse — the model becomes excellent at overrepresented tasks and terrible at underrepresented ones. Track category distributions and oversample minority categories or downsample majority ones. Aim for at least 50–100 examples per task category.

Data Quality Benchmarks

Dataset SizeSuitable ForRecommended MethodExpected Outcome

Decontamination Is Critical: If your training data overlaps with evaluation benchmarks (MMLU, HumanEval, GSM8K), your metrics will be artificially inflated and unreliable. Use n-gram overlap detection (8-gram or 13-gram) against all benchmark test sets before training.

Choosing the right adaptation strategy for your use case.

Adaptation Decision Table
