6. Inference & Serving

KV cache management, continuous batching, speculative decoding, quantization, and production serving frameworks

6.1

The Inference Challenge

Why generating tokens is fundamentally harder than training on them

Training is massively parallel: every token in every sequence is processed simultaneously. Inference is fundamentally different. Autoregressive generation is inherently sequential — each output token depends on all previous tokens. This creates a strict dependency chain that leaves even the most powerful GPUs largely idle, making inference the dominant cost in production LLM deployments.

PHASE 1

Prefill (Prompt Processing)

The entire input prompt is processed in a single forward pass. All tokens are available upfront, so attention is computed in parallel. This phase is compute-bound and achieves high GPU utilization. TTFT is primarily determined by prefill speed.

PHASE 2

Decode (Token Generation)

Tokens are generated one at a time. Each step runs a full forward pass but produces only one token, yielding extremely low arithmetic intensity (FLOPs per byte loaded). This phase is memory-bandwidth-bound — the bottleneck is loading model weights from HBM, not computing with them.

Key Performance Metrics

The Latency vs. Throughput Tradeoff

Batching improves throughput by reusing model weights across requests, but increases per-request latency. Production systems must balance batch size against SLA targets — maximizing throughput within an acceptable latency envelope.

The GPU Utilization Problem

During decode with batch size 1, a typical LLM achieves <5% GPU utilization. The GPU loads the entire model (~140 GB for 70B FP16) from HBM to produce a single token — performing just 2 FLOPs per parameter loaded. On an H100, this limits single-request decode to ~24 tokens/second for a 70B model, regardless of the GPU's 990 TFLOPS compute capacity. The entire inference optimization landscape exists to close this gap.

6.2

KV Cache

Trading memory for compute by caching attention states

In autoregressive generation, each decode step computes attention between the new token's query and the keys/values of all preceding tokens. Without caching, this means re-computing K and V projections for every previous token at every step — an O(n²) total cost. The KV Cache stores computed K and V tensors and reuses them, so each decode step only computes K/V for the single new token and appends to the cache.

KV Cache Memory Formula

KV_cache_bytes = 2 × n_layers × n_kv_heads × d_head × seq_len × precision_bytes

The factor of 2 accounts for both K and V tensors. With GQA, n_kv_heads is the number of KV groups (not query heads). The cache grows linearly with both sequence length and batch size, quickly dominating GPU memory in production.

KV Cache Memory Example: Llama 2 70B (GQA, 8 KV heads) Per token: 2 × 80 layers × 8 heads × 128 dim × 2 bytes = 320 KB Per request @ 4K ctx: 320 KB × 4,096 = 1.28 GB Per request @ 128K ctx: 320 KB × 131,072 = 40 GB ← exceeds single GPU! Batch of 32 @ 4K ctx: 1.28 GB × 32 = 41 GB ← half an 80GB A100
The KV Cache Scaling Problem

KV cache grows along two dimensions: sequence length and batch size. For a 70B model on 80 GB A100s, after loading model weights (~140 GB across 2 GPUs), remaining memory limits you to ~20–30 concurrent 4K-context requests. KV cache management is the single most important optimization in LLM serving.

KEY INNOVATION

PagedAttention (vLLM)

Traditional KV cache reserves a contiguous block for the maximum possible sequence length upfront, causing 60–80% memory waste from internal fragmentation.

PagedAttention (Kwon et al., 2023) manages KV cache like virtual memory: fixed-size blocks allocated on demand, linked by a page table. Blocks are freed immediately when requests complete. This enables near-zero waste, copy-on-write sharing of prefix KV blocks across requests with common system prompts, and parallel sampling where multiple outputs share the same prompt cache. Result: 2–4× throughput improvement over naive implementations.

6.3

Continuous Batching

Maximizing GPU utilization by dynamically managing the request batch

Static batching collects N requests, processes them together, and returns results only when the longest request finishes. Shorter requests sit idle, wasting compute proportional to the variance in output lengths. Continuous batching solves this by inserting new requests into available slots and evicting completed ones at every decode step.

STATIC

Static Batching

Collect N → process all → wait for longest → return all. Short requests block slots from reuse. Effective throughput 2–10× below optimal.

CONTINUOUS

Continuous (In-Flight) Batching

At each decode iteration, the scheduler inserts new requests and evicts completed ones. Freed slots are immediately filled, keeping GPU utilization consistently high across varying response lengths.

Static Batching: Continuous Batching: Step 1: [A B C D] Step 1: [A B C D] Step 2: [A B C D] Step 2: [A B C D] Step 3: [A _ C D] B done Step 3: [A E C D] B done → E joins Step 4: [A _ _ D] C done Step 4: [A E F D] C done → F joins Step 5: [A _ _ D] waiting Step 5: [G E F D] A done → G joins Step 6: [A _ _ _] D done Step 6: [G E F H] D done → H joins Static: 4 requests in 6 steps Continuous: 8 requests in 6 steps (2×)
Iteration-Level Scheduling

Continuous batching operates at iteration-level granularity: each decode step re-evaluates which requests to include. This enables priority-based scheduling, preemption of low-priority requests, and SLA-aware admission control. vLLM, TensorRT-LLM, and SGLang all implement iteration-level continuous batching as their default strategy.

Impact: Continuous batching delivers 2–10× higher throughput over static batching, with improvement scaling with response length variance. It is a baseline requirement for any production LLM serving system.
6.4

Speculative Decoding

Breaking the sequential bottleneck by drafting and verifying in parallel

The key insight: verifying N candidate tokens in a single forward pass is far cheaper than generating N tokens sequentially. A small, fast "draft" model predicts several tokens; the large "target" model verifies all of them at once, accepting correct ones and re-generating from the first disagreement. When the draft model guesses correctly, multiple tokens are produced per large-model forward pass.

Draft-Then-Verify Pipeline

  1. Draft: Small model (e.g., 1B) generates K candidate tokens (K=4–8).
  2. Verify: Large target model processes all K drafts in a single forward pass.
  3. Accept/Reject: Keep tokens where distributions agree; resample from target at first disagreement.
  4. Repeat: Continue from the last accepted token. Average 2–5 accepted per step.
Speedup ≈ avg_accepted_tokens / (draft_time + verify_time) × baseline_decode_time
Mathematically Lossless

The acceptance-rejection scheme ensures the final output distribution is identical to sampling directly from the target model. When a draft token is rejected, the algorithm samples from a corrected distribution. This is a pure speed gain with zero quality degradation.

MethodDraft SourceSpeedupKey Idea
Classic Spec. DecodingSeparate small model2–3×Dedicated draft model (1B for 70B target)
MedusaExtra prediction heads2–3×Multiple lightweight heads on target predict future tokens in parallel
EAGLEAR feature head2.5–3.5×Lightweight AR head over hidden states; better draft quality than Medusa

Speculative decoding works best for predictable tokens (common phrases, code boilerplate, structured outputs). Speedup is lower for creative or highly unpredictable text where the draft model frequently disagrees with the target.

6.5

Quantization

Reducing precision to fit larger models on fewer GPUs and decode faster

Since LLM inference is memory-bandwidth-bound, reducing bytes per parameter directly improves decode speed. Quantization converts weights from FP16/BF16 to INT8, INT4, or FP8. A 70B model at FP16 needs ~140 GB; at INT4, it fits in ~35 GB — a single A100 or two consumer GPUs. Less memory also means less HBM bandwidth consumed per forward pass, directly improving throughput.

GPTQ

  • Approach: Layer-wise quantization using second-order (Hessian) information
  • Precision: INT4 / INT3 with group-wise scaling
  • Speed: ~4 hours for 70B on a single GPU
  • Ecosystem: Widely supported in vLLM, TGI, AutoGPTQ

AWQ (Activation-Aware)

  • Approach: Protects salient weights identified by activation magnitudes
  • Precision: INT4 with per-channel scaling
  • Quality: Consistently better than GPTQ, especially for smaller models
  • Key insight: 1% of weights carry disproportionate importance
HARDWARE-NATIVE

FP8 Quantization

H100 GPUs natively support FP8 (E4M3/E5M2) in tensor cores, enabling 2× throughput over FP16 with negligible quality loss. FP8 preserves floating-point dynamic range while halving memory. The "free lunch" of quantization on Hopper+ hardware.

EXTREME

SqueezeLLM & Sub-4-Bit

SqueezeLLM uses non-uniform quantization with lookup tables, placing more levels where weight values are dense. Research continues into 2-bit and ternary quantization, but quality degrades significantly below INT4 for most tasks.

Practical Guidance

On Hopper GPUs, start with FP8 — nearly free quality-wise, doubles throughput. For memory constraints, AWQ INT4 offers the best quality-per-bit. For local/edge, GGUF Q4_K_M via llama.cpp is the sweet spot. Avoid going below INT4 unless your use case tolerates noticeable quality degradation.

Inference Frameworks

Production-grade engines for efficient LLM serving

Framework choice directly impacts throughput, latency, and operational complexity. All major frameworks now support continuous batching, quantization, and multi-GPU serving, but they differ in optimization strategies and target deployment scenarios.

OPEN SOURCE LEADER

vLLM

The most widely adopted open-source engine. Built around PagedAttention, with continuous batching, tensor parallelism, multi-LoRA serving, and prefix caching. Compatible with OpenAI's API format for drop-in replacement. Active community, rapid feature development, and support for NVIDIA, AMD, and TPU backends.

NVIDIA-OPTIMIZED

TensorRT-LLM

NVIDIA's first-party engine with custom CUDA kernels, native FP8 tensor core support, and inflight batching. Achieves highest raw throughput on NVIDIA GPUs but requires NVIDIA hardware and has a steeper learning curve. Integrates with Triton for production serving.

STRUCTURED OUTPUT

SGLang

Introduces RadixAttention: a radix tree for automatic prefix sharing — common prefix KV is computed once and shared. Excels at constrained decoding (JSON schema, regex), making it ideal for agentic workflows requiring strict output formats.

EDGE / LOCAL

llama.cpp

C/C++ engine running LLMs on consumer hardware via the GGUF format with extensive quantization options (Q2–Q8). Efficient on CPUs, Apple Silicon (Metal), and GPUs. The go-to for local deployment; powers the Ollama ecosystem and numerous desktop apps.

For cloud GPU: vLLM or TensorRT-LLM. For structured outputs and agents: SGLang. For local experimentation or edge deployment: llama.cpp remains unmatched in hardware compatibility.

6.7

Serving Patterns

Production architectures for multi-model, multi-tenant LLM systems

Production LLM serving involves multiple model variants, adapter swapping, intelligent routing, caching layers, and auto-scaling policies. These serving patterns compose the inference primitives into systems that balance cost, latency, quality, and reliability at scale.

MULTI-TENANT

Multi-LoRA Serving

A single base model is loaded once, and multiple LoRA adapters (~10–100 MB each) are swapped at request time per customer/task. Hundreds of adapters from one GPU deployment. vLLM supports this natively — different requests in the same batch can use different adapters, managed transparently.

GPU memory: 1 × base_model + N × lora_adapter (negligible) ≈ 1 × base_model
ROUTING

Model Routing

A router classifies requests by complexity and routes them to appropriately-sized models — simple queries to 8B, complex reasoning to 70B, hardest tasks to the largest model. Reduces average cost 3–5× while maintaining tail quality.

CACHING

Prefix Caching

Compute KV cache for shared system prompts once and reuse across all requests. SGLang's RadixAttention auto-detects common prefixes. Eliminates 50–90% of prefill computation for apps with long system prompts.

SCALING

Auto-Scaling

Scale based on queue depth and P99 latency rather than CPU/memory metrics. Scale-up on threshold breach; scale-down slowly to avoid thrashing. KV cache memory limits max concurrency per instance, making capacity planning predictable.

ADVANCED

Disaggregated Inference

Separate prefill (compute-bound) and decode (memory-bandwidth-bound) onto different hardware optimized for each phase. Prefill cluster transfers KV caches to decode cluster via high-speed interconnect. Enables independent scaling and better hardware utilization.

A/B Testing & Canary Deployments

Offline evals don't fully predict real-world quality. Production systems route traffic percentages to candidate models, measuring user-facing metrics (ratings, task completion, engagement). Shadow deployments run candidates without serving output for latency/error comparison. Gradual rollouts (1% → 10% → 50% → 100%) limit blast radius.

6.8

Streaming & Optimization

Delivering tokens to users and optimizing end-to-end cost

Users perceive streaming responses as significantly faster than waiting for a complete response. Beyond UX, several optimization techniques target specific inference bottlenecks — from reducing TTFT to minimizing cost per token at scale.

PROTOCOL

Server-Sent Events (SSE)

The standard for streaming LLM responses. Server sends data: events over a persistent HTTP connection, each containing one or more tokens. Unidirectional, simple text format, handles reconnection natively. The OpenAI API's stream: true uses SSE, making it the de facto standard.

TRANSPORT

Chunked Transfer Encoding

The HTTP mechanism enabling SSE. Data is sent in variable-size chunks without specifying Content-Length upfront, allowing transmission before total response size is known — essential for autoregressive generation with unpredictable output length.

Token Budget Optimization

Early stopping detects repetitive or low-value generation. Length penalties discourage verbose outputs. Structured output constraints (JSON mode, grammar-guided decoding) prevent wasted tokens outside expected formats. Tight max_tokens limits based on task requirements can cut batch processing costs by 30–50%.

Edge Deployment Considerations

  • Format: GGUF (llama.cpp), CoreML (Apple), TFLite (Android)
  • Quantization: Q4_K_M balances quality and memory for mobile
  • Model size: 1B–3B parameters is the on-device sweet spot
  • Latency: No network round-trip; sub-10ms TTFT achievable
  • Privacy: Data never leaves the device

Cost Optimization Checklist

  • Quantize first: FP8 on Hopper, AWQ INT4 on Ampere
  • Enable prefix caching: Share KV for common system prompts
  • Use continuous batching: Never static batching in production
  • Right-size the model: Route easy queries to smaller models
  • Set max_tokens tightly: Don't allocate 4K for 200-token tasks
  • Monitor GPU util: Target >60% decode utilization via batching
Emerging: Disaggregated inference, speculative decoding as a default, and FP4/INT4 as standard serving precision are converging toward an order-of-magnitude inference cost reduction over 2024–2026 — driven by software and systems innovation as much as new hardware.
