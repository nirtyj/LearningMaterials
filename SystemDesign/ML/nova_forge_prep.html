Amazon Nova Forge

Staff-Level Interview Preparation — Fine-Tuning, Inference, Benchmarking & RL Infrastructure

1. What is Nova Forge

Nova Forge is Amazon's "open training" platform that allows enterprise customers to build custom frontier models (called "Novellas") using Amazon Nova as the base. Unlike traditional fine-tuning services that only modify a fully-trained model at the end, Nova Forge gives customers access to intermediate model checkpoints at pre-training, mid-training, and post-training phases so proprietary data can be injected at the optimal point in the training lifecycle.

Why It Matters (The Elevator Pitch)

Traditional CPT on proprietary data alone causes catastrophic forgetting. Traditional SFT/RLHF operates too late — layering behavior on a frozen knowledge base. Nova Forge solves both: customers blend their data with Amazon-curated datasets at any training phase, preserving foundational capabilities while deeply customizing the model.

Business Context

Launched at AWS re:Invent 2025. Priced at ~$100K/year for platform access (compute is additional via SageMaker). Early adopters include Reddit (content moderation), Nimbus Therapeutics (drug discovery), Sony, Booking.com, Nomura Research Institute, and Cosine AI (software agents).

3. Model Architecture

Nova Pro, Lite, and Micro are all based on the Transformer architecture. The technical report confirms standard decoder-only transformer design. Training data spans 200+ languages with emphasis on 15 key languages. The training pipeline follows: Pre-training → SFT → Reward Model Training → RL from human preferences.

Multimodal Design

For Nova Lite and Pro, multimodal inputs (images, video) are processed through dedicated encoders that produce token embeddings fed into the unified transformer backbone. Nova Canvas uses a diffusion transformer for image generation. Nova Sonic uses a speech encoder/decoder bracketing a core transformer trained through pre-training, SFT, RL, and preference tuning optimized for speech.

Runtime Performance Metrics

Amazon evaluates runtime using three metrics:

  • TTFT — Time to First Token (latency from API request to first token)
  • OTPS — Output Tokens Per Second (throughput)
  • Total Response Time — End-to-end completion latency

5. Data Mixing & Anti-Forgetting

This is Nova Forge's signature innovation. Instead of training only on customer data (which causes catastrophic forgetting), customers blend their proprietary data with Amazon Nova-curated datasets at every training phase.

How Data Mixing Works

  • Amazon provides curated datasets categorized by domain (e.g., code, math, multilingual, general knowledge, RAI)
  • Customers configure mixing ratios in their training recipe via a nova_data configuration block
  • The "RAI" category specifically contains responsible AI alignment data
  • Mixing is available at ALL training phases — CPT, mid-training, SFT, and RFT

Why This Prevents Catastrophic Forgetting

During continued pre-training on only proprietary data, the model's weight updates drift away from the general-capability manifold. By interleaving Nova-curated data from the same distribution used to train the base model, gradient updates maintain coverage of general capabilities while acquiring domain knowledge. The mid-training checkpoint uses a more conservative learning rate than full pre-training, which further reduces forgetting risk.

Interview Answer Pattern

"Catastrophic forgetting occurs when continued training on narrow data causes the model to lose previously learned capabilities. Nova Forge addresses this through data mixing — blending customer data with Amazon-curated datasets that maintain coverage of the original training distribution. The mid-training checkpoint specifically uses a lower learning rate than pre-training, making the model absorb new knowledge without destabilizing existing capabilities. This is fundamentally different from LoRA or adapter-based approaches, which avoid forgetting by freezing most weights but can't embed deep domain knowledge."

Reward Function Types

  • RLVR (Reinforcement Learning via Verifiable Rewards) — Rule-based Lambda functions. Perfect for code execution, math verification, deterministic checks.
  • RLAIF (Reinforcement Learning from AI Feedback) — LLM-as-a-judge evaluates responses against criteria you define. Ideal for subjective qualities (helpfulness, style, brand voice).
  • Remote Reward Functions (Forge) — API-based. Customer runs reward computation in their own VPC infrastructure. Enables physics simulators, complex code eval with nested tool-calling, robotics tasks.

8. GRPO Deep Dive

Group Relative Policy Optimization (GRPO) is the core RL algorithm used in Nova RFT. It's a variant of policy optimization designed specifically for language models.

How GRPO Works

  1. Sample Generation: For each training prompt, generate N responses (configured via number_generation, typically 2-16)
  2. Reward Scoring: Each response is scored by the reward function (Lambda, LLM judge, or remote endpoint)
  3. Advantage Estimation: Compute relative advantage within the group — responses are compared against each other, not against an absolute baseline
  4. Policy Update: Update model weights to increase probability of high-reward responses and decrease probability of low-reward responses
GRPO vs PPO — Key Distinction

PPO requires a separate value network (critic) to estimate advantages, which doubles memory requirements and adds training complexity. GRPO eliminates the critic by computing advantages relative to the group of sampled responses for each prompt. This makes it more memory-efficient and simpler to implement, while maintaining similar alignment quality. The tradeoff is that you need more samples per prompt (N ≥ 2) to estimate meaningful advantages.

Critical Design Decision

BYOO uses off_policy_async rollout strategy with an age_tolerance parameter. This means rollouts can happen asynchronously and be slightly stale — crucial for environments with high latency (physics sims, complex code eval). The trainer doesn't block waiting for every rollout to complete. The timeout parameter sets the max time for the entire conversation between trainer and environment — a timeout causes training failure.

10. SageMaker HyperPod Infrastructure

Nova Forge uses SageMaker HyperPod as its training platform — purpose-built persistent GPU clusters for distributed training at scale.

Key HyperPod Features

Fault Tolerance & Resiliency

  • Automatic node replacement — Detects hardware faults, replaces nodes, resumes from checkpoint
  • Checkpointless training — Recovers via peer-to-peer transfer of model/optimizer states from healthy accelerators. No need for checkpoint-based restart. Achieves >95% training goodput on 1000+ accelerator clusters
  • Saves up to 40% training time over standard approaches

Elastic Training

  • Training jobs automatically scale up/down based on resource availability
  • Preserves global batch size and learning rate across scaling transitions
  • Integrates with Kubernetes control plane and resource scheduler
  • Example: 256 GPU cluster, inference releases 96 GPUs at night → training absorbs them automatically

Task Governance

  • Manages task queues across training, fine-tuning, experimentation, and inference
  • Prioritizes critical tasks, preempts lower-priority work
  • Reduces model development costs by up to 40%

Observability

  • Unified dashboard via Amazon Managed Grafana
  • Metrics published to Amazon Managed Prometheus
  • CloudWatch integration for logs and system metrics

16. System Design Questions

Requirements

Multi-tenant platform where customers upload data, configure training recipes, launch training jobs on GPU clusters, evaluate results, and deploy custom models to production inference.

Core Components

  • UI Layer: SageMaker Studio console — checkpoint selection, dataset upload, recipe configuration, monitoring
  • Job Orchestrator: Receives training requests, provisions infrastructure, manages lifecycle. Uses SageMaker Training Job API or HyperPod cluster allocation.
  • Data Pipeline: Customer data uploaded to S3 → validated → merged with Nova-curated datasets per mixing ratios → JSONL generation → distributed to training nodes via FSx for Lustre
  • Training Cluster: HyperPod with RIG (Restricted Instance Groups) for Forge customers. P5/P5en instances. Slurm or EKS orchestration.
  • Checkpoint Manager: Stores intermediate checkpoints to S3. Manages checkpoint lifecycle across pre-train/mid-train/post-train phases. Optimizes checkpoint frequency.
  • Model Registry: Tracks all model versions, training configs, eval metrics, lineage. Links training job → evaluation job → deployment.
  • Evaluation Pipeline: Separate training jobs that run benchmark/custom evaluations. Results to S3 + TensorBoard.
  • Deployment Pipeline: CreateCustomModel API to Bedrock. CreateCustomModelDeployment for on-demand. Provisioned throughput setup.

Key Design Decisions

  • RIG isolation ensures Forge customers' Nova checkpoints and curated data are only accessible by authorized clusters
  • Checkpointless training on HyperPod for fault tolerance at scale
  • Elastic training to maximize GPU utilization across training + inference workloads
  • Network isolation for inference endpoints

Challenge

The model needs to interact with customer-hosted environments that may be physics simulators, code evaluation systems, or multi-step agent workflows. These environments have high latency (seconds to minutes), require multi-turn conversations, and run in the customer's VPC for data security.

Architecture

  • Training Cluster (HyperPod): Actor model generates responses. Trainer updates weights. Generation server runs vLLM-like inference.
  • Communication Layer (AWS-managed): 3 Lambda functions + 4 SQS queues deployed via SAM template. Acts as async message bus between training cluster and customer environment.
  • Customer Environment (Customer VPC): Receives prompts + model responses, executes domain-specific evaluation, returns reward scores.

Key Tradeoffs

  • Async off-policy with age tolerance: Allows training to proceed without blocking on slow environments. Tradeoff: slightly stale rollouts may reduce training signal quality.
  • SQS for durability: Messages survive Lambda failures, but adds latency vs. direct gRPC. Right choice for reliability over speed.
  • Timeout handling: Rollout timeout causes training failure — must be set conservatively. Customer environments must be reliable.

Requirements

Evaluate custom Nova models against standard benchmarks and custom datasets. Support LLM-as-a-Judge for subjective quality. Scale to millions of examples. Track eval lineage.

Architecture

  • Evaluation Recipes: YAML configs specifying task, strategy, metrics, dataset location
  • Evaluation Container: Nova-specific container that loads model, runs inference on evaluation data, computes metrics
  • Multi-Node Execution: Configure replicas in recipe → container auto-distributes workload → aggregates results
  • LLM-as-a-Judge: Dedicated Nova Judge model. Forward + backward pass for positional debiasing. Bradley-Terry scoring.
  • Results Store: S3 for raw metrics, TensorBoard for visualization, CloudWatch for monitoring

  • Standard benchmarks (MMLU, HumanEval) — automated, recipe-based
  • BYOD — customer dataset, gen_qa strategy
  • BYOM — custom Lambda metrics
  • LLM Judge — pairwise A/B with rationales
  • Rubric Judge — multi-dimensional scoring
  • MM Judge — multimodal evaluation (image inputs)

Requirements

Serve custom Nova models with low latency, auto-scaling, cost optimization. Support both streaming and non-streaming. Handle multiple model sizes on heterogeneous hardware.

Answer: During continued pre-training, each batch contains a mix of customer data and Nova-curated data. The loss function computes gradients from both sources simultaneously. Customer data gradients push weights toward domain-specific representations, but curated data gradients act as a regularizer — they counteract drift by maintaining the gradient signal from the original training distribution. The net gradient update is a weighted average that learns new knowledge while preserving existing capabilities. The mid-training checkpoint amplifies this effect by using a lower learning rate, which means each gradient step makes a smaller absolute change to weights, reducing the risk of overwriting critical parameters.

This is more effective than techniques like EWC (Elastic Weight Consolidation) which computes a Fisher information matrix to penalize changes to "important" weights, because data mixing directly maintains the training distribution rather than estimating parameter importance post-hoc.

Answer:

  1. Sample prompts: Draw a batch of prompts from the training dataset (size = global_batch_size / number_generation)
  2. Generate completions: For each prompt, the generation server produces N completions (number_generation, e.g., 16 for BYOO). Uses temperature sampling for diversity.
  3. Score with reward: Each completion is scored — Lambda function (RLVR), LLM judge (RLAIF), or remote endpoint (Forge). Returns scalar reward per completion.
  4. Compute group advantages: For each prompt, normalize rewards within the group: advantage_i = (reward_i - mean(rewards)) / std(rewards). This relative scoring means no value network is needed.
  5. Policy gradient: Compute loss = -sum(advantage_i * log_prob(completion_i | prompt)). Add KL penalty term against reference policy to prevent reward hacking.
  6. Update weights: Standard optimizer step (AdamW) with the computed gradients.
  7. Iterate: Repeat for max_steps.

The key insight is that by comparing completions within a group, GRPO creates a self-normalizing advantage estimate without the instability and memory overhead of a separate critic model.

Answer: Traditional fault recovery requires: (1) detect failure, (2) pause entire cluster, (3) replace faulty node, (4) load checkpoint from S3, (5) resume. This creates significant idle compute during recovery. Checkpointless training eliminates step 4 by using peer-to-peer state transfer. When a node fails:

  1. HyperPod detects the failure and swaps in a replacement node
  2. The replacement node requests model parameters and optimizer states from healthy peer nodes over the high-bandwidth EFA network
  3. Since FSDP shards state across all nodes, each healthy node holds a shard — the replacement reconstructs its own shard from peers
  4. Training resumes from the current state without reverting to a stale checkpoint

This achieves >95% training goodput on 1000+ accelerator clusters. The cost savings come from eliminating both the checkpoint write overhead (which can be significant for large models) and the recovery read overhead.

Answer: Multi-tenant isolation in Nova Forge operates at several levels:

  • Compute Isolation: Each customer gets dedicated HyperPod instance groups (RIG). No shared GPU memory between tenants.
  • Data Isolation: Customer data stays in their own S3 buckets. Nova-curated data is read-only shared from Amazon-managed buckets. Execution roles with fine-grained IAM policies control access.
  • Network Isolation: Training runs in customer's VPC. Inference endpoints have EnableNetworkIsolation. BYOO environments run entirely in customer VPC.
  • Model Isolation: Custom models are private to the customer account. Deployed to Bedrock as private models with same security as native models.
  • Access Control: forge-subscription IAM tag required on execution roles. Subscription approval process gates access.

Answer: LLM judges are known to prefer responses presented in certain positions (often the first response). Nova's judge addresses this through bidirectional evaluation: for every prompt, it runs the comparison twice — once with Model A's response first and Model B's response second, then reversed. The final Bradley-Terry probability is computed from both orderings, effectively canceling out positional bias. Additionally, the judge was trained specifically for evaluation tasks through SFT + RL on human preference data, which helps it learn position-invariant quality assessment. Each judgment also produces a natural language rationale explaining the preference, enabling humans to audit the judge's reasoning.
