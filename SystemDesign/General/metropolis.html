Platform Ecosystem Map

Metropolis is a platform, not a single product — a critical distinction for anyone preparing for system design discussions. It provides a collection of SDKs, microservices, foundation models, and reference blueprints that can be composed together to build vision AI applications. The platform is designed to be deployable consistently across NVIDIA's entire compute spectrum, from the smallest Jetson Orin Nano (8 TOPS) to DGX Cloud clusters with thousands of H100 GPUs. This consistency is achieved through a shared software foundation: CUDA, cuDNN, the NVIDIA Container Runtime, and a common API surface across the TensorRT, DeepStream, and Triton SDKs. See the Metropolis developer portal for the current product lineup.

The architecture is layered into five distinct tiers, each with well-defined interfaces. Hardware vendors integrate at the bottom; application developers at the top. The middle layers — SDKs and microservices — provide the standardized building blocks that make the platform composable. This layering means you can swap out a Jetson Orin for an A100 server and reuse the same DeepStream pipeline with only hardware-specific engine files changing. The Metropolis product page provides an overview of supported deployment targets and partner ecosystem.

Five-layer architecture: Hardware → Software Foundation → Core SDKs → Microservices → Applications

Why This Layering Matters for System Design

In a system design interview, the layered architecture is your anchor. When asked "design a multi-camera analytics system for a warehouse," you can start at the top (application requirements) and walk down: "We need MTMC tracking across 50 cameras, so we'll use the Metropolis MTMC reference application running on Metropolis Microservices v2. That sits on DeepStream for stream processing and TensorRT for inference. At the hardware level, an L40-based server can handle ~40 concurrent streams with PeopleNet detection + NvDeepSORT tracking."

The key architectural insight is that each layer has well-defined horizontal scaling characteristics. DeepStream scales by adding more GPU streams per card (vertical) or more cards (horizontal). Triton scales with model replicas and Kubernetes HPA. The microservices layer scales with standard K8s patterns. Understanding which layer is the bottleneck for a given workload is the core system design skill.

1

Video Ingestion — VST (Video Storage Toolkit)

IP cameras deliver RTSP/ONVIF streams to the Video Storage Toolkit (VST) microservice. VST handles camera discovery (ONVIF probe), automatic reconnection on stream drops, persistent video storage (recording to local or network storage for later playback), and provides proxy RTSP links for downstream consumers. This decoupling is important: downstream DeepStream pipelines connect to VST's proxy links, not directly to cameras, enabling VST to handle all the messy camera management (authentication, reconnection, stream multiplexing) independently. The DeepStream reference app docs show how VST integrates with perception pipelines. VST also provides REST APIs for camera CRUD operations, making it easy to add/remove cameras at runtime without pipeline restarts.

2

Stream Processing — Decode & Batch

DeepStream receives streams via nvurisrcbin (a convenience bin that auto-selects the right source/decoder elements), hardware-decodes via NVDEC (H.264/H.265/AV1 — NVDEC is a dedicated ASIC on NVIDIA GPUs, separate from CUDA cores, so decode does NOT consume GPU compute), and batches multiple streams through nvstreammux into GPU-resident batches. The nvstreammux plugin reference documents all batching parameters. Key detail: all frame data stays in GPU memory (NVMM) from decode through inference through encoding — there is no CPU-GPU copy in the critical path. This zero-copy design is what enables processing 30+ concurrent 1080p streams on a single GPU.

3

AI Inference — Primary Detection

The nvinfer plugin runs TensorRT-optimized detection models. Common choices include PeopleNet (ResNet-34 backbone, trained on 7.6M+ images for person/face/bag detection), TrafficCamNet, or custom YOLO/DINO models fine-tuned via TAO Toolkit. Alternatively, nvinferserver delegates inference to Triton Inference Server for multi-framework support (TensorFlow, PyTorch, ONNX Runtime). The choice between nvinfer and nvinferserver is a key design decision: nvinfer has lower latency (in-process TensorRT), while nvinferserver enables model versioning, A/B testing, and dynamic model loading via Triton. See the nvinfer plugin reference for configuration details.

4

Object Tracking

nvtracker assigns persistent object IDs across frames using one of five algorithms: IOU (simplest, bbox overlap only), NvSORT (Kalman filter motion prediction), NvDeepSORT (adds ReID CNN for appearance-based association), NvDCF (discriminative correlation filter — highest accuracy, GPU-accelerated), or MaskTracker (SAM2-based, generates segmentation masks per tracked object). The algorithm choice is a latency-accuracy trade-off documented in the nvtracker plugin reference. NvDCF is the only algorithm that generates real tracker_confidence values; all others default to 1.0.

5

Analytics

nvdsanalytics computes higher-order analytics from tracked object metadata: line-crossing counts (configurable lines with direction sensitivity — e.g., "count people entering through the main door"), ROI occupancy (how many objects are currently inside a defined polygon), direction detection (classify object movement direction), and Single-View 3D Tracking (estimate world coordinates from monocular camera using calibration). These analytics are configured via a text file (not code) specifying lines, ROIs, and direction vectors per source. The nvdsanalytics plugin docs provide the full configuration reference.

6

Messaging & Output

Metadata flows to external systems via two plugins: nvmsgconv converts the NvDs metadata hierarchy into JSON payloads (with a customizable schema — you can define your own JSON structure), and nvmsgbroker sends those payloads to Kafka, MQTT, AMQP, Redis, or Azure IoT Hub. Within Metropolis microservices, Redis serves as the intra-cluster message bus, while Kafka handles inter-service communication. Visual output goes through nvdsosd (renders bounding boxes, labels, analytics overlays on frames) to display, RTSP re-streaming, or file recording via NVENC. The messaging architecture is detailed in the nvmsgbroker docs.

7

VLM Reasoning (Optional — New in 2025)

The newest addition to the pipeline: Cosmos Reason or VILA processes keyframes with natural language prompts for zero-shot event detection and summarization via NIM APIs. This is the "agentic" paradigm — instead of training a model per task, you deploy a single VLM and ask it questions in natural language: "Is anyone in this scene not wearing a hard hat?" or "Summarize what happened at this intersection in the last hour." Cosmos Reason specializes in physical-world reasoning (spatial relationships, causality), while VILA handles multi-frame video QA and dense captioning. Both are available as NIM microservices at build.nvidia.com. The Visual AI Agents blog post provides the architectural blueprint.

Edge Tier — Jetson Platform Services

Runs 15+ Docker Compose microservices including VST, DeepStream perception, NanoOWL zero-shot detection, Redis, API gateway, monitoring (Prometheus/Grafana), IoT gateway, and firewall. Designed for autonomous operation — if the network goes down, the edge device continues processing locally and queues events for sync when connectivity returns. The Jetson Platform Services announcement details the microservice architecture. NanoOWL provides zero-shot object detection using OWL-ViT optimized for Jetson, enabling detection of novel objects without retraining.

🏢

On-Premises Tier — Metropolis Microservices v2

Runs on Kubernetes + Helm for enterprise orchestration. Supports multi-camera tracking (MTMC — stitches identities across non-overlapping camera views using transformer-based ReID), occupancy heatmaps, and advanced analytics. Deployed on dGPU servers (A100, H100, L40, RTX PRO 6000). The Enterprise GPU quickstart walks through the full Helm deployment. Key advantage over edge: can process 40-100+ concurrent streams per server with more complex models (larger YOLO, transformer-based detectors).

☁️

Cloud Tier — NIM + DGX + IoT Cloud

Three cloud components: NIM APIs at build.nvidia.com for foundation model inference (Cosmos Reason, VILA), DGX Cloud for TAO training jobs, and IoT Cloud modules for secure edge device management. Edge-to-cloud communication uses SASL/Plain authentication + 2-way TLS. The IoT Gateway on edge maintains an always-on TCP connection, and the API gateway routes cloud requests to specific edge devices by ID. WebRTC enables real-time video streaming from edge to cloud/dashboard. The Metropolis on Jetson overview documents the cloud connectivity architecture.

Multi-Stream Scaling: What to Know for System Design

A common system design question is "how many cameras can one server handle?" The answer depends on the pipeline complexity, resolution, and GPU. Rough benchmarks from NVIDIA documentation and community testing:

Jetson Orin AGX (275 TOPS): 8-16 streams at 1080p30 with PeopleNet detection + NvSORT tracking. NVIDIA L40 (dGPU): 40-60 streams at 1080p30 with PeopleNet + NvDeepSORT. NVIDIA A100: 60-100+ streams at 1080p30 with detection-only (no tracking). These numbers drop significantly with heavier models (YOLOv8-X vs PeopleNet) or higher resolutions (4K). The bottleneck is usually GPU inference compute, not decode (NVDEC) or memory bandwidth. See the DeepStream performance tuning guide for optimization strategies.

DeepStream Pipeline Architecture

DeepStream extends GStreamer with 40+ GPU-accelerated plugins that encapsulate NVIDIA libraries (CUDA, TensorRT, NVDEC, NVENC, VPI). The critical design decision is zero-memory-copy between plugins via NVMM — buffers remain in GPU memory throughout the pipeline. GStreamer's caps negotiation uses video/x-raw(memory:NVMM), format=RGBA to signal NVIDIA-managed memory surfaces. This means that once a frame is decoded by NVDEC into GPU memory, it flows through inference, tracking, analytics, and encoding without ever touching the CPU or system memory in the hot path. The DeepStream architecture overview provides the official plugin taxonomy.

The unified buffer abstraction NvBufSurface wraps GPU/VIC memory across all plugins. On dGPU, the default memory type is NVBUF_MEM_CUDA_DEVICE (pure GPU VRAM — fastest GPU access, invisible to CPU). On Jetson, the default is NVBUF_MEM_SURFACE_ARRAY (VIC/NVDEC hardware memory — enables zero-copy sharing between NVDEC, NVENC, VPI, and CUDA without any memory copies). You can override the memory type in nvvideoconvert using the nvbuf-memory-type property. The NvBufSurfaceMemType API reference documents all memory types.

TensorRT Optimization Pipeline

TensorRT is fundamentally a compiler, not a runtime — a critical distinction. It takes a trained neural network (ONNX, Caffe, or UFF format), applies graph-level optimizations and hardware-specific kernel selection, and produces a serialized .engine file optimized for a specific GPU architecture. The optimization process is documented in the TensorRT Developer Guide, and the latest features are tracked on the TensorRT product page.

⚠ Layer Fusion: Why It's the #1 Optimization

Vertical fusion (Conv+BN+ReLU → single CBR kernel) reduces HBM accesses from 6 to 2 — the dominant bottleneck for memory-bound inference. Without fusion, each operation reads input from HBM and writes output back. With fusion, intermediate results stay in GPU registers/L2 cache. This single optimization typically delivers 2-3× throughput improvement on bandwidth-limited operations. Horizontal fusion is critical for Inception-style architectures where multiple parallel 1×1 convolutions share the same input — they merge into a single wide convolution with concatenated weight matrices. The TensorRT layer fusion documentation explains which patterns are auto-detected. Attention patterns (BMM-Softmax-BMM) require custom plugins; see the TensorRT plugin repository.

Metadata Additive Flow Through Pipeline

Each plugin in the pipeline reads existing metadata and appends its own. This additive design is crucial to understand because it means no plugin destroys upstream metadata — your custom probe at any point in the pipeline can access everything added by upstream plugins.

metadata-additive-flow 
nvstreammux   →  Creates NvDsBatchMeta, attaches to GstBuffer. Each source gets a NvDsFrameMeta.
nvinfer(P)    →  Adds NvDsObjectMeta per detection: bbox coords, class_id, confidence.
                 Optionally adds NvDsInferTensorMeta for raw tensor access.
nvtracker     →  Updates existing NvDsObjectMeta: sets object_id (persistent tracking ID),
                 writes tracker_bbox_info (smoothed coordinates), sets tracker_confidence.
                 Does NOT create new NvDsObjectMeta — modifies in place.
nvinfer(S)    →  For each NvDsObjectMeta, crops the object, runs classifier →
                 appends NvDsClassifierMeta with label_info_list (color=red, type=sedan, etc.)
nvdsanalytics →  Reads tracked objects + configured lines/ROIs → adds analytics events
                 as NvDsUserMeta on frame_user_meta_list (e.g., LC_count=42, ROI_occupied=true)
nvdsosd       →  Reads NvDsDisplayMeta to render bounding boxes, labels, lines on frame.
                 Does NOT modify metadata — purely a consumer.
nvmsgconv     →  Reads ALL metadata → serializes to JSON schema → passes to nvmsgbroker
nvmsgbroker   →  Sends JSON payload to Kafka / Redis / MQTT / AMQP / Azure IoT Hub

Probe Function Pattern — The Primary Custom Access Mechanism

Probes are how you inject custom logic into a DeepStream pipeline without writing a new plugin. You attach a callback function to any element's source or sink pad, and your function gets called for every buffer that passes through. The NvDsBatchMeta Python API is the entry point for all metadata access.

probe_function_example.py — Complete working probe with all metadata levels 
import pyds
from gi.repository import Gst

def osd_sink_pad_buffer_probe(pad, info, u_data):
    """Probe attached to nvdsosd sink pad — sees all metadata from upstream."""
    gst_buffer = info.get_buffer()
    if not gst_buffer:
        return Gst.PadProbeReturn.OK

    # Get the batch metadata from the GstBuffer
    batch_meta = pyds.gst_buffer_get_nvds_batch_meta(hash(gst_buffer))

    # Iterate through each frame in the batch
    l_frame = batch_meta.frame_meta_list
    while l_frame is not None:
        frame_meta = pyds.NvDsFrameMeta.cast(l_frame.data)
        source_id  = frame_meta.source_id      # Which camera
        frame_num  = frame_meta.frame_num       # Absolute frame count
        ntp_ts     = frame_meta.ntp_timestamp   # Camera NTP time
        num_objs   = frame_meta.num_obj_meta    # Detection count

        # Iterate through each detected/tracked object in this frame
        l_obj = frame_meta.obj_meta_list
        while l_obj is not None:
            obj_meta = pyds.NvDsObjectMeta.cast(l_obj.data)

            # Detection info (from nvinfer primary)
            class_id   = obj_meta.class_id          # 0=person, 1=face, 2=bag
            det_conf   = obj_meta.confidence         # Detector confidence [0,1]
            label      = obj_meta.obj_label          # "Person", "Car", etc.

            # Tracking info (from nvtracker)
            track_id   = obj_meta.object_id          # Persistent unique ID
            trk_conf   = obj_meta.tracker_confidence # NvDCF: real value; others: 1.0
            trk_bbox   = obj_meta.tracker_bbox_info.org_bbox_coords
            # trk_bbox.left, trk_bbox.top, trk_bbox.width, trk_bbox.height

            # Classification info (from nvinfer secondary)
            l_cls = obj_meta.classifier_meta_list
            while l_cls is not None:
                cls_meta = pyds.NvDsClassifierMeta.cast(l_cls.data)
                l_lbl = cls_meta.label_info_list
                while l_lbl is not None:
                    label_info = pyds.NvDsLabelInfo.cast(l_lbl.data)
                    # label_info.result_label = "Red", "Sedan", etc.
                    # label_info.result_prob  = 0.92
                    l_lbl = l_lbl.next
                l_cls = l_cls.next

            # Example: Filter to high-confidence persons only
            if class_id == 0 and det_conf > 0.7:
                print(f"[Cam {source_id}] Person #{track_id} conf={det_conf:.2f} "
                      f"at ({trk_bbox.left:.0f},{trk_bbox.top:.0f})")

            l_obj = l_obj.next

        # Access analytics metadata (from nvdsanalytics)
        l_user = frame_meta.frame_user_meta_list
        while l_user is not None:
            user_meta = pyds.NvDsUserMeta.cast(l_user.data)
            if user_meta.base_meta.meta_type == pyds.NvDsMetaType.NVDS_USER_META:
                # Cast to analytics-specific metadata
                pass
            l_user = l_user.next

        l_frame = l_frame.next

    return Gst.PadProbeReturn.OK

💡 Practical Tip: Debugging Memory Copies

If you're seeing unexpectedly low performance, check for hidden CPU↔GPU copies. The most common cause is a plugin that requires CPU-visible memory (e.g., a custom non-GPU processor) inserted between GPU-memory plugins. Use nvvideoconvert with nvbuf-memory-type=3 (CUDA_UNIFIED) to make memory CPU-accessible without a full copy, or restructure the pipeline to keep the CPU-access branch separate from the GPU-accelerated path. NVIDIA's Nsight Systems profiler can trace CUDA memory operations and identify unexpected copies.

INT8 Calibration Algorithms — Detailed Comparison

TensorRT provides four calibration algorithms for INT8 quantization, each finding a different threshold T to map FP32 activation ranges to INT8 [-127, 127]. The calibration table is computed once on a representative dataset (100-1000 samples) and reused for all subsequent engine builds on the same architecture. The TensorRT INT8 calibration guide is the definitive reference.

INT8 Calibration — Algorithm Comparison Quantization 
All algorithms solve: Find threshold T such that scale = 127.0 / T
Calibration targets ACTIVATIONS only. Weight ranges come from the parameters themselves.

ENTROPY (IInt8EntropyCalibrator2)  ← DEFAULT, RECOMMENDED FOR CNNs
─────────────────────────────────
  1. Run FP32 inference on calibration dataset
  2. Collect per-layer activation histograms (128-2048 bins)
  3. For each candidate threshold T:
     a. Quantize histogram to INT8 using T
     b. Compute KL divergence: D_KL(P_fp32 || Q_int8)
  4. Select T* = argmin D_KL
  Best general-purpose balance between accuracy and range preservation.
  The "2" in Calibrator2 means it caches calibration data for faster re-calibration.

MINMAX (IInt8MinMaxCalibrator)     ← RECOMMENDED FOR NLP/TRANSFORMERS
──────────────────────────────
  T = max(|activations|) across calibration dataset
  Simplest algorithm. No information loss at extremes, but outliers dominate.
  Works well for transformers where activation distributions are more uniform.
  Sensitive to single-sample outliers — use more calibration data to mitigate.

PERCENTILE (IInt8LegacyCalibrator) ← ROBUST TO OUTLIERS
──────────────────────────────────
  T = CDF⁻¹(p) where p is typically 99.9% or 99.99%
  Clips extreme outliers while preserving the bulk of the activation range.
  Good middle ground when you have noisy data or occasional extreme activations.
  percentile and quantile parameters configurable.

MSE (IInt8EntropyCalibrator2 with MSE mode)  ← MOST COMPUTE, BEST ACCURACY
──────────────────────────────────────────
  T* = argmin Σ(x_fp32 − Quantize(x_fp32, T))²
  Exhaustive search over the threshold space.
  Directly minimizes the metric you care about (reconstruction error).
  Slowest calibration but typically yields best INT8 accuracy.
  Use when accuracy is paramount and calibration time doesn't matter.

NvDCF Deep Dive — Why It's the Production Default

NvDCF (Discriminative Correlation Filter) is unique among the tracker algorithms because it performs independent visual tracking per object — it doesn't just match detections between frames, it actively searches for each tracked object in the current frame using a learned correlation filter. This means it can maintain tracks even when the detector misses an object for several frames (occlusion, motion blur, etc.).

The DCF computation is GPU-accelerated and optionally uses NVIDIA VPI (Vision Programming Interface) on Jetson for PVA hardware acceleration. NvDCF is the only tracker that generates real tracker_confidence values — all other trackers default to 1.0, which means you can't use tracker confidence as a quality signal with those algorithms. The config YAML supports extensive tuning: minDetectorConfidence, maxShadowTrackingAge, probationAge, earlyTerminationAge, and DCF-specific parameters like featureImgSizeLevel and SearchRegionPaddingScale.

Layer Fusion Patterns — Detailed Breakdown

Layer fusion is the single most impactful TensorRT optimization. The key insight is that modern DNN inference is memory-bandwidth-bound, not compute-bound for most layers. A Conv layer followed by BN followed by ReLU requires 3 separate kernel launches, each reading from and writing to HBM (High Bandwidth Memory). Fusing them into a single CBR kernel reduces HBM accesses from 6 (3 reads + 3 writes) to 2 (1 read + 1 write), with intermediate results staying in GPU registers or L1/L2 cache. The TensorRT optimization layer docs describe auto-detected patterns.

Vertical fusion patterns: Conv+BN+ReLU → single CBR kernel (most common), Conv+Add(residual)+ReLU → CRA kernel (ResNet critical path), MatMul+Bias+Activation → single GEMM kernel (transformer feed-forward), ElementWise+ReLU → fused EW kernel. Horizontal fusion: parallel operations sharing the same input are concatenated into a single wider kernel — critical for Inception-style architectures with multiple parallel 1×1 convolutions. Elimination: consecutive transposes cancel, dead layers (no consumers) are removed, constant expressions are folded at build time. Cannot auto-fuse: BMM-Softmax-BMM attention patterns require custom plugins — see the TensorRT plugin repository for FlashAttention implementations.

Triton Model Configuration Deep Dive

Every model in Triton requires a config.pbtxt protobuf text file defining inputs, outputs, batching, instances, and backend. The model configuration reference is the complete specification. Key design decisions: instance_group controls how many copies of the model run in parallel on each GPU (more instances = higher throughput but more GPU memory). dynamic_batching is almost always worth enabling — it accumulates individual requests and forms optimal GPU batches, dramatically improving throughput. The preferred_batch_size should match the TensorRT engine's optimal batch size.

config.pbtxt — Production TensorRT model with dynamic batching
name: "peoplenet_v2"
platform: "tensorrt_plan"          # TensorRT .plan engine file
max_batch_size: 16                 # Must match TRT engine max_batch_size

input [
  {
    name: "input_1"
    data_type: TYPE_FP32
    format: FORMAT_NCHW             # Channels-first (default for TensorRT)
    dims: [ 3, 544, 960 ]          # C, H, W (batch dim implicit)
  }
]
output [
  {
    name: "output_bbox/BiasAdd"
    data_type: TYPE_FP32
    dims: [ 12, 34, 60 ]           # 4 coords × 3 classes, grid H, grid W
  },
  {
    name: "output_cov/Sigmoid"
    data_type: TYPE_FP32
    dims: [ 3, 34, 60 ]            # Coverage/confidence per class per grid cell
  }
]

# Run 2 instances on GPU 0 — doubles throughput at cost of 2× GPU memory
instance_group [
  { count: 2, kind: KIND_GPU, gpus: [ 0 ] }
]

DeepStream ↔ Triton Integration via nvinferserver

The nvinferserver plugin bridges DeepStream's streaming pipeline with Triton's model serving capabilities. It can connect to Triton via gRPC (out-of-process, enables separate scaling of DeepStream and Triton) or C API (in-process, lowest latency — Triton runs as a shared library within the DeepStream process). The custom pre/post-processing interface IInferCustomProcessor lets you write C++ callbacks for format conversion, custom NMS, or complex output parsing. The nvinferserver reference covers the full configuration.

When to use nvinferserver over nvinfer: Use it when you need multi-framework support (PyTorch model alongside TensorRT), model versioning and A/B testing, ensemble pipelines (chained models), dynamic model loading/unloading without pipeline restart, or when you want to share a single Triton instance across multiple DeepStream pipelines. Use nvinfer when you need the lowest possible latency and are using only TensorRT models.

The VLM Paradigm Shift (2025–2026)

Metropolis has undergone a fundamental paradigm shift from pure CNN-based pipelines (train a model per task, deploy per task) to VLM-powered Visual AI Agents using Cosmos Reason, VILA, and the VSS Blueprint. This eliminates the model-per-task bottleneck — a single VLM can answer arbitrary natural-language queries about video content without any task-specific training. The Visual AI Agents blog post from NVIDIA describes the architecture in detail.

How the old world worked: Want to detect people? Train PeopleNet. Detect vehicles? Train TrafficCamNet. Detect PPE violations? Train a custom model on labeled PPE data. Each task required weeks of data collection, labeling, training, and deployment. Adding a new detection category meant another training cycle.

How the new world works: Deploy Cosmos Reason or VILA once. Ask in natural language: "Is anyone in this scene not wearing a hard hat?" or "Summarize all safety violations in the past 4 hours" or "Count vehicles by type at this intersection during rush hour." No training, no labeling, no per-task models. The VLM reasons about the visual content using its pre-trained understanding of the physical world.

CNN vs VLM Trade-offs — What to Know for System Design

CNNs still win on raw throughput (1000+ FPS dense detection), latency (<5ms per frame), edge deployment (fits on Jetson Orin), and cost per inference (orders of magnitude cheaper). VLMs win on flexibility (zero-shot, no training), setup time (minutes vs weeks), handling novel/unexpected events, natural language interface, and complex reasoning (spatial relationships, causality, temporal patterns). Production deployments increasingly use hybrid architectures: CNN pipelines for high-frequency detection at 30 FPS with VLM reasoning triggered on interesting events (keyframes, anomalies) for complex understanding and summarization. This is the architecture the VSS Blueprint implements.
