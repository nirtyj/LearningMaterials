1

Video Ingestion

IP cameras deliver RTSP/ONVIF streams to Video Storage Toolkit (VST) — handles discovery, reconnection, persistent storage, and provides proxy RTSP links for downstream consumers.

2

Stream Processing

DeepStream receives streams via nvurisrcbin, hardware-decodes via NVDEC (H.264/H.265/AV1), and batches multiple streams through nvstreammux into GPU-resident batches.

3

AI Inference

nvinfer runs TensorRT-optimized detection models (PeopleNet, YOLO, etc.) or nvinferserver delegates to Triton for multi-framework support (TF, PyTorch, ONNX).

4

Object Tracking

nvtracker assigns persistent IDs using NvDCF, NvDeepSORT, NvSORT, or IOU algorithms. Each algorithm trades off between accuracy and compute cost.

5

Analytics

nvdsanalytics computes line-crossing counts, ROI occupancy, direction detection, and single-view 3D tracking based on metadata from prior stages.

6

Messaging & Output

Metadata flows to Redis (intra-cluster) or Kafka (inter-service) via nvmsgbroker, then to Kibana dashboards. Visual overlay via nvdsosd and RTSP re-streaming.

7

VLM Reasoning (Optional)

Cosmos Reason or VILA processes key frames with natural language prompts for zero-shot event detection and summarization via NIM APIs — no model training required.

DeepStream Pipeline Architecture

DeepStream extends GStreamer with 40+ GPU-accelerated plugins that encapsulate NVIDIA libraries (CUDA, TensorRT, NVDEC, NVENC, VPI). The critical design decision is zero-memory-copy between plugins via NVMM — buffers remain in GPU memory throughout the pipeline.

TensorRT Optimization Pipeline

TensorRT Engine Build — 6-Step Compiler Pipeline Optimization 
┌─────────────┐     ┌──────────────────┐     ┌──────────────────────────┐
│ ONNX Model  │────▶│  1. PARSE        │────▶│  2. GRAPH OPTIMIZATION   │
│ (or Caffe/  │     │                  │     │                          │

                     │  Plugin matching │     │  │ Conv+Add+ReLU → CRA │ │
                     │  for unsupported │     │  │ MatMul+Bias+Act → 1 │ │
                     │  ops             │     │  └──────────────────────┘ │
                     └──────────────────┘     │  ┌─ Horizontal Fusion ─┐ │
                                               │  │ Parallel 1×1 CBRs   │ │
                                               │  │ → single wide kernel │ │
                                               │  └──────────────────────┘ │
                                               │  ┌─ Elimination ────────┐ │
                                               │  │ Dead layers, const   │ │
                                               │  │ fold, transpose elim │ │
                                               │  └──────────────────────┘ │
                                               └────────────┬─────────────┘
                                                            │
┌────────────────────────────┐    ┌─────────────────────┐   │

│                            │    │                     │
│  ┌─ INT8 Calibrators ────┐ │    │  Profile: GEMM,     │
│  │ Entropy (KL div)      │ │    │  Winograd, FFT,     │
│  │ MinMax (abs max)      │ │    │  Implicit GEMM      │

Engines are GPU-architecture specific — must rebuild for each target (Orin ≠ A100 ≠ H100)
⚠ Critical: Layer Fusion Impact

Vertical fusion (Conv+BN+ReLU → single kernel) reduces HBM accesses from 6 to 2 — the dominant bottleneck for memory-bound inference. This single optimization typically delivers 2-3× throughput improvement on bandwidth-limited operations.

nvmsgbroker   →  Sends JSON to Kafka / Redis / MQTT / AMQP

Probe Function Pattern

Probes are the primary mechanism for custom metadata access. Attach to any element's pad to read, filter, or modify metadata at any pipeline point.

probe-function.py (Python bindings) 
def osd_sink_pad_buffer_probe(pad, info, u_data):
    gst_buffer = info.get_buffer()
    batch_meta = pyds.gst_buffer_get_nvds_batch_meta(hash(gst_buffer))

    l_frame = batch_meta.frame_meta_list
    while l_frame is not None:
        frame_meta = pyds.NvDsFrameMeta.cast(l_frame.data)

        l_obj = frame_meta.obj_meta_list
        while l_obj is not None:
            obj_meta = pyds.NvDsObjectMeta.cast(l_obj.data)

            # Access tracking ID, bbox, class, confidence
            track_id  = obj_meta.object_id
            class_id  = obj_meta.class_id
            conf      = obj_meta.confidence
            bbox      = obj_meta.tracker_bbox_info.org_bbox_coords
            # bbox.left, bbox.top, bbox.width, bbox.height

            l_obj = l_obj.next
        l_frame = l_frame.next

    return Gst.PadProbeReturn.OK

INT8 Calibration Algorithms

TensorRT INT8 Calibration — Algorithm Comparison Quantization 
FP32 Activation Distribution              → Find threshold T →     INT8 Mapping
                                                                    scale = 127 / T

ENTROPY (IInt8EntropyCalibrator2)          MINMAX
─────────────────────────────────          ──────
  Collect activation histograms             T = max(|activations|)
  across calibration dataset
  T = argmin KL(P_fp32 || Q_int8)           Simple but outlier-
                                            sensitive
  ┌──────────────────────┐
  │   ╱╲                 │                  PERCENTILE
  │  ╱  ╲                │                  ──────────
  │ ╱    ╲    ┊← T       │                  T = CDF⁻¹(99.9%)
  │╱      ╲___┊__        │                  Robust to outliers
  └──────────────────────┘

MSE (Most Compute-Intensive)
────────────────────────────
  T = argmin Σ(x_fp32 − Q(x, T))²
  Exhaustive search over threshold space
  Directly minimizes quantization error
  Best accuracy, slowest calibration

NOTE: Calibration targets activations only.
Weight ranges derived directly from parameters.
🎯

Entropy (Default for CNNs)

Minimizes KL divergence between FP32 and INT8 distributions. Best general-purpose calibrator. IInt8EntropyCalibrator2 is the recommended default.

📊

MinMax (NLP recommended)

Uses absolute max activation value. Simple and fast but sensitive to outliers. Recommended for transformer/NLP models where activation distributions are more uniform.

📈

Percentile

Uses CDF percentile (typically 99.9%) as threshold. Good middle ground — robust to outliers while preserving dynamic range for the majority of activations.

🔬

MSE (Exhaustive)

Searches entire threshold space minimizing mean squared error. Most compute-intensive but directly optimizes the quantization error metric. Use when accuracy is paramount.

Layer Fusion — The Core Optimization

Layer fusion is the single most impactful TensorRT optimization. It reduces HBM (High Bandwidth Memory) accesses — the dominant bottleneck for memory-bound inference workloads.

TensorRT Layer Fusion Patterns Optimization 
VERTICAL FUSION — Merges sequential operations into single kernels

  Before (6 HBM accesses):         After (2 HBM accesses):

  ┌──────┐ read                     ┌─────────────────┐ read
  │ Conv │──▶ write                  │  CBR Fused      │──▶ write
  └──┬───┘                            │  Kernel         │
     ▼                                │                 │
  ┌──────┐ read                     │  Conv+BN+ReLU   │
  │  BN  │──▶ write                  │  in one pass    │
  └──┬───┘                            └─────────────────┘
     ▼
  ┌──────┐ read                     ~3× throughput improvement
  │ ReLU │──▶ write                  on memory-bound layers
  └──────┘

  Other vertical patterns:
    Conv + Add (residual) + ReLU  →  CRA kernel
    MatMul + Bias + Activation    →  single GEMM kernel
    ElementWise + ReLU            →  fused EW kernel


HORIZONTAL FUSION — Aggregates parallel ops with shared inputs

  Before (Inception-style):        After:

          input                             input
       ╱    │    ╲                            │
  ┌────┐ ┌────┐ ┌────┐            ┌──────────────┐
  │1×1 │ │1×1 │ │1×1 │            │ Wide 1×1 CBR │
  │CBR │ │CBR │ │CBR │            │ (concatenated│
  └────┘ └────┘ └────┘            │  weights)    │
       ╲    │    ╱                  └──────────────┘
         concat                           │
                                       output


ELIMINATION — Remove redundant operations

    Consecutive transposes  →  eliminated
    Dead layers (no output) →  removed
    Constant expressions    →  folded at build time

CANNOT AUTO-FUSE:
    BMM-Softmax-BMM (attention)  →  requires custom plugin
    Complex multi-head attention →  FlashAttention plugin

Backend Framework

🔥

TensorRT Backend

Highest throughput for NVIDIA GPUs. Loads .plan engine files directly. Supports all TensorRT features including INT8, dynamic shapes, and multi-profile.

🔶

PyTorch Backend

Loads TorchScript or torch.compile models. Supports GPU execution. Useful for rapid prototyping without TensorRT conversion step.

🔷

ONNX Runtime Backend

Cross-framework support via ONNX. CUDA execution provider for GPU. Good for models from multiple training frameworks.

🐍

Python Backend

Run arbitrary Python code as a model. Custom pre/post-processing, data transformation, business logic. Highest flexibility, lowest raw performance.

🔗

BLS (Business Logic Scripting)

Orchestrate multiple models programmatically from Python. More flexible than ensemble configs — supports conditional logic, loops, and dynamic routing.

⚡

vLLM Backend

Optimized large language model serving with PagedAttention. Continuous batching for maximum throughput on LLM inference workloads.

  OLD: Train PeopleNet → Deploy → Only detects people/face/bag
       Train VehicleNet → Deploy → Only detects vehicles
       Train custom model → Deploy → Only detects custom objects

  NEW: Deploy Cosmos Reason once → Ask any question in natural language
       No training, no labeling, no per-task models
       "Zero-shot" video understanding
Key Insight: CNN vs VLM Trade-offs

CNNs still win on raw throughput (1000+ FPS on dense detection), latency (<5ms), and edge deployment (Jetson Orin). VLMs win on flexibility (zero-shot), setup time (minutes vs weeks), and handling novel/unexpected events. Production Metropolis deployments increasingly use hybrid architectures: CNN pipelines for high-frequency detection with VLM reasoning for complex event understanding and summarization.

Future: Cosmos Transfer & Jetson Thor

🌐

Cosmos Transfer

Bridges 3D simulation (Omniverse/Isaac Sim) to photorealistic video. Enables training on synthetic data that looks real — critical for rare event detection where real training data is scarce (safety violations, equipment failures).

🤖

Jetson Thor

Next-gen edge SoC with transformer engine and up to 800 TOPS. Enables running VLMs (Cosmos Reason, VILA) directly on edge hardware — no cloud dependency for agentic video understanding.

🧠

Cosmos Reason

Physical world reasoning model. Understands spatial relationships, temporal sequences, and causal chains in video. "The forklift turned left, blocking the pedestrian path, causing the worker to step into the vehicle lane."

📹

VILA

Multi-frame video language model. Dense captioning, video QA, temporal grounding. Processes multiple frames to understand actions and events across time — not just single-frame analysis.
