Exponential Backoff with Jitter

Why pure exponential is a stampede, four jitter modes, retry budgets, and the math of storms.

5.1 Why Pure Exponential Backoff Fails

When N clients all hit a downed service, all retry at t=2s, then t=4s, then t=8s — synchronized waves. Each retry storm is as bad as the original outage.

The fix: add randomness (jitter) so retries spread out.

5.2 The Four Modes (Marc Brooker's classification)

import random

def no_jitter(attempt, base=0.1, cap=60):
    return min(cap, base * (2 ** attempt))

def full_jitter(attempt, base=0.1, cap=60):
    return random.uniform(0, min(cap, base * (2 ** attempt)))

def equal_jitter(attempt, base=0.1, cap=60):
    bound = min(cap, base * (2 ** attempt))
    return bound / 2 + random.uniform(0, bound / 2)

def decorrelated_jitter(prev_sleep, base=0.1, cap=60):
    return min(cap, random.uniform(base, prev_sleep * 3))

Marc Brooker's analysis shows full jitter beats the others on convergence speed and tail load reduction in nearly all simulations. Reference: https://aws.amazon.com/builders-library/timeouts-retries-and-backoff-with-jitter/

5.3 Bounded Backoff vs. Open-Ended

Always cap retries: max attempts (e.g., 5) AND max wait (e.g., 60s). Beyond cap → fail and let upstream decide.

5.4 Retry Budgets (Google SRE)

Per the SRE book: each client maintains a moving-window of (requests, accepts). Retry probability:

p_retry = max(0, (requests - K * accepts) / (requests + 1))

with K = 2 typically. When the backend is healthy, accepts ≈ requests, p_retry ≈ 0 → retries are essentially free (rare). When backend is unhealthy, accepts drop, retries get suppressed adaptively.

Reference: https://sre.google/sre-book/handling-overload/

This prevents the canonical retry-storm scenario where 1 retry per failed call doubles upstream load — making outage worse.

5.5 Finagle / Resilience4j RetryBudget

  • Finagle (Twitter): time-bounded budget. e.g., "no more than 20% of requests over the last 10s can be retries."
  • Resilience4j: similar; configurable.

5.6 Retry-After Header Compliance

If server returns Retry-After: <delta-seconds> (RFC 7231), respect it instead of computing local backoff. This lets server steer load — critical for AWS / Stripe / GitHub APIs at limits.

5.8 Quantitative Storm Analysis

Suppose backend can handle 1000 RPS. Outage causes all clients to fail. Without jitter and with 2^n backoff, all clients retry at t = 1, 2, 4, 8, .... At each spike, request rate equals N (number of clients). With N=10,000 clients, every retry instant is 10× over capacity → cascading failure, slow recovery.

With full jitter: retries are uniformly distributed in [0, 2^n). At n=3 (window 8s), 10,000 clients spread over 8s → 1250 RPS. Still over capacity but recoverable. By n=5 (window 32s) → 312 RPS, under capacity.

This is why AWS SDKs default to full jitter with exponential backoff.

5.9 Per-AWS Architecture Blog Recommendations

From Marc Brooker's Builders' Library article: - Cap retries at 3-5 (ideally on the client) — past that, fail and tell upstream. - Cap total time waiting (e.g., 60s aggregate) — beyond that, the user has likely abandoned. - Use full jitter unless you have a specific reason for decorrelated. - Apply circuit breakers at the next layer up (don't stack retries).

5.10 Stacked Retry Anti-Pattern

If layer A retries 3× and layer B (which A calls) also retries 3×, then a single user request can produce 9 backend calls. With three layers: 27. This is how "retry storms" form. Solutions: - Single retry layer: pick one layer to own retries; others propagate failures. - Idempotency keys so retries don't double-bill. - Retry budgets at every layer to cap retry overhead.

Reference: https://thomwright.co.uk/2022/09/12/retries-upon-retries/ Reference: https://thinhdanggroup.github.io/retry-without-thundering-herds/

5.11 Decorrelated Jitter (When to Use)

def decorrelated_jitter(prev_sleep, base=0.1, cap=60):
    return min(cap, random.uniform(base, prev_sleep * 3))

Each sleep is chosen relative to previous sleep, not to the attempt count. Properties: - More uniform load over time. - Tail (max wait) bounded. - Slightly slower convergence than full jitter.

Used in some Stripe SDKs, some retry libraries that want bounded tail behavior.

5.12 Idempotency Interaction

Backoff + retries only works if the operation is idempotent. Otherwise, retries cause duplicate side effects. See §6 for idempotency patterns. Always pair: backoff → retry → idempotency key.

5.13 Client vs Server Side

Two complementary places to throttle: - Client-side: backoff, jitter, retry budgets. Reduces wasted server work. - Server-side: rate limiting, load shedding (return 429/503). Protects server from any client misbehaving.

Don't rely on only one. Server-side is the last line of defense.

5.14 Going Beyond: Adaptive Backoff

For long-running API integrations: - Track per-endpoint success rate. - Adjust base or cap based on observed health. - On extended outage, switch to circuit-breaker mode: 0 retries until a probe succeeds.

This is what client SDKs like the Go AWS SDK v2 or aws-sdk-go-v2 do internally — they ship with adaptive retry.
