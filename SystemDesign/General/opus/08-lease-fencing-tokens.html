Lease & Fencing Tokens

Kleppmann's GC-pause trap, etcd revisions, ZooKeeper sequences, GFS chunk versions — making leases actually correct.

8.1 The Problem (Kleppmann's Famous Example)

A client acquires a distributed lock for "I'm writing to file X". GC pause stalls the client for 30s. Lock expires. Another client acquires it and writes. The first client wakes up, doesn't know the lock expired, and also writes — corruption.

The solution: every lock acquisition assigns a monotonically increasing fence token. The downstream resource (storage, queue, db) only accepts writes with token ≥ last seen token.

   Client 1: acquire() → token=33
       │
       │ (GC pause 30s)
       │
       ▼ resume, write(token=33)   ← REJECTED, last seen=34

   Client 2:                acquire() → token=34
                                   │
                                   ▼ write(token=34) → ACCEPTED
                                                       last seen=34

Reference: Martin Kleppmann, How to do distributed locking, https://martin.kleppmann.com/2016/02/08/how-to-do-distributed-locking.html

8.2 The Fencing Contract

Formally: - The lock service emits a token t per acquisition such that t > all previously emitted tokens. - The protected resource maintains last_token and rejects any write where incoming_token < last_token. - After accepting, last_token = incoming_token.

This converts an unsafe lease (which only works if clients never miss expirations) into a correct protocol — even with arbitrary GC pauses.

8.3 Implementation in Real Systems

etcd

// CAS-based fencing in etcd v3
resp, err := cli.Txn(ctx).
    If(clientv3.Compare(clientv3.Value("lock"), "<", strconv.Itoa(myToken))).
    Then(clientv3.OpPut("lock", strconv.Itoa(myToken))).
    Commit()
if !resp.Succeeded {
    // someone with newer token already wrote; abort
}

The revision field of any etcd write is a global, monotonically increasing 64-bit counter — a natural fence token. Etcd's election uses this directly.

ZooKeeper

EPHEMERAL_SEQUENTIAL znodes give each client a strictly increasing sequence number on creation. The client with the lowest number holds the lock. The czxid (creation transaction ID) of the znode is also globally unique and monotonic — usable as fence token.

Reference: https://medium.com/@m.hassan.def/distributed-locking-using-zookeeper-e6ec7d84feb3

Google Chubby

Chubby explicitly returns a "sequencer" — a string that includes the lock instance ID and generation. The downstream service validates the sequencer against its known-most-recent value. Chubby also has lock-delay (anti-pattern by itself; works only because of sequencers).

Google File System (GFS)

Each chunk has a current "primary" with a chunk lease. The lease has an associated chunk version number. Replicas only accept writes carrying the current version. On primary failover, version is incremented.

Reference: https://hemantkgupta.medium.com/insights-from-paper-the-google-file-system-1c9d7b32c8cc

HBase / Bigtable

RegionServer fencing: when the master detects a stale RegionServer, it removes it from the "live" znode set. Clients verify region location with HBase Master before writes; stale-cache writes get a NotServingRegionException.

8.4 Anti-Patterns

Wall-Clock Leases without Fence Tokens

A lock service that says "you have this lock for 30s" relies on synchronized clocks. NTP skew, VM pauses, and clock drift make this unsafe. Always pair lease + fence token.

Redlock

Antirez's Redlock algorithm proposed using N Redis instances quorum. Kleppmann criticized it: no fence tokens, relies on clock assumptions. Antirez responded; debate ongoing. Most production systems prefer Raft-backed (etcd/Consul) for safety.

References: - Kleppmann critique: https://martin.kleppmann.com/2016/02/08/how-to-do-distributed-locking.html - Antirez response: http://antirez.com/news/101

Non-Monotonic Tokens Across Leadership

If your lock service has a leader, and a new leader emits tokens starting from 0 → bug. The token must be globally monotonic (etcd's revision is; many homegrown systems aren't).

8.5 Fence Token in the Queue's Lease Token

In §7, the lease_token field is exactly a fence token. The ack ack(msg_id, token) is a CAS: if state.in_flight[msg_id].lease_token == token: commit; else reject. This is the fence pattern applied to message processing.

Other places this pattern shows up: - DynamoDB ConditionExpression — write only if some attribute equals expected. - S3 conditional writes — If-Match: ETag. - Postgres UPDATE ... WHERE version = X RETURNING * (optimistic concurrency).

8.7 Worked Race Condition Walkthrough

Setup: distributed key-value store with optimistic concurrency. Two clients hold a "write lock" on key K; lease expires after 30s.

Without fence tokens:

   T=0    Client A acquires lease(K), expires at T=30
   T=15   Client A starts writing; STW GC pause begins
   T=30   A's lease expires (A doesn't know — still paused)
   T=31   Client B acquires lease(K)
   T=32   B writes value V_B, commits
   T=45   A wakes up, writes value V_A — overwrites V_B
   ───────────────────────────────────────────────────────
   Result: silent corruption. B's write lost.

With fence tokens:

   T=0    A acquires lease(K) → token=42
   T=15   GC pause starts on A
   T=31   B acquires lease(K) → token=43
   T=32   B writes V_B with token=43; KV updates last_token[K]=43
   T=45   A wakes; writes V_A with token=42
          KV checks: 42 < last_token[K]=43 → REJECT
   ───────────────────────────────────────────────────────
   Result: correct. A's stale write rejected. B's write intact.

The key property: last_token is monotonically non-decreasing on the resource side, even across leadership changes. The lock service alone is insufficient — the resource must enforce the invariant.

8.8 etcd Reference Implementation

// Acquire lease + fence token
session, err := concurrency.NewSession(client, concurrency.WithTTL(30))
if err != nil { return err }
defer session.Close()

mutex := concurrency.NewMutex(session, "/locks/key")
ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
defer cancel()

if err := mutex.Lock(ctx); err != nil { return err }

// The mutex's revision IS the fence token
fence := mutex.Header().Revision

// Use it in any subsequent write (CAS):
resp, err := client.Txn(ctx).
    If(clientv3.Compare(clientv3.Value("lock_holder"), "<", strconv.FormatInt(fence, 10))).
    Then(clientv3.OpPut("lock_holder", strconv.FormatInt(fence, 10)),
         clientv3.OpPut("data:K", value)).
    Commit()
if !resp.Succeeded {
    return ErrFenced
}

Notes: - mutex.Header().Revision is etcd's internal modification revision, monotonic across leader changes. - The CAS condition embeds the fence into the resource itself (lock_holder key). - Subsequent writes by an older holder fail the Compare.

8.9 ZooKeeper EPHEMERAL_SEQUENTIAL Recipe

// In ZK: create ephemeral sequential znode under /lock/
String path = zk.create("/lock/req_", null, OPEN_ACL_UNSAFE,
                       CreateMode.EPHEMERAL_SEQUENTIAL);
// path = e.g. "/lock/req_0000000042"

long mySeq = parseSeq(path);
List<String> kids = zk.getChildren("/lock", false);
Collections.sort(kids);

if (kids.get(0).equals(getName(path))) {
    // I am the lowest, I hold the lock
    long fence = mySeq;          // monotonic seq is the fence
    doWork(fence);
} else {
    // Watch for predecessor
    int idx = kids.indexOf(getName(path));
    String predecessor = "/lock/" + kids.get(idx - 1);
    zk.exists(predecessor, watcher);
    // ... wait
}

Properties: - ZooKeeper czxid (creation transaction ID) is also monotonic and globally unique — alternative fence token. - Ephemeral nodes auto-delete on session loss (handles client crashes).

8.10 Chubby Sequencer Detail

Chubby (Burrows, OSDI 2006) returns a sequencer on lock acquisition:

   sequencer = {
       lock_name: "/db/users",
       mode: WRITE,
       generation: 12345,           // global-monotonic
       epoch: 7
   }

The sequencer is a string that the client passes to every protected operation. The protected service: 1. Checks generation against last seen for this lock. 2. If incoming generation > last_seen: accept, update. 3. If less: reject as stale.

Chubby also has lock-delay: when a client's session ends, hold the lock unavailable for lock_delay seconds (e.g., 60s). This is a HACK that only works because Chubby has sequencers — without them, lock_delay alone wouldn't prevent split-brain.

8.11 GFS Chunk Lease + Version

Each chunk has: - A primary replica (current write coordinator). - A chunk version: incremented every time the master grants a new lease.

   Master grants lease to replica A: version 5.
   A becomes primary, replicas {A, B, C}.
   Master fails. Re-elected master.
   Master notices A unreachable; revokes lease, increments version → 6.
   Master grants lease to B: version 6.
   B becomes primary, replicas {B, C}.
   Old primary A wakes up, tries to write at version 5.
   Replicas reject: their version is 6.

The chunk version is etched into both the primary and replicas' state. Stale primary's writes are rejected at the replica level.

8.12 The Redlock Controversy

Antirez (Redis creator) proposed Redlock: acquire lock from majority of N independent Redis instances. Kleppmann's critique: 1. No fence tokens: any of the cited issues (GC pause, network delay) cause split-brain. 2. Clock-bound assumptions: relies on clocks not drifting. 3. Failure modes: half-successful acquisitions need garbage collection.

Antirez's response: Redlock is reasonable for non-correctness-critical coordination. For correctness, you need something else (etcd, ZK, fence tokens at resource).

References: - Kleppmann critique: https://martin.kleppmann.com/2016/02/08/how-to-do-distributed-locking.html - Antirez response: http://antirez.com/news/101

The takeaway for interviews: lock services are not safe alone. Always pair with fence tokens at the resource.

8.13 Where Fence Tokens Don't Help

  • External, non-cooperative resources: if the resource doesn't (or can't) check tokens, you can't fence.
  • Example: shelling out to aws s3 cp — S3 itself doesn't know about your lease, so can't reject stale writes. Workaround: use S3 conditional writes (If-Match) or write with object-version-aware logic.
  • Non-deterministic side effects: sending an email, charging a credit card. You can fence the DB record but the email already went. Mitigate with idempotency keys at the email service.

8.14 Practical Pattern: Storage-Backed Fencing

If your resource is Postgres:

UPDATE document
SET content = $1, fence = $2
WHERE id = $3 AND fence < $2;

The WHERE fence < $2 is the fence check; RETURNING * lets the client know if the write committed.

For DynamoDB:

   UpdateItem with ConditionExpression: 'fence < :new_fence'

For S3:

   Use S3 Versioning + If-Match: <ETag>
   On write conflict, the old version's ETag is stale → write rejected.

8.15 Consensus-Backed Fence Tokens in Practice

The fence token can be: - A consensus log index (Raft/Paxos) — globally monotonic, robust. - A timestamp from a strongly synchronized clock (Spanner TrueTime) — more advanced, requires hardware. - A causal ID + counter — see Cosmos DB session tokens.

Spanner's TrueTime gives you a monotonic time-bound; combined with consensus, this is the strongest form. Most systems can't get to this; etcd's revision is the practical equivalent.

8.16 Anti-Patterns

  • Wall-clock-only leases: if now() < lease_expires_at — clock skew breaks this.
  • Token = random UUID: not monotonic; can't compare. Use ascending sequence.
  • Storing fence in client memory only: rebuilding a lease across restarts must persist token to durable storage.
  • Resetting tokens on leader change: the new leader must continue from where the old left off, not restart at 0.
  • Async write with stale token: if the resource async-applies writes, fence check might happen later than the lock check. Apply atomically.

8.17 Common Interview Follow-ups

Q: What if the resource doesn't support fencing? A: Two choices: (1) put a fencing proxy in front (e.g., a service that checks token before forwarding), (2) make operations idempotent so duplicate writes are harmless.

Q: How do fence tokens differ from optimistic concurrency control? A: They're related. OCC uses a per-record version; fence tokens use a per-lease global counter. Fence tokens are stronger because they prevent ANY operation under a stale lease, not just operations on a specific record.

Q: Can I use TTLs (time-to-live) instead of fence tokens? A: Only if you can guarantee no clock drift, no GC pauses, no network delays — i.e., never. TTLs are best-effort; fence tokens are correctness guarantees.

Q: Can fence tokens replace consensus? A: No — you still need consensus to generate monotonic tokens reliably. Fence tokens are a use of consensus output, not a replacement for consensus.
