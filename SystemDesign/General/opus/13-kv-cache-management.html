13.1 What KV Cache Is

In a Transformer decoder, attention at step t reads keys K_1...K_t and values V_1...V_t. To avoid recomputing K and V for tokens 1..t-1 each step, we cache them. The cache size grows linearly with sequence length and dominates GPU memory in long-context workloads.

Per-token KV size formula:

   KV_per_token = 2 (K and V)
                × num_layers
                × num_kv_heads
                × head_dim
                × bytes_per_element (2 for fp16, 1 for fp8)

Worked examples:

Model layers kv_heads head_dim per-tok bytes per-tok @ 128k
Llama-2-70B (MHA) 80 64 128 2.6 MB 320 GB
Llama-3.1-70B (GQA-8) 80 8 128 320 KB 40 GB
Llama-3.1-8B (GQA-8) 32 8 128 128 KB 16 GB
Mistral 7B (SWA-4096) 32 8 128 128 KB capped at 4096 ≈ 512 MB (capped)

GQA (Grouped-Query Attention) and SWA (Sliding Window Attention) are the two memory-saving architectural changes. CLA (Cross-Layer Attention) and MLA (Multi-Latent Attention, DeepSeek) push further.

Variant KV memory Quality
MHA full best
MQA (1 KV head) 1/H small drop
GQA (G KV heads) G/H matches MHA
SWA constant in seq len local context only
CLA shared across layer pairs small drop
MLA (DeepSeek) compressed via low-rank projection matches MHA

13.2 PagedAttention (vLLM)

Classical KV layout: contiguous tensor per request. Wastes memory when requests have different lengths (internal fragmentation) and when memory is freed and re-allocated (external fragmentation).

PagedAttention: KV is stored in fixed-size blocks (e.g., 16 tokens each). Each request has a "block table" pointing to its blocks. New blocks allocated as the sequence grows. Memory is paged like an OS — minimal fragmentation.

   Request R1, seq_len=40, block_size=16:
     block_table = [3, 17, 5]   (slots 0-15 in block 3, 16-31 in block 17, 32-39 in block 5)

   Logical view of K, V for R1:
   [K_0..K_15 | K_16..K_31 | K_32..K_39]
       block 3      block 17      block 5

Shared prefix optimization (CoW): when N requests have the same prompt prefix, point all their block tables to the same blocks until divergence. This is key to high prefix-cache hit rates (Character.AI 95%).

   System prompt cached once:
     block_table_user_A = [3, 7, 11, ..., A_specific_blocks]
     block_table_user_B = [3, 7, 11, ..., B_specific_blocks]
                          ^^^^^^^^^^^^
                          shared blocks (refcounted)

Reference: https://blog.vllm.ai/2023/06/20/vllm.html

13.3 Eviction & Preemption

When KV cache is full and a new request needs admission, an existing request must be preempted:

  • vLLM v0 swap: copy KV blocks to CPU pinned memory; swap back on resume.
  • vLLM v1 recompute: free blocks, re-prefill on resume. Faster on H100 (NVLink swap is bottlenecked).
  • KVFlow / priority-aware: evict by priority (free-tier first).

Eviction policies: - LRU on request: simple, common. - Oldest-prefill-first: prefer evicting requests that haven't started decoding (sunk cost is lower). - By tier: preserve pro requests; evict free first.

13.4 Prefix Cache Hashing

Prefix cache key = hash of (model_id, prompt_token_ids[0:N]). Look up before allocation; on hit, reuse blocks.

Hash chaining:

   h_0 = hash(model_id)
   h_1 = hash(h_0, token_1)
   h_2 = hash(h_1, token_2)
   ...
   h_n = hash(h_{n-1}, token_n)

Block boundary hashes are stored. New request with same prefix → block table starts with cached blocks, refcount++.

13.5 RadixAttention (SGLang)

SGLang generalizes prefix caching to a radix tree of prefixes. Two requests sharing prefix P then diverging into A and B share a root → P → split into A and B subtrees. Higher hit rate than block-aligned hashing.

Reference: https://lmsys.org/blog/2024-01-17-sglang/

13.6 Disaggregated KV (DistServe / Mooncake)

Prefill and decode pools are separate. After prefill finishes, KV must transfer to a decode replica.

   KV size = seq_len × per-token_bytes
   For 1k tokens × 320 KB = 320 MB
   Over NVLink (600 GB/s): 320 MB / 600 GB/s ≈ 0.5 ms — hidden in pipeline.
   Over IB 400 Gb/s: 320 MB / 50 GB/s ≈ 6 ms — still fine.
   Over 25 Gb/s ethernet: too slow.

Mooncake at Kimi runs the disaggregated architecture in production on A800/H800. Their innovation: KVCache-centric scheduling — admission decisions consider whether prefix-cached KV is on the right replica, prefer those replicas.

References: - DistServe: https://arxiv.org/abs/2401.09670 - Splitwise: https://arxiv.org/abs/2311.18677 - Mooncake (FAST '25): https://www.usenix.org/conference/fast25/presentation/qin

13.7 Quantization

KV cache can be stored at lower precision: - FP16 → INT8: 2× savings, < 0.5% quality loss with proper scaling. - INT8 → INT4: 4× savings, ~1-2% quality loss; more careful calibration needed. - FP8 (E4M3 or E5M2): hardware-accelerated on H100/H200; 2× savings.

Character.AI: INT8 KV quantization with custom attention kernel (based on FlashAttention 3). Reduced KV memory 20× combined with MQA + CLA + SWA.

13.8 Memory Budget Math

Suppose 80GB H100, model weights = 140GB (so two H100 with TP=2, ~70GB per GPU). Available KV per GPU = 80 - 70 = 10 GB. With 320 KB/token (Llama-3.1-70B), that's 30k tokens per GPU. With 2 GPUs and TP=2 sharing, ~60k tokens total active. Suppose average request = 4k context + 500 output → 4500 tokens per request → 13 concurrent requests max.

This is why prefix cache + GQA + INT8 are essential — without them, you serve almost no concurrency.

13.10 Block Table Representation Detail

Each request has an associated block table — a list of physical block IDs that store its KV.

# vLLM-style block table
class Sequence:
    def __init__(self, prompt_tokens, block_size=16):
        self.tokens = list(prompt_tokens)
        self.block_size = block_size
        self.block_table = []      # list of physical block ids

    def num_blocks(self):
        return (len(self.tokens) + self.block_size - 1) // self.block_size

class BlockManager:
    def __init__(self, num_blocks, block_size):
        self.free_blocks = list(range(num_blocks))
        self.block_size = block_size
        self.refcount = {}        # block_id → count (for shared blocks)

    def allocate(self, seq):
        needed = seq.num_blocks() - len(seq.block_table)
        for _ in range(needed):
            if not self.free_blocks:
                return False  # OOM
            blk = self.free_blocks.pop()
            seq.block_table.append(blk)
            self.refcount[blk] = 1
        return True

    def share_with(self, parent_seq, child_seq, num_shared_blocks):
        """Copy-on-write: child shares parent's first N blocks."""
        for i in range(num_shared_blocks):
            blk = parent_seq.block_table[i]
            child_seq.block_table.append(blk)
            self.refcount[blk] = self.refcount.get(blk, 1) + 1

    def fork_on_write(self, seq, block_idx):
        """If a shared block is being written to, copy it first."""
        old_blk = seq.block_table[block_idx]
        if self.refcount[old_blk] > 1:
            new_blk = self.free_blocks.pop()
            self._copy_block(old_blk, new_blk)
            seq.block_table[block_idx] = new_blk
            self.refcount[old_blk] -= 1
            self.refcount[new_blk] = 1

    def free(self, seq):
        for blk in seq.block_table:
            self.refcount[blk] -= 1
            if self.refcount[blk] == 0:
                self.free_blocks.append(blk)
                del self.refcount[blk]
        seq.block_table.clear()

The refcount is critical for prefix sharing — multiple sequences can share the same physical blocks until one of them writes (then COW).

13.11 Prefix Cache Hit Detection

class PrefixCache:
    def __init__(self):
        self.cache = {}  # hash → (block_ids, last_used_ts)

    def lookup(self, token_prefix):
        h = self._hash_chain(token_prefix)
        if h in self.cache:
            block_ids, _ = self.cache[h]
            self.cache[h] = (block_ids, time.time())  # update LRU
            return block_ids
        return None

    def store(self, token_prefix, block_ids):
        h = self._hash_chain(token_prefix)
        self.cache[h] = (block_ids, time.time())

    def evict_lru(self, n):
        sorted_keys = sorted(self.cache.items(), key=lambda kv: kv[1][1])
        for k, _ in sorted_keys[:n]:
            del self.cache[k]

    def _hash_chain(self, tokens):
        h = 0
        for t in tokens:
            h = hash((h, t))
        return h

Production systems (vLLM, SGLang) use block-aligned hashing: hash every block_size tokens, store hashes per block. Lookup walks blocks one at a time, allowing partial prefix matches.

13.12 RadixAttention Tree Structure

SGLang's RadixAttention generalizes block-level prefix caching:

              ROOT
               │
        ┌──────┴──────┐
        │             │
       "You"        "Hello"
        │             │
   ┌────┴────┐    ┌───┴───┐
   "are"   "can"  ...     ...
    │       │
   ...     ...

Each node = a token sequence (one or more tokens). Nodes with shared prefix share the same path. Lookups: descend from root following query tokens; longest matching path = cached prefix.

Gives higher hit rate than block-aligned hashing because matches can be sub-block. Reference: https://lmsys.org/blog/2024-01-17-sglang/

13.13 Eviction Policy Decisions

When KV is full and a new request arrives:

def select_victim(running, waiting):
    candidates = [r for r in running if r.priority == "free"]
    if not candidates:
        candidates = sorted(running, key=lambda r: -r.tokens_remaining)
    # Prefer evicting requests with most remaining work (lowest sunk cost progress)
    return min(candidates, key=lambda r: r.tokens_generated_so_far)

Production heuristics: - Evict by tier (free first). - Within tier, evict by sunk-cost-recovered (a request that's 90% done won't be evicted; a request that just started, more likely). - Evict whole sequences, not partial — partial KV can't be reused easily.

13.14 Quantization Math

INT8 KV cache: store K and V as int8 with per-channel scale.

   K_int8 = round(K_fp16 / scale_K)
   K_fp16_recovered = K_int8 * scale_K

Scale chosen to minimize quantization error per (layer, head) channel. Calibration on validation prompts.

INT4: paired (or grouped) quantization to reduce overhead. Per-group scales (e.g., one scale per 64 channels) keep accuracy.

FP8 (E4M3 or E5M2): native on H100. E4M3 (4-bit exponent, 3-bit mantissa) for activations; E5M2 for gradients. Arithmetic done in FP8 then accumulated in FP32.

Reference: https://arxiv.org/abs/2208.09225 (FP8 paper from NVIDIA + Intel + ARM)

13.15 Production Numbers — Character.AI

Character.AI's blog reports a 20× KV memory reduction: - MQA: 1/16 of MHA (16 query heads, 1 KV head). - Cross-Layer Attention (CLA): 1/2 (sharing KV across layer pairs). - Sliding Window Attention: ~1/4 effective at 8k context. - INT8 KV: 1/2.

Combined: 1/16 × 1/2 × 1/4 × 1/2 = 1/256 of MHA fp16. Practical savings closer to 20-25× because of overhead and quality-preserving constraints.

Custom kernel: based on FlashAttention-3, modified to handle their attention layout. ~95% prefix cache hit rate at the application layer.

Reference: https://research.character.ai/optimizing-ai-inference-at-character-ai-part-deux/

13.16 Disaggregated KV Transfer Mechanics

Prefill on GPU A, decode on GPU B:

   1. A finishes prefill of request R; KV resides on A's HBM.
   2. Router publishes assignment: R → B.
   3. KV transfer initiated:
      - intra-node: NVLink (600 GB/s on H100 NVL) — ~1ms for 600MB
      - inter-node: InfiniBand (400 Gb/s = 50 GB/s) — ~12ms for 600MB
      - across DC: too slow; don't disaggregate across DCs.
   4. B receives KV, begins decode.

Key optimization: overlap transfer with prefill of next request. Mooncake reports 100% overlap on Kimi production.

When disaggregation isn't worth it: short prompts (<256 tokens) where prefill is fast and decode is the dominant cost — colocated is fine.

13.17 Memory Pressure Warning Levels

Production systems define thresholds:

Threshold KV Occupancy Action
Green < 70% accept all
Yellow 70-85% accept high-tier only
Orange 85-95% preempt low-tier; reject mid-tier
Red > 95% reject all new; preempt aggressively

Anti-pattern: ignore until OOM, then crash the replica. Always shed load before OOM.

13.18 Common Interview Follow-ups

Q: How much KV does a 70B model need at 100k context? A: 320KB/token × 100k = 32GB just for KV. With 8-bit quantization: 16GB. With MQA: 4GB. Architecture matters more than quantization for long context.

Q: Why not just store KV on disk? A: Decode is memory-bandwidth-bound. Reading 32GB of KV per token from NVMe (~7GB/s) takes 4.5s/token. NVMe is 100× slower than HBM. Disk-based KV is only viable for very long contexts where you also tolerate batch latency.

Q: How do you handle KV-cache eviction without losing user state? A: vLLM v0 swaps to CPU pinned memory (paged from HBM). vLLM v1 recomputes from prompt. Recompute is faster on H100 because NVLink is the bottleneck.

Q: What's the relationship between KV cache and context window? A: KV cache size grows linearly with context. To support 1M-token contexts, you need MLA-like compression or sliding window, otherwise memory is prohibitive.
