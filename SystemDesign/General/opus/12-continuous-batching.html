12.1 The Problem with Static Batching

In a fixed-size batch, all requests must produce the same number of tokens. The batch finishes when the longest request finishes. Short requests sit idle waiting:

   Static batch (size 4):

   t=0  ┌────┬────┬────┬────┐    R1 wants 50 tokens
        │ R1 │ R2 │ R3 │ R4 │    R2 wants 100
   t=49 │ R1 ✓                   R3 wants 30 ✓ at t=29  but stays in batch
   t=29 │           │ R3 ✓          (idle for 70 steps)
   ...
   GPU is idle for short requests waiting for longest.

12.2 Dynamic Batching (Old)

NVIDIA Triton dynamic batching: form a batch from incoming requests every T ms or when batch is full, dispatch, return when all done. Slight improvement, still suffers from "batch finishes when longest finishes".

12.3 Continuous Batching (Orca, OSDI '22)

Per-iteration batching: at every decode step, check the active set, evict finished requests, admit new ones. Each iteration has its own batch:

   Continuous batching:

   t=0:  batch = {R1, R2, R3, R4}
   t=1:  batch = {R1, R2, R3, R4}     each does 1 step
   ...
   t=29: R3 finishes → evict
   t=30: batch = {R1, R2, R4}; admit R5 → {R1, R2, R4, R5}
   t=49: R1 finishes → evict; admit R6
   ...

Each request makes progress; GPU is fully utilized; new requests admitted as soon as KV cache has room. Orca reports 36.9× throughput vs. FasterTransformer.

12.4 vLLM V1 Scheduler

vLLM v1 (released Jan 2025) unified prefill and decode into a single primitive. Each iteration step admits a token budget:

   schedule_step(token_budget=512):
       admit_set = {}
       remaining = token_budget

       # 1. Continue running requests (decode = 1 token each)
       for req in running:
           admit_set[req.id] = 1
           remaining -= 1

       # 2. Resume preempted requests (prefill from saved tokens)
       for req in preempted:
           if remaining >= req.remaining_prefill:
               admit_set[req.id] = req.remaining_prefill
               remaining -= req.remaining_prefill

       # 3. Admit new requests up to budget
       for req in waiting:
           chunk = min(req.prompt_len, remaining)
           admit_set[req.id] = chunk
           req.prompt_offset += chunk
           remaining -= chunk
           if remaining == 0: break

       return admit_set

V1 dropped CPU swap (V0 had it); recompute is faster than swap on H100. KV-cache pages of preempted requests are simply freed; on resume, the prefill chunks re-run.

12.5 Sarathi-Serve: Stall-Free Chunked Prefill

The interleave problem: a long prefill (say, 16k tokens) takes 100 ms in one iteration, blocking all decodes. Sarathi-Serve splits prefill into chunks and interleaves with decodes:

   Naïve:
   t=0   prefill R_new (16k tokens, 100ms)  ← all decodes blocked
   t=100 decode all  (10ms)

   Sarathi-Serve (token_budget=512):
   t=0   prefill R_new chunk 1 (512) + decode 8 in same fwd pass (~4ms)
   t=4   prefill R_new chunk 2 (512) + decode 8 (~4ms)
   ...
   t=128 prefill done; all decodes have made steady progress

Every iteration packs chunk_prefill_tokens + decode_tokens ≤ token_budget. TBT (time-between-tokens) is bounded; throughput maintained.

Token budget tuning (Sarathi paper): - 7B model on A100: 256. - 70B model on 4×A100: 512. - 180B Falcon: 5.6× capacity at strict SLO.

Reference: https://arxiv.org/abs/2403.02310

12.6 Selective Batching (Orca paper)

The non-attention ops (LayerNorm, MLP, residual) are simple matmul/elementwise — they batch as [L_total, H] with no padding overhead, where L_total = sum of seq lens in batch. Attention is sequence-specific — done per-request via PagedAttention.

   Batch state:
     R1: seq_len=20, doing decode (1 token)
     R2: seq_len=512, doing prefill (chunk of 256)
     R3: seq_len=1, doing decode

   For non-attn ops: input is [277, H] (20+256+1) — single big matmul
   For attn ops: PagedAttention launches a kernel per request
                 (each attends to its own KV cache block list)

This is why FlashAttention and FlashInfer have specialized kernels for "varlen" — variable length attention with packed sequences.

12.7 Throughput vs TBT Curves (H100, Llama-2-70B)

Roughly: - Static batch=1: 50 tokens/s, TBT = 20 ms. - Static batch=8: 250 tokens/s, TBT = 30 ms. - Continuous batch (vLLM): 1000 tokens/s aggregate, p99 TBT = 35 ms. - + chunked prefill (Sarathi): 1500 tokens/s, p99 TBT = 25 ms.

Numbers vary heavily with prompt length distribution.

12.9 Iteration-Level Granularity (vs Request-Level)

Static batching: schedule decisions made when batch starts and ends. Continuous batching: decision at every model forward pass.

This means "selective batching" on attention but full-batch on everything else. The model is fed:

   forward_pass_input:
     embeddings = stack(all_active_tokens_this_step)   # [B_total, H]
     positions  = positions_for_each_token             # [B_total]
     block_table_per_request                            # for PagedAttention

Non-attention layers (LayerNorm, MLP, residual) compute over [B_total, H] once. Attention is invoked per request, indexing into its KV blocks.

12.10 vLLM V0 vs V1 Architectural Differences

Aspect v0 v1
Scheduler step separate paths for prefill/decode unified (req_id → n_tokens)
Preemption swap KV to CPU recompute on resume
Iteration loop Python overhead on hot path Rust core for hot path
Chunked prefill optional plugin native, default-enabled
CUDA graphs manual capture automatic

V1 is ~30% faster on average, dramatically faster on tail latency.

Reference: https://blog.vllm.ai/2025/01/27/v1-alpha-release.html

12.11 FlashAttention 1/2/3 — Why It Matters Here

Continuous batching needs fast attention because attention is invoked per-request per-step. FlashAttention (Dao et al.) reorders attention computation to fuse softmax and matmul, reducing HBM traffic from O(N²) to O(N).

FA-3 (Hopper) further uses TMA + warp specialization. Numbers (H100 Llama-2-7B, 2048 ctx): - Naive PyTorch: 32 TFLOP/s. - FA-2: 580 TFLOP/s. - FA-3: 740 TFLOP/s.

For decode (sequence length 1 query against length N KV), the throughput-relevant version is FlashDecoding / FlashInfer's varlen kernels.

References: - FA-1: https://arxiv.org/abs/2205.14135 - FA-2: https://arxiv.org/abs/2307.08691 - FA-3: https://arxiv.org/abs/2407.08608 - FlashInfer: https://github.com/flashinfer-ai/flashinfer

12.12 Sarathi-Serve Algorithm Pseudocode

def schedule(running, waiting, token_budget):
    """Compute next batch step to maintain TBT bound."""
    batch = []
    used = 0

    # Decode tokens always go first (they're cheap, 1 each)
    for r in running:
        if used + 1 > token_budget: break
        batch.append((r, "decode", 1))
        used += 1

    # Then chunk-prefill from waiting, up to budget
    for r in waiting:
        if used >= token_budget: break
        chunk = min(r.remaining_prefill, token_budget - used)
        batch.append((r, "prefill", chunk))
        used += chunk
        r.remaining_prefill -= chunk
        if r.remaining_prefill > 0:
            r.state = "running"  # will continue chunked

    return batch

Token budget tuning: - Smaller budget → tighter TBT, lower throughput. - Larger budget → higher throughput, looser TBT. - Sweet spot: 256-512 for 70B-class models on A100/H100.

The genius: chunked prefill doesn't compromise prefill latency much (it would have taken the same total time anyway), but it absolutely bounds decode TBT.

12.13 Speculative Decoding

Orthogonal optimization that integrates with continuous batching:

   Use small "draft" model to propose K tokens.
   Run main model on the K-token proposal in one fwd pass.
   Accept the longest prefix that matches.
   ────────────────────────────────────────────────
   If draft is right ~70% of the time: ~3-4× decode speedup.

Variants: vanilla speculative (Leviathan et al.), Medusa (multi-head draft), EAGLE (verifier-aware), DeepMind's StaffSeer. Production: vLLM supports it; OpenAI uses something similar internally per their dev day talks.

The scaling-on-load behavior reveals: continuous batching has near-linear throughput up to memory limits; static batching plateaus at the longest-request-length.

12.15 Production Implementations

  • vLLM: open source; widest adoption; v1 is current.
  • TensorRT-LLM: NVIDIA's; tightly coupled to TRT runtime; best perf on H100.
  • SGLang: research-driven; RadixAttention; strong for structured outputs.
  • Hugging Face TGI: continuous batching since v0.9.x; widely deployed.
  • DeepSpeed-MII: Microsoft; integrates Splitwise.
  • Friendli (PeriFlow): original Orca authors; commercial.
  • NVIDIA Triton + custom backend: lowest-level integration.

12.16 Common Interview Follow-ups

Q: Why does continuous batching work better than dynamic batching? A: Dynamic batching forms a batch and runs it to completion. Continuous batching reforms the batch every iteration — short requests exit early, GPU stays busy.

Q: What's the throughput-latency tradeoff? A: Higher batch sizes → higher throughput, higher latency per token. Sarathi's chunked prefill lets you decouple: maintain throughput at small token budgets while bounding TBT.

Q: How do you handle the batch being smaller than KV-cache budget? A: Admit more requests until KV cache fills. Continuous batching naturally drives this — when running set is small, scheduler grabs from waiting queue.

Q: When does continuous batching not help? A: When all requests have similar length (e.g., embedding generation with fixed-length output). Static batching is fine there.
