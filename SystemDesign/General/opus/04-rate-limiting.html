4.1 The Five Canonical Algorithms

4.1.1 Fixed Window Counter

   bucket = floor(now / window_size)
   if redis.incr(f"rl:{key}:{bucket}") > limit:
       return DENY
  • Pro: Trivial; minimal storage (1 int per window).
  • Con: Boundary spike — a client can do 2×limit in a 2-second span by hitting at t=window_end - 0.5s and t=window_end + 0.5s.

4.1.2 Sliding Window Log

Store every request timestamp; on each request, drop entries older than now - window, count remaining.

  • Pro: Exact.
  • Con: O(N) memory per key per window. Awful at scale.

4.1.3 Sliding Window Counter

Approximate. Two adjacent fixed windows + linear interpolation:

   curr = count[current_window]
   prev = count[previous_window]
   weight_prev = (window_size - elapsed_in_curr) / window_size
   approx_count = prev * weight_prev + curr

Cloudflare reports 0.003% wrong allow/deny on 400M requests at peak — accurate enough for almost all use cases. (blog.cloudflare.com)

4.1.4 Token Bucket

Tokens accumulate at rate r up to capacity C. Each request consumes 1 token. If empty, deny (or wait).

class TokenBucket:
    def __init__(self, rate, capacity):
        self.rate = rate; self.capacity = capacity
        self.tokens = capacity; self.last = time.monotonic()
    def try_acquire(self, n=1):
        now = time.monotonic()
        delta = now - self.last
        self.tokens = min(self.capacity, self.tokens + delta * self.rate)
        self.last = now
        if self.tokens >= n:
            self.tokens -= n; return True
        return False
  • Pro: Allows controlled bursts (up to C); steady-state is r.
  • Used by: AWS, Stripe (older), most SDK clients (tenacity, retry).

4.1.5 Leaky Bucket

Requests are inserted into a queue; queue drains at constant rate r. If queue full, deny.

  • Pro: Output rate is constant (smooths bursts).
  • Con: Adds latency (queueing); requires queue memory.
  • Used by: NGINX limit_req, traffic shapers.

4.1.6 GCRA (Generic Cell Rate Algorithm)

Constant-time, single-state-variable equivalent of leaky bucket. Used by Stripe (Lua in Redis), redis-cell module.

def gcra(now, tat, period, burst):
    # tat = "Theoretical Arrival Time" (state we keep)
    increment = period           # 1 / rate
    new_tat = max(tat, now) + increment
    allow_at = new_tat - burst * increment
    if now < allow_at:
        return False, tat        # too early; deny
    return True, new_tat

Atomic Redis Lua:

-- KEYS[1] = key  ARGV: now, period, burst
local tat = tonumber(redis.call('GET', KEYS[1]) or '0')
local now = tonumber(ARGV[1])
local period = tonumber(ARGV[2])
local burst  = tonumber(ARGV[3])
local new_tat = math.max(tat, now) + period
local allow_at = new_tat - burst * period
if now < allow_at then
    return 0
end
redis.call('SET', KEYS[1], new_tat, 'PX', math.ceil(burst * period))
return 1

4.2 Distributed Rate Limiting

Single-node bucket doesn't work when the rate-limited entity (a tenant) talks to many servers. Two designs:

Centralized (Redis)

  • All servers INCR against a Redis key.
  • Pro: Exact.
  • Con: 1 RTT per request to Redis; Redis is SPOF; ~100k ops/s ceiling per node.

Local + Sync

  • Each node has its own bucket sized as total / N.
  • Periodically sync surplus tokens (every 1s) so under-utilized nodes can lend to busy ones.
  • Pro: No per-request RTT.
  • Con: Approximate; fairness across nodes not perfect.

Lyft's ratelimit (used by Envoy): centralized Redis, gRPC service, supports hierarchical descriptors ({user_id: alice, endpoint: /api/expensive}). Used at scale at Lyft, Reddit, others.

4.3 Hierarchical Token Buckets (HTB)

For multi-tier limits (per-user, per-tenant, global):

                    ┌──────────── Global limit (10K rps) ────────────┐
                    │                                                │
              ┌─────┴─────┐                                ┌─────────┴──────┐
              │ Tenant A  │                                │   Tenant B     │
              │ (3K rps)  │                                │   (5K rps)     │
              └─┬─────┬───┘                                └────────┬───────┘
                │     │                                             │
            ┌───┘     └───┐                                          │
            ▼             ▼                                          ▼
       User a1 (1K)   User a2 (1K)                            User b1 (4K)

A request must pass all levels. Excess capacity at one level can be lent to siblings. Linux tc HTB is a classic implementation.

4.4 Per-Endpoint vs Per-Tenant vs Per-Both

In webhooks, the natural axes: - Per-tenant: protect each customer's quota. - Per-endpoint: customer endpoint has its own SLA (might be slower than tenant's quota). - Per-event-type: prioritize critical events.

Typically combine: minimum of per-tenant and per-endpoint rates.

4.6 Atomic Redis Lua for Token Bucket

-- KEYS[1] = bucket key
-- ARGV[1] = capacity, ARGV[2] = refill_rate (per sec), ARGV[3] = now_ms, ARGV[4] = tokens_requested
local capacity = tonumber(ARGV[1])
local rate     = tonumber(ARGV[2])
local now_ms   = tonumber(ARGV[3])
local need     = tonumber(ARGV[4])

local data = redis.call('HMGET', KEYS[1], 'tokens', 'last_ms')
local tokens  = tonumber(data[1]) or capacity
local last_ms = tonumber(data[2]) or now_ms

local elapsed_ms = now_ms - last_ms
tokens = math.min(capacity, tokens + (elapsed_ms / 1000.0) * rate)

local allowed = 0
if tokens >= need then
    tokens = tokens - need
    allowed = 1
end

redis.call('HMSET', KEYS[1], 'tokens', tokens, 'last_ms', now_ms)
redis.call('PEXPIRE', KEYS[1], math.ceil((capacity / rate) * 1000 * 2))
return {allowed, tokens}

Single EVAL is atomic; survives concurrent clients. Returns {allowed, remaining} for the client to display rate-limit headers.

4.7 Sliding Window Counter — Cloudflare's Math

For window of size W with two adjacent counters: - weight_prev = (W - elapsed_in_curr) / W - approx = floor(prev × weight_prev) + curr

This is exact only if traffic is uniform within each window. In bursty traffic, it can over- or under-estimate by up to one window's burst. Cloudflare measured 0.003% wrong allow/deny on 400M requests, demonstrating accuracy is fine for almost all use cases. Reference: https://blog.cloudflare.com/counting-things-a-lot-of-different-things/

4.8 GCRA Worked Example

Limit: 10 requests per second, burst 5. - period = 1/10 = 100 ms. - burst = 5.

Trace:

   t=0:   tat=0
          new_tat = max(0, 0) + 100 = 100
          allow_at = 100 - 5*100 = -400
          0 < -400? No → ALLOW; tat = 100

   t=10:  new_tat = max(100, 10) + 100 = 200
          allow_at = 200 - 500 = -300
          10 < -300? No → ALLOW; tat = 200

   ... (after 5 rapid requests at t≈0)
   t=4:   tat=400 (after 4 prior)
          new_tat = max(400, 4) + 100 = 500
          allow_at = 500 - 500 = 0
          4 < 0? No → ALLOW; tat = 500

   t=5:   new_tat = max(500, 5) + 100 = 600
          allow_at = 600 - 500 = 100
          5 < 100? YES → DENY; tat unchanged
   ...
   t=100: new_tat = max(500, 100) + 100 = 600
          allow_at = 100
          100 < 100? No → ALLOW; tat = 600

Single state variable (tat), constant time. Exact rate; configurable burst. Why Stripe and redis-cell use it.

4.9 Distributed Token Bucket — Sync Strategies

When local buckets approximate global:

Borrowing

Each node has quota = global / N. When a node depletes its quota, it borrows from a central reservoir. Central reservoir is replenished periodically.

Periodic Sync

Each node tracks consumed; every 1s, syncs to central counter. If global is exceeded, central tells nodes to throttle.

Probabilistic

Each request, sample whether to apply local or central limit (e.g., 90% local, 10% central). Central enforces "ground truth"; local is best-effort.

Lyft's ratelimit (https://github.com/envoyproxy/ratelimit): centralized Redis with descriptors. Used in Envoy global rate limits.

4.10 Hierarchical Token Bucket Implementation

class HTBNode:
    def __init__(self, parent, rate, ceil):
        self.parent = parent
        self.rate = rate          # guaranteed
        self.ceil = ceil          # max with borrowing
        self.tokens = rate
        self.last = time.monotonic()
        self.children = []

    def try_acquire(self, n=1):
        now = time.monotonic()
        delta = now - self.last
        self.tokens = min(self.ceil, self.tokens + delta * self.rate)
        self.last = now
        if self.tokens >= n:
            if self.parent and not self.parent.try_acquire(n):
                return False
            self.tokens -= n
            return True
        # try borrowing from parent
        if self.parent and self.parent.try_acquire(n):
            return True
        return False

Each request walks up the tree; all levels must allow. Borrowing is implicit through ceil > rate.

Linux tc htb is the canonical implementation; AWS API Gateway also uses HTB-like semantics.

4.11 Penalty / Soft / Hard Limits

For UX, separate: - Soft limit (e.g., 80% of hard): warn the client; no actual rejection. - Hard limit (100%): reject with 429.

Helps clients self-throttle before hitting hard limits. Useful for paying customers — they can scale down gracefully.

4.12 Burst Allowance (Token Bucket as a Feature)

Token bucket allows bursts up to capacity. This is desirable for client SDKs that batch: - Steady state: 100 RPS limit. - Burst: 500 in 5s allowed.

Leaky bucket forbids bursts entirely (output is always smooth). Use leaky bucket when downstream is fragile (DB, slow API); token bucket when downstream tolerates spikes.

4.13 What to Limit

  • Per-user / per-API-key: standard.
  • Per-endpoint: protect specific expensive operations.
  • Per-tenant: in B2B SaaS.
  • Per-IP: anti-abuse (DDoS).
  • Composite: (user, endpoint) — fairness across users on each endpoint.

In webhooks: per-tenant + per-endpoint with min of the two.

4.14 Headers (Standardization)

Per IETF draft RateLimit Header Fields for HTTP: - RateLimit-Limit: 100, 100;w=60 (100 req/min) - RateLimit-Remaining: 87 - RateLimit-Reset: 47 (seconds until reset) - Retry-After: 30 (on 429)

Older convention (still widely used): X-RateLimit-*.
