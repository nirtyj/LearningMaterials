Watermarks & Event-Time Processing

Closing the window on time you can't trust: watermarks, allowed lateness, triggers, and the Dataflow model.

2.1 The Core Problem

Events flow through a distributed pipeline with multiple sources, partitions, and stages. Each event has two timestamps: - Event time: when the event happened in the real world (set at the producer). - Processing time: when the system observes the event.

The skew processing_time - event_time is unbounded in theory and large in practice (mobile clients, batch uploads, network outages, queue buildup). To produce correct windowed results in event time, the system needs to know "when is it safe to close window [T1, T2)?" That's the watermark's job.

2.2 Watermark Definition

A watermark W(t) is a monotonic estimate that all events with event_time ≤ W will have arrived by processing_time t.

When the watermark passes the end of a window, the window is "complete" and may be emitted.

There's an inherent completeness vs. latency tradeoff: - Generous watermark (large lag) → high completeness, high latency. - Aggressive watermark (small lag) → low latency, possible late events (incorrectness).

The Dataflow paper formalizes this: there's no point estimator that's both perfectly complete and zero-latency in an unbounded out-of-order stream.

2.3 Watermark Generation Strategies

2.3.1 BoundedOutOfOrderness

WatermarkStrategy
  .<Event>forBoundedOutOfOrderness(Duration.ofSeconds(5))
  .withTimestampAssigner((e, ts) -> e.eventTimeMs);

Implementation: track max_seen_event_time, emit W = max_seen_event_time - 5s. Simple, robust.

2.3.2 Punctuated Watermarks

Some events carry watermark info themselves (e.g., a periodic "marker" record from the source). Better when the system has knowledge of source completeness (e.g., Kafka __transaction_state markers).

2.3.3 Perfect Watermarks (when possible)

If your source can guarantee in-order events per partition (e.g., a single producer writing to a Kafka partition with enable.idempotence=true), the watermark per partition equals the latest event time. Multi-partition watermark = min over partitions.

2.3.4 Adaptive (Heuristic)

Track event-time lag distribution (p99) over a rolling window; set watermark = max_event_time - p99(lag). Implemented in some Flink connectors and in Beam's heuristic source watermark.

2.4 Watermark Propagation Through a DAG

   Source A (partition 0, W=100)  ───┐
   Source A (partition 1, W=120)  ───┤
   Source A (partition 2, W= 95)  ───┼─▶ min ─▶ Operator (W = 95)
                                     │
   Source B            (W=110)    ───┘                │
                                                      ▼
                                              ┌──────────────┐
                                              │  Window Op   │
                                              │  closes any  │
                                              │  win end ≤95 │
                                              └──────────────┘

The min rule is critical: an operator's output watermark is the minimum of its input watermarks. Without this, downstream may emit windows before all input partitions have caught up.

2.5 The Watermark Stall Problem

If one input partition stops sending data (idle source), min will not advance, and the entire downstream pipeline stalls.

Solution: idle source detection.

WatermarkStrategy
  .<Event>forBoundedOutOfOrderness(Duration.ofSeconds(5))
  .withIdleness(Duration.ofSeconds(30));

After 30s of no events on a partition, the operator marks it idle and excludes it from min. When data resumes, the partition rejoins. This is essential for Kafka topics with sparse partitions.

2.6 Allowed Lateness & Late-Data Handling

Allowed lateness extends the window's "open" period beyond watermark passing the end. Events arriving in [window_end, window_end + allowed_lateness] trigger re-firing.

.window(TumblingEventTimeWindows.of(Time.minutes(1)))
.allowedLateness(Time.minutes(10))
.sideOutputLateData(lateOutputTag)
.aggregate(new MyAgg());

Trade-off: - Larger allowedLateness → state is kept longer (memory cost). - Beyond allowedLateness → events are dropped or sent to a side output for later batch backfill.

2.7 Triggers (Beam Model)

A trigger decides when a window's pane is emitted. Beam supports:

  • AfterWatermark.pastEndOfWindow() — fire when watermark crosses window end (the default).
  • AfterProcessingTime.pastFirstElementInPane().plusDelayOf(Duration) — fire D after first element.
  • AfterCount(N) — fire every N elements.
  • Compound triggers: AfterWatermark.pastEndOfWindow().withEarlyFirings(...).withLateFirings(...).
# Beam trigger combining early speculative + on-time + late updates
trigger = (
  AfterWatermark(
      early=AfterProcessingTime(60),     # fire every 60s before watermark
      late=AfterCount(1)                  # fire on each late event
  )
)

Accumulation modes: - DISCARDING: each pane only contains new data since previous firing. - ACCUMULATING: each pane contains all data seen so far. - ACCUMULATING_AND_RETRACTING: each pane contains all data and emits retractions for prior values (essential for downstream incremental maintenance).

2.9 Implementation in Flink (Internals)

   Operator T1                Operator T2 (window)
   ┌─────────────┐            ┌────────────────────┐
   │ Receives    │            │  WatermarkOperator │
   │ event(ts)   │ ─emit─▶   │  ───────────────  │
   │             │            │  pendingWindows    │
   │ periodic    │            │   [w1, w2, ...]    │
   │ Watermark   │            │                    │
   │ (every 200ms│            │  on Watermark(W):  │
   │ by default) │            │    for w in pending│
   │             │ ─emit─▶   │      if w.end ≤ W: │
   │             │            │        fire(w)      │
   └─────────────┘            └────────────────────┘

Key implementation details: - Watermarks travel as in-band records in the stream (not control messages on a sidecar). - Each operator tracks currentInputWatermark[partition] and emits currentOutputWatermark = min(...) only when it changes. - Window operator stores (window, key) → state in keyed state. When watermark crosses window.end + allowed_lateness, state is purged.

2.10 Common Interview Follow-ups

Q: What's the difference between event time and processing time? A: Event time is when the event happened. Processing time is when the system saw it. The skew is variable and unbounded; correct window semantics require event time + watermarks.

Q: When would you use processing time? A: Operational monitoring (counts of events as they arrive at the system), simple dashboards, when source clocks are untrusted. Always cheaper since no state for late events.

Q: How does Flink ensure exactly-once with watermarks and side outputs? A: Distributed snapshots (Chandy-Lamport with barriers). Barriers flow alongside data; each operator snapshots its state when it has received barriers from all inputs. On recovery, all operators rewind to the same snapshot. Sinks must be transactional or idempotent for end-to-end EOS.

Q: How do retractions work? A: Each output is paired with a + (insert) or - (delete) marker. Downstream maintains a running aggregate that subtracts on -. Materialize and Flink's RetractStream support this natively.

2.12 Watermark Heuristics in Practice

Real systems rarely use pure BoundedOutOfOrderness. Production approaches:

Quantile-Based Heuristic (Beam)

Track event-time-lag distribution over a moving window. Set watermark to max_event_time - p99(lag). Adapts to changing skew patterns automatically. Used in Beam's Pub/Sub source.

Marker-Based (Kafka's "End-of-Day" Pattern)

Producer emits a special marker record at known intervals (e.g., end of every minute, day). Watermark = timestamp of last marker received. Strong correctness guarantee but requires producer cooperation.

Per-Source vs Global

With multiple Kafka topics or partitions, each can advance independently. The operator's effective watermark is min(per-source watermarks). With idleness detection, idle sources are excluded.

2.13 Implementing Watermarks in a Custom System

class WatermarkTracker:
    def __init__(self, max_skew_ms=5000, idle_timeout_ms=30000):
        self.max_seen_per_partition = {}  # partition → max event ts
        self.last_event_at = {}            # partition → wall-clock of last event
        self.max_skew_ms = max_skew_ms
        self.idle_timeout_ms = idle_timeout_ms
        self.current_watermark = 0

    def on_event(self, partition, event_ts):
        wall_now = time.time() * 1000
        self.max_seen_per_partition[partition] = max(
            self.max_seen_per_partition.get(partition, 0), event_ts)
        self.last_event_at[partition] = wall_now
        self._recompute()

    def on_periodic_tick(self):
        self._recompute()

    def _recompute(self):
        wall_now = time.time() * 1000
        active_partitions = [
            p for p, t in self.last_event_at.items()
            if wall_now - t < self.idle_timeout_ms]
        if not active_partitions:
            return
        new_wm = min(self.max_seen_per_partition[p] for p in active_partitions) - self.max_skew_ms
        # Watermarks must be monotonic
        if new_wm > self.current_watermark:
            self.current_watermark = new_wm
            self._fire_pending_windows()

    def _fire_pending_windows(self):

2.14 Kafka Streams Stream-Time

Kafka Streams uses "stream-time" — the maximum timestamp seen across all partitions of an input topic. Stream-time only advances; never goes back.

KIP-328 added a grace period: events older than window_close + grace are dropped, but updates within the grace window cause re-emission.

TimeWindows.ofSizeAndGrace(Duration.ofMinutes(5), Duration.ofMinutes(2))
   // 5-min windows; allow late events for 2 more minutes

suppress operator: hold window emissions until window closes (no updates), then emit final. Useful for downstream consumers that can't tolerate retractions.

.windowedBy(TimeWindows.ofSizeAndGrace(...))
.aggregate(...)
.suppress(Suppressed.untilWindowCloses(BufferConfig.unbounded()))

2.15 Spark Structured Streaming

Spark uses withWatermark:

df.withWatermark("eventTime", "10 minutes")
  .groupBy(window(col("eventTime"), "1 minute"))
  .count()

Late events past the watermark are dropped silently; no side output by default. Output modes: - Append: each window emits exactly once when watermark passes its end. - Update: emit each time the window's value changes. - Complete: emit full result table on every trigger (only for small aggregations).

2.16 Beam's Pub/Sub Watermark Heuristic

Pub/Sub source in Beam uses a heuristic based on: - Backlog age (oldest unacked message). - Per-shard watermark estimation. - Adaptive lag detection.

It's allowed to be "wrong" — events past the watermark go to the late panes via the trigger system. This is the "pragmatic" approach: estimate aggressively, handle late data correctly.

2.17 Allowed Lateness vs Grace Period

Different terminology, same concept: - Beam / Flink "allowed lateness": time after watermark crosses window-end during which late events trigger re-firing. - Kafka Streams "grace period": time after window-end to accept late events; specified at window definition.

Both extend the window's "open" lifetime beyond watermark. After lateness/grace, state is purged.

2.18 Practical Sizing

How wide to set the lateness budget? - 99% of events arrive within 5 minutes of their event time → set lateness to 5-10 minutes. - Bursty mobile clients with hours of buffering → 24h+ allowed lateness, but use side output to a separate batch backfill table. - Tradeoff: lateness × event rate = state memory.

2.19 Watermark Holds (Beam)

Beam allows "holds" — programmatic suppression of watermark advancement when a stateful operator has pending work. Used by stateful processing operators to delay downstream firings until output is ready.
