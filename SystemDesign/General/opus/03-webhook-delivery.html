3.1 Requirements & Constraints

Functional: - Tenants register webhook endpoints with event subscriptions. - Producer system emits events; platform delivers via HTTP POST. - Retry on 5xx / network errors with exponential backoff + jitter. - DLQ after N retries. - Per-tenant rate limit (e.g., 100 RPS to a slow endpoint). - Optional FIFO per (tenant, ordering_key). - Signed payloads (HMAC SHA-256 with rotation). - Replay / redelivery API. - 99.9% delivery within 1 minute SLO at p99 (when endpoints are healthy).

Non-functional: - Millions of events/sec ingest. - Tens of thousands of concurrent in-flight deliveries. - Endpoint failure isolation (one slow tenant must not stall others). - Multi-tenant fairness. - Audit log: every delivery attempt + response code + body (truncated) for support.

3.4 Low-Level: Worker Loop

   Worker thread (Goroutine / async task):

   loop:
     msg = consume_from_local_priority_heap()  # by (visible_at, tenant_score)
     if msg is None: sleep(50ms); continue

     # 1. Rate limit gate (token bucket / GCRA in Redis)
     if not gate.try_acquire(msg.tenant_id):
        msg.visible_at = now + retry_after_for_throttle()
        re_enqueue(msg); continue

     # 2. Circuit breaker check
     if breaker[msg.endpoint_url].is_open():
        msg.visible_at = now + breaker_cooloff()
        re_enqueue(msg); continue

     # 3. HMAC sign
     signature = hmac_sha256(secret, f"{msg.timestamp}.{msg.body}")
     headers = {
        "X-Signature": signature,
        "X-Timestamp": msg.timestamp,
        "X-Idempotency-Key": msg.event_id,
        "X-Attempt": str(msg.attempt + 1),
        "User-Agent": "WebhookSvc/1.0"
     }

     # 4. Dispatch with timeout
     try:
        resp = http_client.post(msg.endpoint_url, body=msg.body,
                                headers=headers, timeout=15s)
     except TimeoutError, ConnectionError:
        record_attempt(msg, "network_error")
        breaker[msg.endpoint_url].record_failure()
        if msg.attempt + 1 >= MAX_ATTEMPTS:
           dlq.put(msg); continue
        msg.attempt += 1
        msg.visible_at = now + backoff(msg.attempt)
        re_enqueue(msg); continue

     # 5. Classify response
     if 200 <= resp.status < 300:
        breaker[msg.endpoint_url].record_success()
        ack(msg)
     elif resp.status in (410, 404):  # gone — disable endpoint
        disable_endpoint(msg.endpoint_id)
        record_attempt(msg, f"http_{resp.status}_disabled")
     elif resp.status == 429:
        retry_after = parse_retry_after(resp) or backoff(msg.attempt)
        msg.visible_at = now + retry_after
        re_enqueue(msg)
     elif 500 <= resp.status < 600:
        breaker[msg.endpoint_url].record_failure()
        if msg.attempt + 1 >= MAX_ATTEMPTS:
           dlq.put(msg); continue
        msg.attempt += 1
        msg.visible_at = now + backoff(msg.attempt)
        re_enqueue(msg)
     else:
        # 4xx other — don't retry (client bug)
        dlq.put(msg)

3.5 Component Deep Dives

3.5.1 Ingestion (Transactional Outbox)

To avoid the dual-write problem ("we wrote to DB, but Kafka publish failed"), use the outbox pattern:

BEGIN;
INSERT INTO orders (id, ...) VALUES (...);
INSERT INTO outbox (event_id, topic, payload, created_at, status)
   VALUES (uuid_v7(), 'order.created', '...', now(), 'pending');
COMMIT;

Then a CDC poller (Debezium) or polling worker reads outbox rows and publishes them to Kafka with at-least-once semantics. On success, marks status=published. Idempotent because each outbox row has a unique ID.

Reference: Confluent — The Transactional Outbox Pattern. https://developer.confluent.io/courses/microservices/the-transactional-outbox-pattern/

3.5.2 Durable Event Log

Choices: - Kafka: high throughput, partitioned by (tenant_id, endpoint_id) for ordering. Consumer offset tracking is a pain at high consumer count. - Redis Streams: simpler, in-memory + AOF. Good for medium scale (millions of events/day). - Postgres: SELECT ... FOR UPDATE SKIP LOCKED enables pull-based queue. Works up to ~10k msg/s per shard. Used by Sidekiq alternatives like Solid Queue and by Svix. - AWS SQS / SNS: managed; SQS FIFO supports per-MessageGroupId ordering. Throughput-limited at 300 msg/s per group (3000 with high-throughput mode).

3.5.3 Per-Tenant Fairness (DRR / WFQ)

Without fairness, a noisy tenant with 1M pending deliveries will starve other tenants. Use Deficit Round Robin:

class DeficitRoundRobin:
    def __init__(self, quantum=10):
        self.queues = {}        # tenant -> deque
        self.deficit = {}       # tenant -> int
        self.active = []        # round-robin order
        self.quantum = quantum

    def enqueue(self, tenant, item):
        if tenant not in self.queues:
            self.queues[tenant] = collections.deque()
            self.deficit[tenant] = 0
            self.active.append(tenant)
        self.queues[tenant].append(item)

    def dequeue(self):
        while self.active:
            t = self.active[0]
            self.deficit[t] += self.quantum
            served_this_round = []
            while self.queues[t] and self.deficit[t] >= 1:
                item = self.queues[t].popleft()
                self.deficit[t] -= item.cost   # cost = bytes / a fixed 1
                served_this_round.append(item)
                yield item
            if not self.queues[t]:
                self.active.pop(0)
                del self.deficit[t]; del self.queues[t]
            else:
                self.active.append(self.active.pop(0))

DRR gives O(1) per packet (no priority queue) and bounded fairness. Used in router QoS (Cisco), Linux tc qdisc.

3.5.4 HMAC Signing (Stripe-style)

import hmac, hashlib, time

def sign(secret: bytes, body: bytes, ts: int = None) -> tuple[str, int]:
    ts = ts or int(time.time())
    payload = f"{ts}.".encode() + body
    sig = hmac.new(secret, payload, hashlib.sha256).hexdigest()
    return sig, ts

def verify(secret: bytes, body: bytes, ts: int, sig: str, tolerance=300) -> bool:
    if abs(time.time() - ts) > tolerance:    # reject replays
        return False
    expected, _ = sign(secret, body, ts)
    return hmac.compare_digest(expected, sig)

Header format (Stripe): Stripe-Signature: t=1700000000,v1=<hex>,v1=<hex_next_secret>. Multiple v1 values let you rotate secrets without downtime — verify against any active secret.

3.5.5 Circuit Breaker

Three states: closed (normal), open (fail fast, no calls), half-open (probe one request).

class CircuitBreaker:
    def __init__(self, fail_threshold=5, reset_after=30):
        self.state = "closed"
        self.failures = 0
        self.opened_at = 0

    def is_open(self):
        if self.state == "open":
            if time.time() - self.opened_at > self.reset_after:
                self.state = "half-open"
                return False
            return True
        return False

    def record_success(self):
        self.failures = 0; self.state = "closed"

    def record_failure(self):
        self.failures += 1
        if self.failures >= self.fail_threshold:
            self.state = "open"; self.opened_at = time.time()

In production, use rolling-window counts (last 60s of calls) and trip on failure ratio > X% with a min request count (else flapping on low traffic).

References: Hystrix, Resilience4j, GitHub circuit-breaker doc.

3.5.6 SSRF / Endpoint Trust

Customer endpoints are untrusted. Defenses: - Resolve DNS yourself; reject if it resolves to RFC1918, link-local, loopback, or your internal VPC ranges. - Disable HTTP redirects (a 302 to http://169.254.169.254 would exfiltrate cloud metadata). - Set Connection: close to avoid HTTP/2 connection reuse to a different :authority. - Egress through a hardened proxy that re-validates the IP at TCP connect time (TOCTOU on DNS).

3.6 Production Systems Deep Dives

Stripe Webhooks

  • Retries up to 3 days with exponential backoff.
  • Signature in Stripe-Signature with t= (timestamp) and v1= (HMAC-SHA256 of t.body).
  • Tolerance window 5 minutes for replay protection.
  • Idempotency key retention: 24 hours.
  • Endpoints disabled after long failure streak; can be reactivated via dashboard.
  • See: https://docs.stripe.com/webhooks

Svix (open-source webhook infra)

  • Built in Rust; backed by Postgres + Redis.
  • Uses omniqueue (their own queue abstraction — Postgres or Redis Streams backend).
  • HMAC-SHA256 signing with timestamp-prefixed body (v1,t=1700000000,sig=...).
  • Ed25519 supported for newer customers.
  • Retry schedule: immediately, 5s, 5m, 30m, 2h, 5h, 10h, 1d, 1d (9 attempts total over ~24h).
  • Per-app SSRF blocking, no following redirects.
  • Source: https://github.com/svix/svix-webhooks

Hookdeck

  • "Event gateway" — sits between producer and consumer.
  • Adds queue, retry, replay, transformation, filtering.
  • Per-destination rate limiting and queueing.
  • See: https://hookdeck.com/event-gateway

GitHub Webhooks

  • Max payload 25 MB.
  • Signature in X-Hub-Signature-256 (sha256=<hex>).
  • X-GitHub-Delivery: GUID for idempotency / debugging.
  • Redelivery API: POST /repos/{owner}/{repo}/hooks/{hook_id}/deliveries/{delivery_id}/attempts.
  • See: https://docs.github.com/en/webhooks

Shopify Webhooks

  • 48h retry window; 19 failed attempts before auto-removal.
  • X-Shopify-Hmac-Sha256: base64-encoded HMAC of body (note: not t.body).
  • X-Shopify-Event-Id: idempotency token.
  • See: https://shopify.dev/docs/apps/webhooks

AWS EventBridge

  • Archive-and-replay: capture events for N days, replay arbitrary ranges into rules.
  • Rules can fan out to up to 5 targets each.
  • See: https://aws.amazon.com/blogs/compute/archiving-and-replaying-events-with-amazon-eventbridge/

Slack Events API

  • 3 retries + delayed events queue (24h).
  • X-Slack-No-Retry: 1 response header tells Slack not to retry on a specific failure (good for permanent errors).
  • See: https://docs.slack.dev/apis/events-api/

Twilio Webhooks

  • Validates with X-Twilio-Signature = base64(HMAC-SHA1(token, full_url + sorted_params)).
  • Supports TwiML responses inline (for voice/SMS routing).
  • See: https://www.twilio.com/docs/usage/webhooks/webhooks-security

SendGrid Event Webhooks

  • Posts JSON arrays of events.
  • sg_event_id for idempotency.
  • See: https://docs.sendgrid.com/for-developers/tracking-events/event

3.7 CAP Tradeoffs

  • Availability: customer endpoint is up but slow → we MUST keep retrying without dropping (durability of the queue is paramount).
  • Consistency: ordering per (tenant, ordering_key) requires a single consumer per partition — sacrificing parallelism.
  • Partition tolerance: between dispatcher and customer endpoint we always assume partition (it's the internet). Backoff + retry + DLQ.

3.8 Common Interview Follow-ups

Q: How do you ensure ordering when retries reorder events? A: Per-(tenant, ordering_key) FIFO queue. On retry, the message stays in its queue and blocks subsequent messages. Customers opting into ordering accept that a slow event blocks subsequent ones.

Q: What happens when an endpoint is down for an hour? A: Circuit breaker opens after N failures in a window. We stop hammering and let the message wait. After cooldown, half-open probe. Total retry budget = MAX_ATTEMPTS (e.g., 19) with backoff up to MAX_BACKOFF (e.g., 6h). Beyond that → DLQ + alert customer (email).

Q: How do you scale to 10M events/sec? A: Partition log by tenant_id; ~thousands of partitions. Dispatcher pool auto-scales on queue depth. Use HTTP/2 connection pooling. Saturate at NIC, not CPU.

Q: How do you guarantee no duplicate delivery? A: You can't (network is unreliable; the 200 OK can be lost). Best you can do is at-least-once + give consumer an idempotency key (X-Webhook-ID). They dedupe on their end.

Q: How do you handle the "noisy neighbor" tenant? A: Per-tenant rate limit + per-tenant queue depth cap. If a tenant exceeds depth, new events are rejected at ingestion (return 503 to producer with Retry-After).

3.9 Production Gotchas

  • Slow endpoints holding connections: HTTP timeout MUST be aggressive (15s typical). Worker pool can blow up otherwise.
  • DNS TTL = 1s for endpoint hostnames; aggressive caching can mask outage.
  • Customer's 100 Continue quirk: send body in same request, don't wait.
  • TLS handshake cost: connection pooling per origin.
  • Large payloads: cap at 5MB; over that, send a pre-signed URL instead.
  • Thundering herd on resume: when an endpoint comes back, all queued events flood it. Add jitter to wake-up.
