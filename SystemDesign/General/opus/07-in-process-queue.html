7.1 Core Semantics

  • At-least-once: every successfully enqueued message is delivered to (at least) one consumer, eventually. Possible duplicates if a worker crashes after processing but before acking.
  • Visibility timeout: when a worker dequeues a message, the message becomes invisible to other consumers for T_vt seconds. If the worker doesn't ack within T_vt, the message becomes visible again.
  • Lease token: every dequeue increments a monotonic counter; when the worker acks, it includes the token. If the token is stale (a re-delivery happened in the interim), the ack is rejected — preventing the late-finishing-worker double-commit.

This last property is the fence-token pattern (see §8) applied to message acks.

7.2 High-Level Architecture

   ┌─────────────────────────────────────────────────────────────┐
   │                    QUEUE PROCESS                            │
   │                                                             │
   │   ┌─────────────────────────────────────────────────────┐   │
   │   │         CLIENT API (HTTP / gRPC / lib)              │   │
   │   │  enqueue(body) | dequeue() | ack(id, token)         │   │
   │   │  extend(id, token, secs) | nack(id, token)          │   │
   │   └─────────┬───────────────────────────────────────────┘   │
   │             ▼                                               │
   │   ┌─────────────────────────────────────────────────────┐   │
   │   │              IN-MEMORY QUEUE STATE                   │   │
   │   │   ready_heap: min-heap by (visible_at, seq)         │   │
   │   │   in_flight:  HashMap<msg_id, (lease_token,         │   │
   │   │                              expires_at, body)>      │   │
   │   │   acked_log:  set<msg_id>  (for replay safety)       │   │
   │   └─────────┬───────────────────────────────────────────┘   │
   │             ▼                                               │
   │   ┌─────────────────────────────────────────────────────┐   │
   │   │     WAL (write-ahead log) — durable                  │   │
   │   │     append-only file with CRC32C per record         │   │
   │   │     records: ENQ | DEQ(id, token) | ACK(id, token)  │   │
   │   │              | NACK | EXTEND | TICK(timestamp)      │   │
   │   └─────────┬───────────────────────────────────────────┘   │
   │             ▼                                               │
   │   ┌─────────────────────────────────────────────────────┐   │
   │   │        PERIODIC SNAPSHOT (every N records or M MB)  │   │
   │   │        compacts WAL, truncates old segments         │   │
   │   └─────────────────────────────────────────────────────┘   │
   └─────────────────────────────────────────────────────────────┘

7.3 Mid-Level: State Transitions

                  enqueue(body)
                       │
                       ▼
                 ┌──────────┐
                 │  READY   │◀───────────┐
                 └─────┬────┘            │
                       │ dequeue()       │
                       │ (assigns lease) │ visibility expired
                       ▼                 │ OR  nack()
                 ┌─────────────┐         │
                 │ IN_FLIGHT   │─────────┘
                 │ (lease=L)   │
                 └─────┬───────┘
                       │ ack(id, L)
                       ▼
                 ┌──────────┐
                 │   DONE   │
                 └──────────┘

When a message moves READY → IN_FLIGHT, lease counter increments. If worker fails to ack before expires_at, a sweeper moves it back to READY with a new lease counter on next dequeue. Stale acks (workers using old lease) are rejected.

7.4 Low-Level: Reference Implementation

import collections, heapq, hashlib, os, struct, time, threading, uuid
from dataclasses import dataclass, field
from typing import Optional

@dataclass(order=True)
class _ReadyEntry:
    visible_at: float
    seq: int
    msg_id: str = field(compare=False)

@dataclass
class _InFlight:
    msg_id: str
    body: bytes
    lease_token: int            # monotonically increasing per dequeue
    expires_at: float
    enq_seq: int                # original enqueue seq for stable ordering on requeue

class Queue:
    def __init__(self, wal_path, default_vt=30.0, max_attempts=10):
        self.lock = threading.Lock()
        self.ready: list[_ReadyEntry] = []
        self.in_flight: dict[str, _InFlight] = {}
        self.bodies: dict[str, bytes] = {}
        self.attempts: dict[str, int] = {}
        self.dlq: list[bytes] = []
        self.seq = 0
        self.lease_ctr = 0
        self.default_vt = default_vt
        self.max_attempts = max_attempts
        self.wal = open(wal_path, "ab+", buffering=0)

    # ─── WAL primitives ─────────────────────────────────────────
    def _wal_write(self, op: bytes, payload: bytes):
        rec = op + payload
        crc = self._crc32c(rec)
        hdr = struct.pack("<II", len(rec), crc)
        self.wal.write(hdr + rec)
        os.fsync(self.wal.fileno())   # group-commit in real impl

    @staticmethod
    def _crc32c(data: bytes) -> int:
        # In production use crcmod / google-crc32c. Placeholder:
        import zlib
        return zlib.crc32(data) & 0xFFFFFFFF

    # ─── Public API ─────────────────────────────────────────────
    def enqueue(self, body: bytes) -> str:
        with self.lock:
            self.seq += 1
            msg_id = uuid.uuid4().hex
            self.bodies[msg_id] = body
            self.attempts[msg_id] = 0
            heapq.heappush(self.ready, _ReadyEntry(time.monotonic(), self.seq, msg_id))
            self._wal_write(b"E", msg_id.encode() + b"\x00" + body)
            return msg_id

    def dequeue(self, vt: Optional[float] = None) -> Optional[tuple[str, bytes, int]]:
        vt = vt or self.default_vt
        with self.lock:
            now = time.monotonic()
            self._reap_expired_locked(now)
            while self.ready:
                entry = self.ready[0]
                if entry.visible_at > now:
                    return None
                heapq.heappop(self.ready)
                if entry.msg_id not in self.bodies:
                    continue       # already acked or dropped
                self.lease_ctr += 1
                tok = self.lease_ctr
                expires = now + vt
                self.in_flight[entry.msg_id] = _InFlight(
                    entry.msg_id, self.bodies[entry.msg_id], tok, expires, entry.seq)
                self.attempts[entry.msg_id] = self.attempts.get(entry.msg_id, 0) + 1
                self._wal_write(b"D", entry.msg_id.encode()
                                + b"\x00" + str(tok).encode())
                return entry.msg_id, self.bodies[entry.msg_id], tok
            return None

    def ack(self, msg_id: str, token: int) -> bool:
        with self.lock:
            entry = self.in_flight.get(msg_id)
            if not entry or entry.lease_token != token:
                return False        # stale lease — rejected
            del self.in_flight[msg_id]
            self.bodies.pop(msg_id, None)
            self.attempts.pop(msg_id, None)
            self._wal_write(b"A", msg_id.encode() + b"\x00" + str(token).encode())
            return True

    def nack(self, msg_id: str, token: int, requeue_after: float = 0.0) -> bool:
        with self.lock:
            entry = self.in_flight.get(msg_id)
            if not entry or entry.lease_token != token:
                return False
            del self.in_flight[msg_id]
            if self.attempts.get(msg_id, 0) >= self.max_attempts:
                self.dlq.append(self.bodies.pop(msg_id))
                self._wal_write(b"X", msg_id.encode())
            else:
                heapq.heappush(self.ready,
                               _ReadyEntry(time.monotonic() + requeue_after,
                                           entry.enq_seq, msg_id))
                self._wal_write(b"N", msg_id.encode())
            return True

    def extend(self, msg_id: str, token: int, secs: float) -> bool:
        with self.lock:
            entry = self.in_flight.get(msg_id)
            if not entry or entry.lease_token != token:
                return False
            entry.expires_at = time.monotonic() + secs
            self._wal_write(b"V", msg_id.encode() + b"\x00" + str(secs).encode())
            return True

    # ─── Internal: visibility expiry sweeper ────────────────────
    def _reap_expired_locked(self, now):
        expired = [m for m, e in self.in_flight.items() if e.expires_at <= now]
        for msg_id in expired:
            entry = self.in_flight.pop(msg_id)
            if self.attempts.get(msg_id, 0) >= self.max_attempts:
                self.dlq.append(self.bodies.pop(msg_id))
                self._wal_write(b"X", msg_id.encode())
            else:
                heapq.heappush(self.ready,
                               _ReadyEntry(now, entry.enq_seq, msg_id))
                self._wal_write(b"R", msg_id.encode())   # "Re-queued"

    # ─── Recovery: replay WAL on startup ────────────────────────
    @classmethod
    def recover(cls, wal_path):
        q = cls(wal_path + ".tmp")
        # ... read records from old WAL, apply to in-memory, re-write to new WAL
        # ... atomically rename .tmp → wal_path
        # In production: snapshot every N records to bound replay time.
        return q

Why the lease token solves the race:

   T=0   Worker A dequeues msg X, gets lease=42, vt=30s
   T=15  Worker A's process pauses (GC, network hiccup)
   T=30  Visibility expires; sweeper requeues X (still attempt #1)
   T=31  Worker B dequeues X, gets lease=43
   T=32  Worker B processes X, calls ack(X, 43) → SUCCESS
   T=33  Worker A wakes up, calls ack(X, 42) → REJECTED (stale lease)

Without the token, A's ack would commit a second time. The token makes ack a CAS on (msg_id, lease).

7.5 Component Deep Dives

7.5.1 WAL Design (Group Commit)

Single-record fsync per enqueue is throughput-bottlenecked at ~200/s on commodity SSD. Group commit: batch up to N writes or M ms, then one fsync. Each batch is an atomic commit unit.

class GroupCommitWAL:
    def __init__(self, path, max_batch=100, max_ms=5):
        self.f = open(path, "ab+", buffering=0)
        self.pending = []
        self.cv = threading.Condition()
        # background thread flushes
        threading.Thread(target=self._flusher, daemon=True).start()

    def append(self, rec):
        with self.cv:
            fut = threading.Event()
            self.pending.append((rec, fut))
            self.cv.notify()
        fut.wait()

    def _flusher(self):
        while True:
            with self.cv:
                while not self.pending:
                    self.cv.wait()
                batch = self.pending; self.pending = []
            for rec, _ in batch:
                self.f.write(rec)
            os.fsync(self.f.fileno())
            for _, fut in batch:
                fut.set()

Throughput: 100k+ msgs/s on a single SSD (vs. ~200/s without group commit).

7.5.2 Visibility Timeout Tracking — Hashed Wheel Timer

Naive for msg in in_flight: if expired ... is O(N) per tick. Better: hashed wheel timer (Varghese-Lauck 1987). Used in Netty, Kafka, Akka.

   Wheel of W buckets, each bucket = list of (msg_id, expiry).
   tick_ms = e.g., 100ms.
   add(msg, expires_at):
      slot = (expires_at / tick_ms) % W
      buckets[slot].append((msg_id, expires_at))
   on tick:
      slot = (now / tick_ms) % W
      for (msg_id, exp) in buckets[slot]:
          if exp <= now: requeue(msg_id)

Reference: https://netty.io/4.0/api/io/netty/util/HashedWheelTimer.html

7.5.3 Recovery & Snapshots

WAL grows unbounded. Periodically snapshot in-memory state:

   1. Acquire write lock.
   2. Note current WAL offset O.
   3. Serialize state to snapshot.tmp.
   4. Release lock.
   5. fsync(snapshot.tmp); rename → snapshot.bin.
   6. Truncate WAL records < O.

On restart: load snapshot.bin, then replay WAL from offset.

7.6 Production Systems Comparison

AWS SQS

  • Visibility timeout: 0–43200s (12h max).
  • FIFO queues: ordering per MessageGroupId; throughput 300 msg/s standard, 3000/s high-throughput mode.
  • Deduplication: 5-min window via MessageDeduplicationId or content hash.
  • Retention: 14 days max.
  • 120k in-flight messages max per queue (limit).
  • See: https://docs.aws.amazon.com/AWSSimpleQueueService/latest/SQSDeveloperGuide/sqs-visibility-timeout.html

BullMQ (Node.js, Redis)

  • Built on Redis Streams + atomic Lua scripts.
  • Sliding-window visibility via processed-stalled-jobs.lua.
  • Job priorities, delayed jobs (sorted set keyed on time), repeating jobs (cron).
  • See: https://docs.bullmq.io/

Sidekiq (Ruby, Redis)

  • BRPOP from list; reliable-fetch variant uses LMOVE to a "private" queue per worker (visibility tracking).
  • See: https://github.com/sidekiq/sidekiq/wiki

RabbitMQ

  • Quorum queues (since 3.8): Raft-replicated; at-least-once delivery; dead-letter routing on x-overflow=reject-publish-dlx.
  • Per-consumer prefetch_count caps in-flight per channel (max 2000 in modern versions).
  • See: https://www.rabbitmq.com/docs/confirms

Kafka Consumer Groups

  • Offsets are visibility tokens (committed after processing).
  • max.poll.interval.ms = 5 min default; if processing exceeds, consumer is removed and partition reassigned.
  • Cooperative-sticky rebalancing minimizes disruption.
  • At-least-once via enable.auto.commit=false + manual commit after processing.

Google Pub/Sub

  • Ack deadline 10s default, extendable up to 600s.
  • Exactly-once delivery (newer feature) via dedup window + fence-token-equivalent.
  • See: https://docs.cloud.google.com/pubsub/docs/lease-management

Apache Pulsar

  • Subscription types: Exclusive, Failover, Shared, Key_Shared.
  • Key_Shared = ordered per key but parallel across keys — great for tenant ordering.
  • See: https://dattell.com/data-architecture-blog/subscription-types-in-apache-pulsar/

Beanstalkd

  • TTR (time-to-run) = visibility timeout.
  • bury / kick for DLQ-style operations.
  • Single binary; very lean. Used by older PHP shops.

NATS JetStream

  • Stream-based; consumer state durably tracked.
  • Pull and push consumers; MaxAckPending controls in-flight cap.
  • See: https://docs.nats.io/nats-concepts/jetstream

Temporal (Durable Execution)

  • Not a queue per se; workflow state machines with sticky task queues.
  • History replay reconstructs state on worker crash.
  • See: https://docs.temporal.io/sticky-execution

Step Functions (AWS)

  • Task tokens: a long-lived activity sends SendTaskSuccess(taskToken, output).
  • Token is essentially a fence — bound to a specific execution attempt.

SQLite-Backed Queues (Litequeue, Solid Queue, …)

  • PRAGMA journal_mode=WAL; PRAGMA synchronous=NORMAL; PRAGMA busy_timeout=5000;
  • BEGIN IMMEDIATE; SELECT ... ORDER BY id LIMIT 1; UPDATE ... SET state='IN_FLIGHT', lease_token=lease_token+1 WHERE id=?; COMMIT; is the dequeue.
  • See: https://github.com/litements/litequeue

7.7 Common Interview Follow-ups

Q: How do you guarantee at-most-once instead? A: Don't redeliver on visibility timeout — drop. But you'll lose messages on worker crash. Better: at-least-once + idempotent consumer with inbox pattern (§6.5).

Q: How do you support priority? A: Multiple ready heaps (one per priority); dequeue tries higher-priority first. Or weighted fair queueing across priority classes to avoid starvation.

Q: How do you support delayed messages? A: visible_at = now + delay_secs. The same heap structure handles it.

Q: Memory blows up with 100M ready messages? A: Spill to disk. Postgres / RocksDB backing for ready set. In-memory only for in-flight (which is bounded by worker count × per-worker concurrency).

Q: Two workers race on dequeue — is the lease token enough? A: For ack correctness, yes. But the dequeue itself must be atomic — under a lock or via atomic-CAS in the storage backend.

7.8 Production Gotchas

  • Long-running consumers: extend lease before expiry. Worker that takes 10min on a 30s VT will see message redelivered. Have consumers periodically call ChangeMessageVisibility.
  • Poison pills: a message that always crashes the worker — without max-attempts → DLQ, infinite retry loop.
  • Clock skew between sweeper and worker: use time.monotonic(), not wall clock, for visibility math.
  • Ack-after-reboot: WAL replay must idempotently re-establish state. Use sequence numbers in WAL records.
  • In-flight message count caps: SQS limits 120k. Helps prevent runaway producers.
