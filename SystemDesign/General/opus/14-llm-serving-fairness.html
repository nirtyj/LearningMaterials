14.1 Why Naive Fairness Fails for LLMs

Classical fair queueing (round-robin, WFQ) treats each request as a unit. But LLM requests have: - Variable prefill cost (proportional to prompt length). - Variable decode cost (proportional to output length). - Variable KV memory (proportional to total length).

Round-robin gives equal requests, not equal work. A user with 100k-token prompts gets the same share as a user with 10-token prompts — unfair to the small-prompt user, who's effectively donating GPU time.

14.2 Virtual Token Counter (VTC)

VTC (OSDI '24, Sheng et al.) tracks per-user token consumption with virtual time accounting:

class VTC:
    def __init__(self, weights):
        self.weights = weights        # tier → weight
        self.vt = {}                   # user → virtual time consumed

    def on_admit(self, req):
        # weight tokens by tier weight
        cost = (req.prompt_tokens + req.expected_output) / self.weights[req.tier]
        self.vt[req.user] = self.vt.get(req.user, self._min_vt()) + cost

    def _min_vt(self):
        return min(self.vt.values()) if self.vt else 0

    def pick_next(self, candidates):
        # pick request from user with smallest vt
        return min(candidates, key=lambda r: self.vt.get(r.user, 0))

Counter-lift on re-appearance: a user who hasn't sent requests for a while shouldn't be allowed unbounded backlog of credit. Lift their vt to min(global_vt) on re-entry.

The paper proves a 2× tight bound: VTC ensures any two users with continuous demand within the same window receive shares within a factor of 2 of their fair share.

Reference: https://www.usenix.org/conference/osdi24/presentation/sheng

14.3 DLPM (Locality-Aware Fair) — DeepSched 2025

VTC ignores cache locality. DLPM adds a "stickiness bonus" — same user's next request prefers a replica that already has their KV/prefix cached. 2.87× throughput, 7.18× lower latency vs. raw VTC. Reference: https://arxiv.org/abs/2501.14312

14.4 DRF — Dominant Resource Fairness

Multiple resource types (compute, KV memory). DRF: each user has a "dominant share" = max over resources of (their use / total). Equalize dominant shares.

For LLMs: compute = TFLOPs/s, memory = KV bytes. A long-prompt user has dominant share in compute; a long-context one in memory. DRF correctly balances both.

14.5 Skip-Join MLFQ (FastServe)

FastServe (2023): multi-level feedback queue where new requests start at level 0, demoted on each iteration. Short requests finish at level 0; long requests sink to level N. Higher-level queues get scheduled first.

Reports 31.4× throughput improvement over FCFS at same SLO.

Reference: https://arxiv.org/html/2305.05920v3

14.6 FairServe (Interaction-Aware)

For chat: a user's "perceived" experience is dominated by TTFT and per-token latency, not throughput. FairServe weights the scheduler by user-perceived latency.

14.7 Equinox (Holistic Fairness)

Equinox (2025) integrates fairness across multiple dimensions: throughput, TTFT, TBT, prefix-cache hit rate. Multi-objective optimization.

Reference: https://arxiv.org/html/2508.16646

14.8 Justitia (Agentic Workloads)

Agent workloads (multiple sequential LLM calls per user task) need different fairness — Justitia "selectively pampers" agentic tasks to avoid them being stuck behind long single-shot users.

Reference: https://arxiv.org/html/2510.17015

14.9 Composite Production Scheduler

A production-quality LLM scheduler combines:

  1. Admission control: EDF; reject if SLO can't be met.
  2. Per-tier WFQ: hierarchical token bucket with weights {free:1, plus:4, pro:16}.
  3. Within tier: VTC: fair across users.
  4. Tie-break with DLPM: cache locality for picking replica.
  5. Per-replica: Sarathi chunked prefill with token budget.
  6. Live migration (Llumnix) for rebalance.
   ┌──────────────────────────────────────────────┐
   │       INCOMING REQUESTS                      │
   └─────────────────┬────────────────────────────┘
                     ▼
   ┌──────────────────────────────────────────────┐
   │   ADMISSION (EDF + reserved/shared pools)    │
   └─────────────────┬────────────────────────────┘
                     ▼
   ┌──────────────────────────────────────────────┐
   │   PER-TIER WFQ (free/plus/pro/api)           │
   └─────────────────┬────────────────────────────┘
                     ▼
   ┌──────────────────────────────────────────────┐
   │   WITHIN TIER: VTC (per-user)                │
   └─────────────────┬────────────────────────────┘
                     ▼
   ┌──────────────────────────────────────────────┐
   │   ROUTING: DLPM (cache-aware replica pick)   │
   └─────────────────┬────────────────────────────┘
                     ▼
   ┌──────────────────────────────────────────────┐
   │   REPLICA: Sarathi chunked-prefill scheduler │
   └──────────────────────────────────────────────┘

14.10 Common Interview Follow-ups

Q: Free tier user has been waiting 60s. Pro user just arrived. Who runs? A: With pure VTC, pro runs (lower vt per token due to weight). With aging (free's vt decays over time), free eventually wins. Tradeoff: aging gives better tail fairness but slightly weaker pro-tier guarantees. Most production: aging with cap (free tier latency capped at X min).

Q: Pro user has 100M tokens of credit (paid up). Should they monopolize? A: No — convert credit to weighted virtual time, not absolute slots. Monopolization would degrade other pro users. Apply per-user rate caps even within tier.

Q: How do you ensure the scheduler itself isn't a bottleneck? A: Decentralize: each replica has a local scheduler; global router just does layer-7 routing. Global VTC counter is in Redis with periodic batch updates.

14.12 The Full VTC Algorithm (with Counter-Lift)

The Sheng et al. (OSDI '24) paper presents this in detail. Key invariants:

  1. Service counter: each user has a virtual time vt[u] representing service consumed.
  2. Tier weight: w[u] = tier weight (free=1, plus=4, pro=16).
  3. Cost per token: cost = tokens_processed / w[u] — high-tier users "pay less" per token.
  4. Min-vt selection: scheduler picks the request with the lowest vt, breaking ties by FCFS.
  5. Counter-lift: when a user re-appears after silence, set vt[u] = max(vt[u], min_active_vt) so they don't accumulate unbounded credit.

Detailed Pseudocode

class VTCScheduler:
    def __init__(self, tier_weights):
        self.weights = tier_weights
        self.vt = {}  # user → virtual time
        self.last_seen = {}  # user → last activity timestamp

    def admit_request(self, req):
        u = req.user
        # Counter-lift if user was inactive
        if u in self.last_seen and time.time() - self.last_seen[u] > 60:
            self.vt[u] = max(self.vt.get(u, 0), self._min_active_vt())
        elif u not in self.vt:
            self.vt[u] = self._min_active_vt()  # join at minimum
        self.last_seen[u] = time.time()
        # Estimate work; will be debited as tokens process
        return req

    def on_token_processed(self, req, n_tokens=1):
        u = req.user
        cost = n_tokens / self.weights[req.tier]
        self.vt[u] += cost

    def pick_next(self, candidates):
        # Pick request from user with smallest vt
        return min(candidates, key=lambda r: (self.vt[r.user], r.arrival_time))

    def _min_active_vt(self):
        if not self.vt:
            return 0
        return min(self.vt.values())

Tightness Theorem

VTC guarantees that for any two continuously-active users u₁ and u₂ over interval [t₁, t₂]:

   |service(u₁) / w[u₁] - service(u₂) / w[u₂]| ≤ 2 × max_request_size

Tight to within 2× max_request_size. Both proven and simulated in the OSDI paper.

14.13 Counter-Lift Subtleties

Without counter-lift, a user who's been silent for hours could "bank" infinite credit. Then return and dominate.

   Naive (no lift):
   t=0: user A active, vt=0
   t=60: A goes idle. min_vt advances as others use service. 
         At t=600, min_vt = 100. A's vt is still 0.
   t=601: A returns. vt[A]=0 < min_vt=100 → A scheduled exclusively
          until vt[A] catches up = ~100 requests of starvation.

Lift fixes this:

   t=601: A returns. Lift vt[A] to max(0, 100) = 100. A enters as equal.

14.14 DLPM (Locality-Aware Fair) — Why Cache Matters

VTC doesn't know about cache locality. If user A's KV is on replica R1, scheduling A on R2 wastes the cache.

DLPM's key idea: weight the scheduler's selection by cache locality bonus.

def dlpm_pick(replicas, candidates):
    best = None
    best_score = -1
    for c in candidates:
        for r in replicas:
            if not r.has_capacity(): continue
            cache_hit = r.prefix_cache_hit(c.prompt)
            fairness_score = -vtc.vt[c.user]   # higher is better
            locality_bonus = 0.5 if cache_hit else 0
            score = fairness_score + locality_bonus
            if score > best_score:
                best, best_score = (c, r), score
    return best

Reports: 2.87× throughput, 7.18× lower latency vs raw VTC. Reference: https://arxiv.org/abs/2501.14312

14.15 DRF Worked Example

Two users, two resources (compute, KV memory): - User A: 80% compute, 20% memory per request. - User B: 20% compute, 60% memory per request.

Dominant share: - A: max(80%, 20%) = 80% — compute-dominated. - B: max(20%, 60%) = 60% — memory-dominated.

DRF schedules so dominant shares are equal. If DRF allocates X% to A and Y% to B: - Minimize X·0.8 - Y·0.6 (deviation in dominant-share) - Subject to X·0.2 + Y·0.6 ≤ 100 (memory) - X·0.8 + Y·0.2 ≤ 100 (compute)

Solving: X = 75, Y = 50. A gets dominant share 60%, B gets 30%. Both feel "weighted-fair" in their dominant resource.

DRF source: Ghodsi et al., NSDI 2011. https://www.usenix.org/conference/nsdi11/dominant-resource-fairness-fair-allocation-multiple-resource-types

14.16 Skip-Join MLFQ Detail

FastServe's MLFQ for LLMs: - N priority levels (e.g., 8). - New requests start at level 0 (highest priority). - After processing K tokens at level i, demote to level i+1. - Pick from level 0 first; cascade.

"Skip-join": predict request length from prompt; large requests directly enter mid-level instead of starting at 0 and walking down.

Result: short requests (chat messages) finish at level 0 fast. Long requests (RAG with reranking) sink to lower levels and don't block short ones.

Reports 31.4× throughput improvement at SLO. Reference: https://arxiv.org/html/2305.05920v3

14.17 FairServe (Interaction-Aware)

Standard fairness measures token throughput. But user-perceived fairness is about: - TTFT (time-to-first-token). - Per-token TBT (smooth streaming). - Total response time.

FairServe weights its scheduler by these perceived metrics, not raw throughput. A user with TTFT > SLO gets boosted priority until satisfied.

Reference: https://arxiv.org/html/2411.15997

14.18 Equinox Holistic Fairness

Equinox (2025) integrates: 1. Fairness across users (VTC-like). 2. Fairness across SLO dimensions (TTFT, TBT, throughput). 3. Locality (cache hit rate). 4. Tier weights.

Multi-objective optimization with adjustable tradeoffs. Helps avoid Pareto-suboptimal combinations.

Reference: https://arxiv.org/html/2508.16646

14.19 Justitia for Agentic Workloads

Agent workloads = sequence of LLM calls per user task (e.g., AutoGPT, LangChain agents). Fairness over individual calls misses the per-task fairness.

Justitia treats each task as a unit; selectively "pampers" agentic tasks (gives priority bumps) so they don't get stuck behind a single long-prompt user.

Reference: https://arxiv.org/html/2510.17015

14.20 Composite Production Scheduler — Detailed

def production_schedule_step(self):
    # 1. Admission control: EDF-based, reserved capacity per tier
    for req in self.incoming:
        if not self.admit(req):
            req.reject(retry_after=self.estimate_retry_time(req))
        else:
            self.queue_per_tier[req.tier].append(req)

    # 2. WFQ: select tier this step
    tier = self.wfq.next_tier()

    # 3. Within tier: VTC pick
    candidates = self.queue_per_tier[tier]
    chosen = self.vtc.pick_next(candidates)

    # 4. DLPM: pick replica with cache locality
    replica = self.dlpm.pick_replica(chosen)

    # 5. Send to replica's local Sarathi-Serve scheduler
    replica.enqueue(chosen)

class WFQ:
    def __init__(self, weights):
        self.weights = weights  # tier → weight
        self.virtual_time = {t: 0 for t in weights}

    def next_tier(self):
        # Pick tier with smallest virtual_time / weight
        return min(self.virtual_time.items(),
                   key=lambda kv: kv[1] / self.weights[kv[0]])[0]

    def served(self, tier, work):
        self.virtual_time[tier] += work

14.21 Live Migration (Llumnix) for Rebalance

Without live migration, a hot replica stays hot until queue drains. Llumnix moves running requests across replicas via NVLink KV transfer, treating rebalance as a memory copy:

   Imbalance detected at scheduling tick:
     R1 KV: 95% full, queue 50 deep.
     R2 KV: 30% full, queue 5 deep.

   Pick request X from R1 (the one with smallest progress).
   NVLink transfer X's KV from R1 to R2 (~10ms for 1GB).
   Resume X on R2.

This makes rebalance a 10ms operation vs the alternative of waiting for X to finish (could be seconds). Reference: https://www.usenix.org/conference/osdi24/presentation/sun-biao

14.22 Tail Latency Controls

p99.9 TTFT/TBT requires explicit attention: - Hedging: send identical request to two replicas; cancel the slower. - Speculative execution: run a small model on the same prompt while the big model spins up; if outputs converge, return small model's. - Deadline propagation: every request carries a deadline; replicas reject when they can't meet it (return 503). - Smaller-batch fast lane: dedicated queue for sub-100-token requests; no contention with long-context workloads.

14.23 Common Interview Follow-ups

Q: How is VTC different from WFQ? A: WFQ is per-flow; VTC is per-token-with-tier-weights. WFQ allocates flows fair shares of bandwidth. VTC weighs each token by tier. Combined: WFQ at tier level, VTC within each tier.

Q: What if tier weight is 1000 — pro user dominates? A: Aging caps starvation: free user's vt decays over wait time. Even with 1000× weight, free users will eventually run.

Q: How do you measure fairness? A: Jain's fairness index: J = (Σx_i)² / (n × Σx_i²). J=1 means perfectly fair. Most production systems target J > 0.9 across users in steady state.

Q: What if a user submits 100 requests at once? A: Per-user concurrency cap (e.g., 5 concurrent). Excess queued. VTC then sees "1 active per user" rather than disproportionate dispatcher pressure.
