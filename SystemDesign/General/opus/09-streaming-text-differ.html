Token-Level Streaming Text Differ with Rollback

From naive O(n) rollback to persistent trees with O(log n) snapshots, Myers diff, and how ChatGPT branches conversations.

9.1 Requirements

  • Input: a stream of tokens (token_id, position, timestamp, text).
  • Output: incremental diff events: INSERT(pos, text), DELETE(pos, len), MODIFY(pos, old, new).
  • Support: random access rollback to any timestamp T (return text-as-of-T).
  • Performance: O(log n) insert/rollback, not O(n).

This is the OpenAI internal use case for assistant message editing — a token mid-stream gets revised, and downstream sees the delta.

9.2 Why Naïve Designs Fail

Naïve: Snapshot Every Token

Memory: O(N²). Untenable.

Naïve: List + Linear Diff

Each rollback: O(N) memory copy. Each insert: O(N) shift. Quadratic in the worst case.

The interviewee's actual answer was a versioned tree where each token maintains a chain of versions and the differ walks the chain — close to a CRDT (RGA-flavored), but with O(n) rollback. The improvement is to use a persistent (immutable) tree for O(log n) rollback.

9.4 Mid-Level: Persistent Vector

Clojure's PersistentVector is a 32-way trie + tail buffer. Update: walk the path, copy each node along the path with the change applied, leave siblings shared. Per update: O(log₃₂ N) nodes copied = ~7 nodes for N up to 4 billion.

              ┌──┐
              │  │
        ┌──┼──┼──┼──┐
        ▼     ▼     ▼
      ┌──┐  ┌──┐  ┌──┐
      │  │  │  │  │  │           After update at index i:
   ┌──┼──┼──┴──┼──┴──┼──┐        - copy root → root'
   ▼     ▼     ▼     ▼            - copy 1st-level child on path
   ...                            - copy 2nd-level child on path
                                  - copy leaf
                                  All other nodes are shared.

Implementation reference: Clojure's PersistentVector.java. Python: pyrsistent.pvector.

9.5 Token-Level Diff Generation: Myers Algorithm

def myers_diff(a: list, b: list):
    """O((N+M)·D) where D = edit distance. The Git default."""
    N, M = len(a), len(b)
    max_d = N + M
    v = {1: 0}
    trace = []
    for d in range(max_d + 1):
        trace.append(dict(v))
        for k in range(-d, d+1, 2):
            if k == -d or (k != d and v.get(k-1, -1) < v.get(k+1, -1)):
                x = v.get(k+1, 0)        # down (insert from b)
            else:
                x = v.get(k-1, 0) + 1    # right (delete from a)
            y = x - k
            while x < N and y < M and a[x] == b[y]:
                x += 1; y += 1
            v[k] = x
            if x >= N and y >= M:
                return _trace_back(trace, a, b)
    return []

Variants: - Patience diff (Bram Cohen): finds longest common subsequence of unique lines first; used by Git when --patience. Better for code with reordered functions. - Histogram diff: Git's default since 2.18; faster than patience, similar quality. - Hirschberg: O(N) space variant of LCS; doubled time. - Heckel (1978): O(N) for low-edit cases; used by Mercurial in some paths.

Reference: https://luppeng.wordpress.com/2020/10/10/when-to-use-each-of-the-git-diff-algorithms/

9.6 Low-Level: Streaming Differ with Rollback

from pyrsistent import pvector

class StreamingDiffer:
    def __init__(self):
        self.versions = [pvector()]      # version_index → state
        self.timestamps = [0]            # version_index → ts

    def insert(self, pos, token, ts):
        prev = self.versions[-1]
        new = prev.set(pos, token) if pos < len(prev) else prev.append(token)
        self.versions.append(new)
        self.timestamps.append(ts)
        yield ("INSERT", pos, token)

    def delete(self, pos, ts):
        prev = self.versions[-1]
        new = prev.delete(pos)
        self.versions.append(new)
        self.timestamps.append(ts)
        yield ("DELETE", pos)

    def modify(self, pos, new_token, ts):
        prev = self.versions[-1]
        old = prev[pos]
        new = prev.set(pos, new_token)
        self.versions.append(new)
        self.timestamps.append(ts)
        yield ("MODIFY", pos, old, new_token)

    def rollback(self, ts):
        # binary search for last version with timestamps <= ts
        import bisect
        idx = bisect.bisect_right(self.timestamps, ts) - 1
        return list(self.versions[idx])

    def diff_between(self, ts1, ts2):
        s1, s2 = self.rollback(ts1), self.rollback(ts2)
        return list(myers_diff(s1, s2))

Memory per edit: O(log n) shared nodes. 1M tokens × 64 nodes/edit × 32 bytes = 2 GB for full history; with structural sharing it drops to a small fraction.

9.7 Production Systems

Operational Transforms (OT)

Used in Google Docs, Etherpad, ShareJS. Each edit is an "operation" (Insert/Delete with position). When two clients concurrently edit, the OT function transforms one operation against the other so applying both yields the same final state.

Properties: - TP1: O1' = T(O1, O2) and O2' = T(O2, O1) such that O1 . O2' = O2 . O1'. - TP2: required for triple-concurrent edits; harder to satisfy.

Drawback: O(N²) ops to transform a sequence of N ops. Doesn't scale to long histories. CRDTs replaced OT in many places.

CRDTs (see §10)

Yjs (JavaScript CRDT)

YATA algorithm (Yet Another Transformation Approach). Block-based: each insert is a "block" with (client_id, clock). Yjs is the dominant CRDT in production: - TipTap, BlockNote (rich-text editors). - Liveblocks. - Hocuspocus (Yjs server). - JupyterLab real-time collaboration.

Reference: https://github.com/yjs/yjs

Automerge (JSON CRDT)

Kleppmann's CRDT for arbitrary JSON. Uses RGA for arrays, MVR for registers, OR-Set for objects. v2 (Rust) is dramatically faster than v1.

Benchmark on B4 trace (Kleppmann's standard): - Yjs ~970 ms / 3 MB. - Automerge v1 291 s / 880 MB. - Automerge v2 < 5 s.

Reference: https://martin.kleppmann.com/2017/04/24/json-crdt.html

Loro (Rust CRDT, Eg-walker)

Loro implements the Eg-walker algorithm — a sub-O(N) text CRDT competitive with Yjs but with persistent history (DAG-based). DAG checkpoint API for "what did doc look like at op X".

Git Internals

  • Blob: file content (compressed).
  • Tree: directory snapshot (list of mode name blob_sha).
  • Commit: tree + parent + author + msg.
  • Pack files: offset-deltas — file v2 stored as delta against v1.

Persistent tree of trees → snapshot at any commit is just a tree-walk.

Mercurial RevLog

Each file has a revlog: alternating snapshot + delta. Reconstructing rev N: find nearest snapshot, apply deltas forward. Tradeoff between space (deltas cheap) and time (long delta chains slow to reconstruct).

VS Code Piece Tree

For large files, VS Code uses piece tree: a red-black tree where each node represents a "piece" (a slice of either the original file or the add buffer). Insert = split piece + insert. Delete = split + remove range. Each piece tree node caches line_count, allowing O(log N) line-number lookups.

Reference: https://code.visualstudio.com/blogs/2018/03/23/text-buffer-reimplementation

Figma Multiplayer

Figma uses LWW per property + tree CRDT for hierarchy. NOT a full CRDT system overall — they explicitly state that for atomic operations like "move object to new parent", they use server-side ordering (server is the arbiter), not CRDT semantics. See: https://www.figma.com/blog/how-figmas-multiplayer-technology-works/

ChatGPT Message Tree

ChatGPT models conversation as a tree: each message has parent_id. When you edit a message, a new sibling is created (not in-place). Regenerate creates a sibling response. The "current" view is a single root-to-leaf path. This is a branching version tree, like Git.

Reference: https://knowledge.buka.sh/the-hidden-fork-how-editing-messages-in-chatgpt-lets-you-branch-conversations/

9.8 Streaming JSON Partial Parsing

LLMs emit JSON token-by-token. Downstream UIs want to render partial state (e.g., rendering as fields fill in). Solutions:

  • partial-json-parser (Promplate): vendored by OpenAI Node SDK; handles partial keys/values by inserting closing brackets.
  • partialjson (Python).
  • openai-partial-stream.
  • Aha! engineering note: naive recursive parsers are O(N²) on growing input; use an incremental parser.

Reference: https://www.aha.io/engineering/articles/streaming-ai-responses-incomplete-json

9.9 Common Interview Follow-ups

Q: How do you handle concurrent edits from multiple sources? A: CRDT (RGA / Yjs) for true concurrency; OT if you control all clients and have a single source of truth.

Q: How do you bound memory if history is unbounded? A: Periodic snapshots + compaction. Keep last N versions in fine detail, older ones at coarse granularity.

Q: O(n) rollback in your initial design — how do you fix? A: Path-copying persistent vector / RRB-tree gives O(log n) rollback. Each version shares O(n - log n) nodes with predecessors.

Q: Token-level vs character-level diffing? A: Token-level matches LLM unit of generation. Character-level matches user expectations (e.g., word-level highlighting in UI). Both are valid; tokenize first, then diff per granularity.

Q: How do you handle out-of-order tokens? A: Each token has a logical position from the LLM (vector-clock-style or unique ID). Use a CRDT-like ordering: arriving tokens are inserted at their logical position regardless of arrival order.
