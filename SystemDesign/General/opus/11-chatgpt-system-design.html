Mid-Level: Per-Model Inference Topology

For ChatGPT-scale models, disaggregated prefill/decode wins:

   ┌─────────────────────────────────────────────────────────────┐
   │                      ROUTER                                 │
   │   choose model, admit request, attach prefix-cache hint     │
   └────────────┬────────────────────────────────────────────────┘
                ▼
   ┌─────────────────────────────────────────────────────────────┐
   │                  PREFILL POOL (compute-bound)               │
   │   ── runs the prompt through the model in parallel          │
   │   ── produces KV cache for the prompt                       │
   │   ── chunked-prefill (Sarathi-Serve) to bound iteration time │
   └────────────┬────────────────────────────────────────────────┘
                │ KV cache transferred over NVLink / IB / NVMe-oF
                ▼
   ┌─────────────────────────────────────────────────────────────┐
   │                  DECODE POOL (memory-BW-bound)              │
   │   ── continuous batching of decode-only iterations          │
   │   ── PagedAttention block table; prefix cache hits          │
   │   ── streaming SSE/WebSocket back to client                 │
   └─────────────────────────────────────────────────────────────┘

Why disaggregate? Prefill is compute-bound (matmul on long prompt); decode is memory-bandwidth-bound (read entire KV cache per token). Co-locating them means prefill (long-running iteration) blocks decode (fast iterations) — head-of-line blocking. DistServe (Tencent), Splitwise (Microsoft), Mooncake (Moonshot/Kimi) all show 1.4×–7.4× throughput from disaggregation.

11.2 GPU Capacity Allocation (the interview's deep question)

Tier model

Define tier weights:

Tier Weight
Free 1
Plus 4
Pro 16
API varies (per-customer SLA)

These weights drive the fair scheduler's quanta — see §14.

Cross-tier traffic correlation

Spikes are correlated: a viral event drives free, plus, pro all up at once. So slack from one tier rarely covers another's surge. Strategies:

  1. Reserved capacity per tier (in GPUs): Pro gets 20% of fleet reserved, regardless of utilization. Prevents starvation but wastes capacity at off-peak.
  2. Shared pool with WFQ: tiers share GPU but compete with weighted fair queueing. When idle, all tiers can use full capacity.
  3. Hybrid: small reserved per tier (5%) + large shared pool with WFQ. Best of both.

The hybrid is the production answer.

Admission control

When the fleet is at capacity, new requests must be rejected — but selectively. Pro requests get accepted up to their reserved quota even when free is at capacity.

   on incoming(req):
       if pool.utilization < 0.85:
           accept; route_to_replica()
       elif req.tier == "free" and free_pool.utilization > free_pool.reserved:
           reject 429 with Retry-After
       elif req.tier == "plus" and plus_pool.utilization > plus_pool.reserved:
           reject (or queue with deadline)
       else:
           accept; queue with priority by tier

EDF (earliest-deadline-first) admission gives the best goodput per Hao-Zhang's DistServe analysis.

Scheduler bias toward pro without starving free

Two complementary techniques:

  1. VTC (Virtual Token Counter) — see §14. Each user has a token-debt counter weighted by tier. The scheduler picks the request with lowest weighted debt. Provably fair.
  2. Aging: free requests gain priority the longer they wait; eventually beat any pro request by enough margin. Prevents starvation regardless of pro inflow.

11.3 Conversation Across Model Versions

User opens a chat at 9 PM with GPT-5.5; you deploy 5.5.1 at 11 PM; user sends a message at 11:30 PM.

Options: 1. Sticky to original version: every message in this conversation goes to GPT-5.5.0 — until that version is decommissioned. Pros: behavioral consistency. Cons: keeps old model alive; can't ship hotfixes. 2. Always latest: switch to 5.5.1 mid-conversation. Pros: fresh code. Cons: KV cache from 5.5.0 isn't reusable on 5.5.1 (architecture is the same but weights differ; KV is weight-dependent). Re-prefill the entire history. Latency spike. 3. Tokenizer-compatible upgrade with re-prefill: same tokenizer, run prefill on 5.5.1 over the full history, mark conversation as upgraded. One-time cost.

Most production systems pick (2) with re-prefill, but mark "conversation upgraded" so the user sees a banner (or doesn't, if behavioral diff is small).

Detecting tokenizer drift: hash the tokenizer config; reject re-prefill if old conversation has tokens that don't decode under new tokenizer.

11.4 Autoscaling Under Different Latency Profiles

Naive: scale on CPU/GPU utilization. Wrong because LLM serving is bursty and bound by KV cache memory, not utilization.

Better signals: - Queue depth × E[remaining_output_tokens]: forward-looking; predicts whether queue can drain. - KV cache occupancy: the actual memory pressure. - TTFT (time-to-first-token) p95: directly tied to user experience. - TBT (time-between-tokens) p95: ditto, decode side.

When TTFT p95 > SLO for T seconds, scale up. When KV cache occupancy < 50% for T seconds, scale down.

References: - Hao-AI-Lab on learning-to-rank scheduling: https://hao-ai-lab.github.io/blogs/vllm-ltr/ - DistServe: https://arxiv.org/abs/2401.09670 - KEDA scaling: https://keda.sh/

Scaling on goodput, not throughput

DistServe defines goodput = requests per second meeting SLO. Maximize this, not raw tokens/s. Otherwise you can saturate the GPU but fail TTFT.

Cold-start

Model Cold-start time
7B ~30 s
70B ~2 min
400B+ ~20 min

Mitigations: - Always-warm pool of standby replicas. - Spot+on-demand mix. - Pre-warmed KV cache for popular prompts.

Different latency profiles

GPT-5.5 has higher TTFT (longer prefill) than 4o. Each model gets its own scaling group with own SLO targets and own queue. Don't share scaling logic across models.

Llumnix live migration

Llumnix (Alibaba, OSDI '24) migrates running requests across replicas — moving the KV cache via NVLink. This makes load-rebalancing as cheap as a memory copy, enabling tighter packing without restarting requests. Reference: https://www.usenix.org/conference/osdi24/presentation/sun-biao

11.5 Headline Numbers (Production Reports)

  • DistServe (Hao-AI-Lab, OSDI '24): 7.4× more requests at SLO vs co-located baseline.
  • Splitwise (Microsoft): 1.4× throughput at 20% lower cost.
  • Mooncake (Moonshot/Kimi): 115% / 107% more requests on A800 / H800 in production at Kimi.
  • Sarathi-Serve: 5.6× capacity at strict TBT SLO on Falcon-180B.
  • vLLM PagedAttention: 2-4× throughput vs. baseline (HuggingFace TGI of 2023).
  • Character.AI: 95% prefix cache hit rate; KV cache memory reduced 20× via MQA + Cross-Layer Attention + SWA + INT8.

11.6 Other Components (Brief)

Conversation Service

  • Postgres for durable state (tree of messages: parent_id → children).
  • Redis for hot session caching.
  • Edits create siblings, not in-place modification (preserves history; matches §9 design).

Request Routing

  • Layer 7: pick model (per user choice + tier + availability).
  • Within model: pick replica via Maglev consistent hashing on (conversation_id, prefix_hash) to maximize prefix-cache hit rate.

Telemetry

  • Prometheus + Grafana for app metrics.
  • DCGM for GPU metrics.
  • Distributed tracing (OpenTelemetry) for cross-service.
  • Per-replica logging at high volume → Vector → S3.

11.7 Common Interview Follow-ups

Q: Pro user submits a 100k-token prompt. How do you handle? A: Chunked prefill (Sarathi-Serve): split into 1k-token chunks, interleave with decode iterations of other requests. Prevents head-of-line. Total prefill time same, but other users not stalled.

Q: A free user is in a tight retry loop, hammering you. A: Per-user RPS at edge gateway; circuit at session level. If still abusive, soft-block with captcha, then hard-block.

Q: KV cache OOM mid-generation. A: Preempt the lowest-priority request: write its KV to CPU RAM (swap) or recompute on resume (vLLM v1 default — recompute is faster on H100). Resume when memory frees.

Q: A NCCL collective hangs. A: Watchdog kills the replica after timeout (e.g., 30s). Replica restarts. Outstanding requests on that replica are rerouted with re-prefill. Track NCCL hang rate as fleet KPI.

Q: How do you A/B test a new model version? A: Shadow traffic — same prompt to both; compare outputs offline. Then: 1% canary, 10%, 50%, 100% over days. Sticky per-user (consistent hash on user_id) so they don't see flipping.

11.8 Production Gotchas

  • Long-tail stragglers: 99th percentile request takes 50× the median. Speculative decoding, request-level deadlines.
  • KV cache math: Llama-3.1-70B GQA with 128k context = ~40 GB just for one user's KV. Plan capacity accordingly.
  • Prefill blocks decode — disaggregate or chunk.
  • Tensor parallelism across AZ collapses — TP requires NVLink-grade bandwidth; doesn't survive multi-AZ. Pipeline parallelism is more forgiving.
  • OOM cascades: one replica OOMs → router shifts to others → others OOM. Need admission control upstream of replicas.
  • Tokenizer drift on model upgrades: invalidate prefix cache; warn users.
  • Inflight migrations during deploys: have a graceful drain (stop admitting, finish in-flight).
