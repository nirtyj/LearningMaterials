01 Streaming · Q1

Real-Time Event Aggregator with Rolling Windows

Maintain rolling counts over 1m / 5m / 1h windows over an out-of-order event stream — without the deque-per-window trap.

1.1 Problem Decomposition

Functional requirements: - Ingest events (event_id, timestamp_ms, key, value) at arbitrary throughput. - Maintain rolling counts (and ideally sums, distinct counts, percentiles) over multiple sliding windows simultaneously. - Out-of-order tolerance: an event arriving at processing time T_p may have event_time substantially earlier (network delay, clock skew, mobile buffering). - Query: count(key, window_size) returns the count of events for key in the last window_size seconds, with a freshness SLO (e.g., ≤ 1s lag at p99).

Non-functional: - Sub-millisecond write latency, single-digit-ms read latency at p99. - Horizontal scalability — partition by key (consistent hash). - Bounded memory regardless of event rate or skew. - Exactly-once semantics for aggregations (or at least idempotent re-processing).

Why a deque per window breaks: A naïve collections.deque per window pushes events in arrival order. Eviction works only when stream is in event-time order — when an event with t = now - 30s arrives after now, it must be inserted in the middle of the deque, which is O(n). Out-of-order events break the monotone invariant the deque relies on.

The fix (sorted bucket structure): Discretize time into fixed-width buckets (e.g., 1-second granularity), keep buckets in a sorted map (treemap / BTreeMap / SortedDict), and answer queries by summing buckets that overlap the window. Insertion into the right bucket is O(log B) where B is the number of buckets in the largest window. Evict buckets older than max window when their right edge falls outside [now - max_window, now]. For 1h window at 1s granularity, B = 3600 — very small.

1.4 Low-Level Data Structure Options

Structure Insert Range Query Memory Out-of-order Note
deque (in-order) O(1) amortized O(W/G) O(B) ❌ broken Original naive
deque + binary search insert O(B) O(W/G) O(B) ✅ but slow
Sorted bucket map (BTreeMap) O(log B) O(W/G) O(B) ✅ The interview pivot
Ring buffer (mod arithmetic) O(1) O(W/G) O(B fixed) ✅ if within range Fastest; bounded; what Flink rolling state uses internally for hot keys
Fenwick (BIT) over time slots O(log B) O(log B) O(B) ✅ Prefix-sum range queries
Segment tree over time slots O(log B) O(log B) O(B) ✅ More flexible (min/max/count)
Skip list keyed on ts O(log B) avg O(log B + k) O(B) ✅ RocksDB memtable shape
Two-stack window-min O(1) amortized O(1) O(W) ❌ Only for in-order streams; useful for sliding min/max

B = number of buckets in largest window, W = window size, G = bucket granularity.

The right answer in interview: "I'd discretize time into fixed buckets (1s for 1m/5m windows, 1m for 1h window — multi-resolution), store each key's bucket map as a BTreeMap<bucket_start, AggValue>. Insertion is O(log B). Range query is O(W/G). Out-of-order events are inserted in their correct bucket. To bound memory I evict buckets older than now - max_window lazily on insert and via a periodic sweeper. For high cardinality, I use ring buffers per key with a fixed slot count = max_window/granularity, since B is bounded (3600 for 1h@1s)."

1.5 Ring Buffer Implementation (the production answer)

class RollingCounter:
    """
    Multi-resolution ring buffer counter.
    - 1s granularity for 1-min and 5-min windows
    - 1m granularity for 1-hour window (60 × less memory)
    """
    def __init__(self):
        # 1-second buckets, 5 minutes deep (covers 1m + 5m windows)
        self.fine_slots = [0] * 300
        self.fine_ts    = [0] * 300         # bucket_start_ms each slot represents
        # 1-minute buckets, 60 minutes deep (covers 1h window)
        self.coarse_slots = [0] * 60
        self.coarse_ts    = [0] * 60

    def _idx_fine(self, t_ms):    return (t_ms // 1000) % 300
    def _idx_coarse(self, t_ms):  return (t_ms // 60_000) % 60

    def add(self, ts_ms, n=1):
        i = self._idx_fine(ts_ms)
        bucket = ts_ms - (ts_ms % 1000)
        if self.fine_ts[i] != bucket:
            self.fine_slots[i] = 0
            self.fine_ts[i]    = bucket
        self.fine_slots[i] += n

        j = self._idx_coarse(ts_ms)
        cb = ts_ms - (ts_ms % 60_000)
        if self.coarse_ts[j] != cb:
            self.coarse_slots[j] = 0
            self.coarse_ts[j]    = cb
        self.coarse_slots[j] += n

    def count(self, now_ms, window_ms):
        if window_ms <= 300_000:                       # ≤ 5 min: use fine
            slots, ts, gran, depth = self.fine_slots, self.fine_ts, 1000, 300
        else:                                          # 1 hour: use coarse
            slots, ts, gran, depth = self.coarse_slots, self.coarse_ts, 60_000, 60
        lo = now_ms - window_ms
        total = 0
        for k in range(depth):
            if lo <= ts[k] <= now_ms:
                total += slots[k]
        return total

Key insight: The lazy zeroing via timestamp check (if self.fine_ts[i] != bucket: zero it) is what gives us O(1) writes for out-of-order events without an explicit eviction pass — when you wrap around the ring, the slot is implicitly "expired" because its ts is from a previous epoch.

1.6 Component Deep Dives

1.6.1 Watermark Generator

The watermark is the system's belief that "no event with event_time < W will arrive after this point." It's the single most important concept for out-of-order processing. See §2 for full deep dive.

Common strategies: - BoundedOutOfOrderness(d): W = max_observed_event_time - d. Used in Flink. Simple, robust. - Punctuated: emit watermark on specific marker events (e.g., every Nth event includes a bound). - Adaptive: track lag percentile and shift watermark accordingly. - Idle source detection: if no events for T_idle, mark partition as idle to avoid stalling.

1.6.2 State Backend (RocksDB)

Apache Flink's RocksDB state backend stores keyed state as RocksDB key-value pairs in an embedded LSM tree. For windowed aggregations, the key is (operator_id, key, namespace, window).

Why RocksDB: - Handles state larger than RAM (paged from local SSD). - Supports range scans (needed to fetch all buckets for a key's window). - Incremental checkpoints: only ship newly-written SSTs to remote storage. - Compaction merges old SSTs in background.

Tuning gotchas: - write_buffer_size too small → frequent flushes → high write amplification. - Bloom filters on block_based_table_factory reduce SST scan cost on point reads. - Use prefix extractor for windowed state (key prefix = (operator, key)) so range scans hit a single SST. - For Flink ≥ 1.15: enable changelog state backend (DSTL) to make checkpointing async and incremental at second-granularity (Flink docs). - Flink 1.20 introduced ForStStateBackend — RocksDB fork tuned for disaggregated cloud storage (S3-backed).

1.6.3 HyperLogLog for Distinct Counts

If we want distinct(user_id) over a rolling window, exact counting needs a set per bucket → memory explosion. HyperLogLog (Flajolet-Fusy-Gandouet-Meunier 2007) gives O(1) merge with controlled error:

import math, hashlib

class HLL:
    def __init__(self, p=14):           # 2^p registers; p=14 → 16384 registers, ≈ 0.81% error
        self.p = p
        self.m = 1 << p
        self.regs = [0] * self.m
        self.alpha = 0.7213 / (1 + 1.079 / self.m)

    def add(self, x):
        h = int(hashlib.md5(str(x).encode()).hexdigest(), 16)
        idx = h & (self.m - 1)
        w = h >> self.p
        # rank = position of leftmost 1-bit in w
        rank = (w.bit_length() ^ 0xFFFFFFFFFFFFFFFF) if w == 0 else (((w & -w).bit_length()))
        self.regs[idx] = max(self.regs[idx], rank)

    def estimate(self):
        Z = 1.0 / sum(2 ** -r for r in self.regs)
        E = self.alpha * self.m * self.m * Z
        # small-range correction
        if E <= 2.5 * self.m:
            zeros = self.regs.count(0)
            if zeros: E = self.m * math.log(self.m / zeros)
        return int(E)

    def merge(self, other):
        self.regs = [max(a, b) for a, b in zip(self.regs, other.regs)]

Memory: 16 KiB per sketch (HLL++ in BigQuery, Redis PFADD, Druid all use this). Per-bucket merging across windows is associative and commutative — perfect fit.

1.6.4 Count-Min Sketch for Frequency Estimation

For "top-K events" or "is this key heavy hitter?" — Count-Min Sketch (Cormode-Muthukrishnan 2005). Keep d × w 2D counter array. On insert, hash key with d independent functions and increment all d cells. Estimate is the min of d cells (which bounds the over-estimation; under-estimation is impossible since we only add). Used in Druid (approxCount), Pinot, Apache DataSketches, Redis-Bloom CMS.

1.6.5 DDSketch / t-digest for Quantiles

For p50/p95/p99 over a rolling window: - t-digest (Ted Dunning): clustered centroids, good for extreme quantiles. - DDSketch (Datadog, VLDB 2019): relative-error guarantees, mergeable. Used in Datadog, OTLP. - Reference: DDSketch paper

1.8 Common Interview Follow-ups

Q: How would you handle a key with 10 million events/sec? A: Pre-aggregate at the source (combiner / aggregateByKey push-down). Sub-shard the hot key (key + bucket-of-time-prefix) and merge at query time. Use approximate sketches if exact count isn't required. See Cassandra-style hot key isolation and Dataflow's combine.PerKey.

Q: How do you handle the watermark stalling because one source partition is idle? A: Mark the partition as idle after T_idle. Downstream watermark = min over active partitions only. Flink: WatermarkStrategy.withIdleness(Duration.ofSeconds(30)).

Q: What if events can be late by 24 hours? A: Use allowed-lateness with side output to a dead-letter topic for super-late events. Reprocess via batch backfill. Don't make the watermark wait 24 hours — that destroys freshness.

Q: How does the system survive a worker crash? A: Periodic checkpoints to durable storage (S3/GCS). Flink: distributed snapshot via Chandy-Lamport-style barriers. On restart, restore state and rewind input offset to last checkpointed position.

Q: Two-stack constant-time window approach — when does it work? A: Only for in-order streams (it's the classic "queue using two stacks" with min/max tracking). Doesn't generalize to out-of-order. Mention it to show breadth.

1.9 Production Gotchas

  • Bucket boundary skew: if granularity is 1s and you align buckets at exact second boundaries, every event at t=N.000 lands in the same bucket — fine. But if you compute window edges with floating-point math, you'll get off-by-one boundary bugs.
  • Time-zone bugs: store and reason in UTC milliseconds. Display TZ at the edge.
  • Skewed keys: 1% of keys can carry 90% of traffic. Combine + sub-shard.
  • State explosion under cardinality: a million unique keys × 3600 buckets × 64 bytes = 230 MB. Manageable. A billion keys = 230 GB. Need RocksDB + TTL.
  • Checkpoint stalls: with hundreds of GB of state, full checkpoints can take minutes. Use incremental checkpoints + DSTL.
  • Late event correctness: depending on accumulation mode (DISCARDING vs ACCUMULATING vs ACCUMULATING_AND_RETRACTING), downstream sees different truths. Choose consciously.
