15.1 The Problem

You have N keys you want to map to M servers. Naive hash(key) % M works — until M changes. Adding or removing a server changes M, which changes the modulo, which remaps almost all keys. For a cache, that's catastrophic (cold cache → backend overload). For a sharded DB, it's a re-shuffle of all data.

We want: a hash function such that adding or removing a server only remaps O(N/M) keys. That's "consistent hashing" in the general sense; multiple algorithms achieve this with different tradeoffs.

15.2 Karger's Ring (1997)

Conceptually place servers as random points on a unit circle. Hash a key to a point on the same circle. Walk clockwise from the key; the first server encountered owns the key.

Adding a server places one new point; only keys in the arc between the new point and its predecessor get remapped. Removing one moves its keys to its successor.

                    ┌─────────────┐
                    │      A      │
                    │   (server)  │
                    │             │
                    │  K1 → A     │
              D ────│             │──── B
            (svr)   │  K2 → B     │  (svr)
                    │             │
                    │  K3 → C     │
                    │      C      │
                    │   (server)  │
                    └─────────────┘
                  hash(key) lands on circle;
                  walk clockwise to next server.

Implementation

import bisect
import hashlib

class ConsistentHashRing:
    def __init__(self, vnodes_per_node=128):
        self.vnodes = vnodes_per_node
        self.ring = []          # sorted list of (hash_position, node_id)
        self.positions = []     # sorted list of just hash_positions for bisect

    def _hash(self, key: str) -> int:
        return int(hashlib.md5(key.encode()).hexdigest(), 16)

    def add_node(self, node_id: str):
        for i in range(self.vnodes):
            h = self._hash(f"{node_id}:{i}")
            bisect.insort(self.ring, (h, node_id))
        self.positions = [h for h, _ in self.ring]

    def remove_node(self, node_id: str):
        self.ring = [(h, n) for h, n in self.ring if n != node_id]
        self.positions = [h for h, _ in self.ring]

    def get_node(self, key: str) -> str:
        if not self.ring:
            raise RuntimeError("ring empty")
        h = self._hash(key)
        idx = bisect.bisect_right(self.positions, h)
        if idx == len(self.ring):
            idx = 0
        return self.ring[idx][1]

    def get_n_nodes(self, key: str, n: int) -> list[str]:
        """For replication factor n: return next n distinct nodes."""
        if not self.ring:
            return []
        h = self._hash(key)
        idx = bisect.bisect_right(self.positions, h)
        seen = []
        while len(seen) < n and len(seen) < len(set(x[1] for x in self.ring)):
            if idx >= len(self.ring):
                idx = 0
            node = self.ring[idx][1]
            if node not in seen:
                seen.append(node)
            idx += 1
        return seen

Why Virtual Nodes

If each physical node = 1 ring position, distribution variance is high (some nodes get much larger arcs by chance). Each physical node = V virtual positions reduces standard deviation by 1/√V. V=128 typical; 256 historically Cassandra default; modern Cassandra (4.x) is reducing to 16.

Reference (Cassandra vnode tradeoffs): https://thelastpickle.com/blog/2021/01/29/impacts-of-changing-the-number-of-vnodes.html

Cost

  • Lookup: O(log(N·V)) per key (binary search).
  • Memory: O(N·V) (the sorted ring).
  • Add/Remove: O(V·log(N·V)) per operation.

15.3 Jump Consistent Hash (Google, 2014)

For scenarios where buckets are numbered 0..M-1 (not arbitrary IDs) and you want to map a 64-bit key to a bucket. No memory at all — pure computation.

def jump_hash(key: int, num_buckets: int) -> int:
    b, j = -1, 0
    while j < num_buckets:
        b = j
        key = (key * 2862933555777941757 + 1) & 0xFFFFFFFFFFFFFFFF
        j = int((b + 1) * (1 << 31) / ((key >> 33) + 1))
    return b

Five lines. O(log M) time, O(1) memory. Used in Google internal systems, Riak Core, some Pulsar paths.

Limitation: buckets must be 0..M-1 contiguous integers. You can't remove an arbitrary bucket — only shrink the largest index. So it's perfect for stateless services with a single autoscaling group, but awkward for systems where servers come and go arbitrarily.

Reference: https://arxiv.org/abs/1406.2294

15.4 Rendezvous Hashing (HRW, 1996)

Highest Random Weight: for each key, compute hash(key, server) for every server; pick the server with the highest hash.

import hashlib

def rendezvous(key: str, nodes: list[str]) -> str:
    def score(node):
        return int(hashlib.md5(f"{node}:{key}".encode()).hexdigest(), 16)
    return max(nodes, key=score)
  • O(M) per lookup.
  • O(0) memory (besides node list).
  • Trivially supports add/remove (just change node list).
  • Variance is excellent — every key independently selects.

Best for small M (M < 100). Used in CARP web caching protocol, some Cassandra paths internally.

Weighted HRW (Schindelhauer-Schomaker)

For unequal-capacity nodes, weight each node:

import math, hashlib

def weighted_hrw(key: str, nodes: list[tuple[str, float]]) -> str:
    """nodes = [(node_id, weight), ...]"""
    def score(node, w):
        h = int(hashlib.md5(f"{node}:{key}".encode()).hexdigest(), 16)
        # Normalize hash to (0, 1), then -ln(u)/w gives an exponential variate
        u = (h + 1) / (1 << 128)
        return -w / math.log(u)
    return max(nodes, key=lambda nw: score(*nw))[0]

This gives bucket sharing exactly proportional to weights — a property the basic ring doesn't have without manual vnode count tuning.

15.5 Maglev Hashing (Google, NSDI 2016)

Designed for L4 network load balancers where you need: - Constant lookup time (single array index). - Uniform distribution. - Resilience to backend changes (consistent enough that flow state isn't disrupted).

Builds a permutation table of size M (M = prime, typically 65537). Each backend has a "preference list" — a permutation of [0..M-1]. Build the lookup table by having each backend in turn place itself at its next preferred slot:

def build_maglev_table(backends: list[str], M: int = 65537) -> list[str]:
    N = len(backends)
    # Each backend produces an offset and skip
    offsets = [hash_a(b) % M for b in backends]
    skips   = [hash_b(b) % (M-1) + 1 for b in backends]
    next_idx = [0] * N
    table = [None] * M
    filled = 0
    while filled < M:
        for i in range(N):
            # Find next slot for backend i
            while True:
                slot = (offsets[i] + next_idx[i] * skips[i]) % M
                next_idx[i] += 1
                if table[slot] is None:
                    break
            table[slot] = backends[i]
            filled += 1
            if filled == M:
                break
    return table

def maglev_lookup(table: list[str], key: str) -> str:
    return table[hash_key(key) % len(table)]
  • Lookup: O(1) — one hash, one array access.
  • Build: O(M) per backend on average.
  • Add/Remove a backend: rebuild the entire table; ~1-2% of slots change (per the paper, with proper M ≫ N).

Used in Google's Maglev L4 LB, Envoy's MAGLEV cluster type, Cilium eBPF datapath.

References: - Maglev paper: https://research.google/pubs/maglev-a-fast-and-reliable-software-network-load-balancer/ - Erlang impl: https://github.com/nomasystems/nmaglev

15.6 Multi-Probe Consistent Hashing

A variant where each key hashes K times; you pick the bucket where the hash falls closest to a node. Reduces variance compared to single-probe ring without needing vnodes. Used in some FB internal systems.

15.7 Anchor Hashing (2018)

A modern alternative to ring hashing with O(1) lookups (without Maglev's table overhead). Each bucket has an "anchor" position; lookup walks anchors. Comparable distribution to vnode ring with less memory.

Reference: https://arxiv.org/abs/1812.09674

15.8 Consistent Hashing with Bounded Loads (Mirrokni-Thorup-Zadimoghaddam, 2017)

The basic ring has unbounded load imbalance — a popular key can swamp one server. Bounded loads adds a constraint: no server may carry more than c × (N/M) keys for some c > 1. If a server is "full", spill to next.

Lookup is O(log) instead of O(M) on average; falls back when overloaded.

def get_node_bounded(self, key, max_load_factor=1.25):
    target_load = total_load * max_load_factor / num_nodes
    h = self._hash(key)
    idx = bisect.bisect_right(self.positions, h)
    for _ in range(len(self.ring)):
        if idx >= len(self.ring): idx = 0
        node = self.ring[idx][1]
        if self.load[node] < target_load:
            return node
        idx += 1
    raise RuntimeError("all nodes overloaded")

Used by: - HAProxy hash-balance-factor parameter. - Vimeo's case study reported 8× improvement: https://blog.vimeo.com/news/everything-about-the-internet-load-balancer/ - Google research blog: https://research.google/blog/consistent-hashing-with-bounded-loads/

15.9 Chord (DHT, 2001)

Original distributed hash table. Each node has a "finger table" of size log N pointing to nodes at exponentially increasing distances. Lookup: O(log N) hops over the network. Used in BitTorrent DHT, IPFS, original Cassandra design (later replaced with simpler gossip-based ring).

Reference: https://pdos.csail.mit.edu/papers/chord:sigcomm01/

15.12 Hot-Key Mitigations

A hot key is fundamentally a load-balancing problem, not a hashing problem. Solutions:

  1. Bounded-loads consistent hashing (HAProxy, Vimeo).
  2. Replicate hot keys: top-K most-accessed keys are replicated to N servers; client picks one randomly.
  3. Two-level routing: first level partitions by hash(key); second level partitions by hash(key, sub_id) where sub_id rotates.
  4. Partition-key bucketing (Discord): hash(channel_id) partitions; within a hot channel, bucket by message ID range to spread.
  5. Aggregation at the edge: hot writes batched/coalesced before reaching the hot partition.

15.13 Common Interview Follow-ups

Q: Modulo hashing fails because resize remaps everything. Why doesn't a deterministic-hash + modulo hashing work? A: Because changing M is the issue, not the hash quality. hash(k) % M itself is deterministic, but M changes when the cluster resizes. Consistent hashing decouples key→position from M so the position is stable.

Q: How do you choose vnode count? A: Tradeoff between distribution variance (1/√V) and overhead (memory, lookup). 100-300 typical for caches. Cassandra moved to 16 because gossip and streaming overhead per vnode is non-trivial.

Q: Why not just use Maglev everywhere? A: Maglev rebuild is O(M·log) per backend change. For systems with frequent changes (autoscaling), ring is better. Maglev shines for stable or slow-changing topology (L4 LB).

Q: How does prefix-cache routing for LLMs use consistent hashing? A: Hash on (conversation_id, prefix_hash) rather than just conversation_id — improves cache hit rate when prompt tokens overlap across conversations. Use bounded-loads to avoid one replica getting all requests for one popular prompt.
