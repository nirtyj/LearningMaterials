16.1 The Core Pattern

Producer P sends to Consumer C. C is slower than P. Without intervention, items queue between them. Eventually queue exhausts memory.

Three high-level responses: 1. Block the producer (synchronous coupling). 2. Drop items (sample / reject / load-shed). 3. Demand-driven: consumer pulls only what it can handle.

The third is "true" backpressure. The first two are degraded modes.

16.2 Bounded Queue + Strategy

class BoundedQueue:
    def __init__(self, capacity, strategy="block"):
        self.q = collections.deque()
        self.cap = capacity
        self.strategy = strategy
        self.cv = threading.Condition()

    def offer(self, item, timeout=None):
        with self.cv:
            if len(self.q) >= self.cap:
                if self.strategy == "drop_newest":
                    return False
                elif self.strategy == "drop_oldest":
                    self.q.popleft()
                elif self.strategy == "block":
                    if not self.cv.wait_for(lambda: len(self.q) < self.cap, timeout):
                        return False
                elif self.strategy == "reject":
                    raise QueueFullError()
            self.q.append(item)
            self.cv.notify()
            return True

    def take(self):
        with self.cv:
            self.cv.wait_for(lambda: self.q)
            x = self.q.popleft()
            self.cv.notify()
            return x

Strategy implications: - Block: producer slows; latency rises but no loss; thread waste if many producers. - Drop newest: prefer keeping older items (good for state — last update unimportant). - Drop oldest: prefer freshest (good for telemetry — old samples worthless). - Reject: explicit signal upstream; use Retry-After. Aligns with HTTP 429.

RED (Random Early Detection)

Don't wait until queue is full to drop. Above threshold T1, drop probabilistically; at T2 (full), drop unconditionally. Smooths burst-induced retransmits in TCP-like protocols. Used in Cisco routers, Linux qdisc.

CoDel (Controlled Delay, Nichols & Jacobson, 2012)

Parameterless: drop the head-of-queue packet if it's been queued > 5ms (target) for > 100ms (interval). Adapts to queue's natural drain rate without configuration. Used in Linux fq_codel qdisc, iOS network stack.

Reference: https://en.wikipedia.org/wiki/CoDel

16.3 Reactive Streams (Demand-Based)

Standardized in JDK 9 (java.util.concurrent.Flow), implemented in Project Reactor, RxJava, Akka Streams. Four interfaces:

public interface Publisher<T> { void subscribe(Subscriber<? super T> s); }

public interface Subscriber<T> {
    void onSubscribe(Subscription s);
    void onNext(T t);
    void onError(Throwable t);
    void onComplete();
}

public interface Subscription {
    void request(long n);   // ← demand signal (the backpressure)
    void cancel();
}

public interface Processor<T,R> extends Subscriber<T>, Publisher<R> {}

The consumer issues subscription.request(n) — only after that does the producer push up to n items. No demand → no items → no buffer growth.

public class BackpressureExample {
    public static void main(String[] args) {
        Flow.Publisher<Integer> publisher = subscriber -> {
            subscriber.onSubscribe(new Flow.Subscription() {
                int next = 0;
                public void request(long n) {
                    for (long i = 0; i < n && next < 100; i++) {
                        subscriber.onNext(next++);
                    }
                    if (next >= 100) subscriber.onComplete();
                }
                public void cancel() {}
            });
        };

        publisher.subscribe(new Flow.Subscriber<Integer>() {
            Flow.Subscription sub;
            public void onSubscribe(Flow.Subscription s) {
                this.sub = s;
                s.request(10);                  // initial demand
            }
            public void onNext(Integer x) {
                process(x);
                sub.request(1);                  // 1-for-1 replenishment
            }
            public void onError(Throwable t) {}
            public void onComplete() {}
        });
    }
}

Reference: https://github.com/reactive-streams/reactive-streams-jvm

16.4 Credit-Based Flow Control

A generalization of demand-based: receiver sends "credits" representing buffer slots. Sender may send up to its credit balance. On each receive, receiver reclaims a credit and (eventually) sends more.

Used in: - Apache Pulsar: per-consumer flow control via consumer.receive() triggering credit replenishment. - AMQP 1.0: explicit flow frames with credit values. - RDMA / InfiniBand: hardware-level credits per QP. - HTTP/2 and gRPC: flow-control windows per stream and per connection. WINDOW_UPDATE frames replenish credits.

   [Sender]                    [Receiver]
       │ INITIAL_WINDOW=65535    │
       │←────────────────────────│
       │                         │
       │ DATA[0..1024]           │
       │────────────────────────→│  (window: 65535-1024 = 64511)
       │ DATA[1024..2048]        │
       │────────────────────────→│  (window: 63487)
       │                         │
       │ WINDOW_UPDATE +16384    │
       │←────────────────────────│  (window: 79871)

Common pitfall: window-stall when receiver is slow to send WINDOW_UPDATE. Mitigate by replenishing aggressively (every X% of consumed window, e.g., every 25%).

16.5 TCP rwnd vs cwnd

TCP has two windows: - rwnd (receive window): how much the receiver can buffer. Advertised in TCP headers. Pure backpressure. - cwnd (congestion window): how much the sender estimates the network can carry. Computed from packet-loss / RTT signals. Network-level backpressure.

Effective window = min(rwnd, cwnd). Modern algorithms: - Reno / NewReno: AIMD (additive increase, multiplicative decrease). - CUBIC: nonlinear cwnd growth; default in Linux. - BBR (Google): model-based; estimates bottleneck bandwidth and round-trip propagation. Doesn't react to single losses; better in deep buffers.

Reference: RFC 9293 (TCP), https://datatracker.ietf.org/doc/html/rfc9293

16.6 Token Bucket as Shaper

Use a token bucket to shape the output rate of a producer. Producer waits when no tokens; smooth rate maintained. Classic egress shaping in Linux tc tbf. Combine with bounded queue for safe burst handling.

16.7 Adaptive Concurrency Limits (Netflix concurrency-limits)

Static concurrency caps are wrong for elastic workloads. Adaptive algorithms learn the optimal concurrency from latency feedback:

Vegas-style limit

Inspired by TCP Vegas. Maintain limit. Compare actual RTT to estimated minimum RTT. - If RTT_actual ≈ RTT_min: capacity available → increase limit. - If RTT_actual ≫ RTT_min: queueing → decrease.

Gradient2

class Gradient2Limit:
    def __init__(self, init=10, min_limit=1, max_limit=200):
        self.limit = init
        self.min_limit = min_limit
        self.max_limit = max_limit
        self.long_avg_rtt = 0.0
        self.short_avg_rtt = 0.0

    def on_sample(self, in_flight, rtt_ms, did_drop):
        if did_drop:
            self.limit = max(self.min_limit, int(self.limit * 0.9))
            return
        # EMA
        a_long, a_short = 0.95, 0.7
        self.long_avg_rtt  = a_long  * self.long_avg_rtt  + (1 - a_long)  * rtt_ms
        self.short_avg_rtt = a_short * self.short_avg_rtt + (1 - a_short) * rtt_ms
        gradient = max(0.5, min(1.0, self.long_avg_rtt / self.short_avg_rtt))
        new_limit = self.limit * gradient + 4
        self.limit = max(self.min_limit, min(self.max_limit, new_limit))

Drop on excess concurrency: signal upstream with Retry-After. Reference: https://github.com/Netflix/concurrency-limits

16.8 Bulkhead Pattern

Isolate resources by partition: separate thread pools, connection pools, semaphores per dependency. Failure of one dependency can't exhaust shared resources used by others. Used in Hystrix, Resilience4j.

import asyncio
class Bulkhead:
    def __init__(self, max_concurrent):
        self.sem = asyncio.Semaphore(max_concurrent)
    async def call(self, fn, *args):
        async with self.sem:
            return await fn(*args)

Per-dependency bulkheads: hot dependency gets its own bulkhead; cold ones share. Prevents single dependency saturation from blocking everything.

16.9 LLM-Specific Backpressure

LLM serving has unique challenges: - Output length is unknown until completion. - KV memory is the dominant resource, not CPU. - Premature drop wastes prefill (sunk cost).

Strategies: 1. Queue-depth-weighted-by-output-tokens for autoscale signal (see §11.4). 2. Per-tenant adaptive concurrency limits rather than static rate limits. 3. KV-budget admission: estimate KV_needed = prompt_tokens + max_output × 320KB (Llama-3.1-70B). Reject if exceeds available. 4. Cancellation propagation: client disconnects → server stops generating. gRPC RST_STREAM, HTTP/2 RST_STREAM. vLLM v1 supports cancellation natively.

   Backpressure signals in LLM serving:
   ─────────────────────────────────────
   Edge gateway:   per-API-key RPS via token bucket (Redis)
   Router:         queue depth × E[output_tokens] vs SLO budget
   Replica:        KV occupancy vs total; reject @ 90%
   Worker:         continuous batching natural backpressure
                    (admit only what fits in token budget)

16.10 Google SRE Adaptive Throttler (Client-Side)

For RPCs to a backend, client maintains rolling window of (requests, accepts). Throttle probability:

   p_throttle = max(0, (requests - K * accepts) / (requests + 1))

with K = 2. When backend rejects, client throttles itself rather than retrying.

class ClientThrottler:
    def __init__(self, k=2.0, window_s=60):
        self.k = k
        self.window = collections.deque()       # (timestamp, accepted_bool)

    def _gc(self):
        cutoff = time.time() - 60
        while self.window and self.window[0][0] < cutoff:
            self.window.popleft()

    def should_send(self):
        self._gc()
        reqs = len(self.window)
        accepts = sum(1 for _, a in self.window if a)
        p = max(0, (reqs - self.k * accepts) / (reqs + 1))
        return random.random() >= p

    def record(self, accepted: bool):
        self.window.append((time.time(), accepted))

Reference: https://sre.google/sre-book/handling-overload/

16.11 Decision Sheet

Context Recommended pattern
In-process pipeline (single language) Reactive Streams / Project Reactor
Cross-process (within DC) gRPC + flow control + adaptive concurrency
Cross-DC Token bucket + retry budget + circuit breaker
Streaming media Credit-based with WINDOW_UPDATE
LLM serving KV-budget admission + queue-depth-weighted scaling
Network qdisc fq_codel (CoDel)
Load balancer Adaptive concurrency limits (Netflix)
Untrusted external producer Reject 429 with Retry-After
Telemetry / metrics ingest Drop-oldest with sampling

16.12 Common Interview Follow-ups

Q: Producer is faster; what do you do? A: Apply backpressure or drop. Choose based on the data's value: state updates → block; telemetry → drop oldest with sampling.

Q: How do you avoid head-of-line blocking when one slow consumer holds up all producers? A: Bulkheads — independent queues per consumer. Or asynchronous fanout (each consumer pulls from a per-consumer offset, like Kafka topics).

Q: How do you size a bounded queue? A: Little's Law: L = λ × W. If max latency budget is 100ms and arrival rate 1000/s, queue capacity > 100 keeps you in spec. Add 2× buffer for bursts. Beyond that, drop or reject.

Q: What's the connection between TCP and Reactive Streams? A: Both are demand-driven backpressure protocols. TCP rwnd is to bytes as Reactive Streams request(n) is to messages. Conceptually identical.

Q: How do you avoid tail latency from long-running requests blocking new ones? A: Pre-emption (Llumnix for LLM), priority queues, quota-based scheduling. Long-running requests pay-as-they-go; can be evicted when over their quota.
