Idempotency in Distributed Systems

Stripe's reference state machine, transactional outbox/inbox, Kafka EOS, and the SAGA-with-compensation playbook.

6.1 The Problem

A client sends a payment request; the server processes it; the response is lost in the network. Client times out and retries. Without care, the customer is charged twice.

The fix: idempotency keys — a client-generated unique ID per logical operation. The server records the key on first processing; subsequent requests with the same key return the same response without re-executing side effects.

6.2 The Stripe Pattern (Brandur's Reference Impl)

Reference: https://brandur.org/idempotency-keys

CREATE TABLE idempotency_keys (
    key                 TEXT PRIMARY KEY,
    user_id             BIGINT NOT NULL,
    request_method      TEXT NOT NULL,
    request_path        TEXT NOT NULL,
    request_params_hash TEXT NOT NULL,    -- SHA256 of canonical body
    response_status     INTEGER,
    response_body       JSONB,
    recovery_point      TEXT NOT NULL,    -- state machine cursor
    locked_at           TIMESTAMPTZ,
    created_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
    expires_at          TIMESTAMPTZ NOT NULL DEFAULT now() + interval '24h'
);
CREATE UNIQUE INDEX ON idempotency_keys (user_id, key);

State machine for an operation that has multiple side effects (e.g., charge card → record payment → send email):

   started → charged → recorded → emailed → finished

Each state transition is a Postgres transaction that updates recovery_point. If the client retries, the server resumes from recovery_point instead of restarting. This is resumable, idempotent execution.

If request_params_hash differs from stored, return 422 Unprocessable Entity (key reuse with different body — likely a client bug).

6.3 The Crunchy Variant

Brandur's Crunchy variant uses 1h retention (vs. 24h Stripe) — tradeoff between safety and storage. Reference: https://brandur.org/fragments/idempotency-keys-crunchy

6.4 IETF Draft: Idempotency-Key

Standardizes the header. Recommends: - Server stores response for ≥ 24h. - On in-progress concurrent retry, return 409 Conflict. - On mismatch (same key, different body): 422.

Draft: https://www.ietf.org/archive/id/draft-ietf-httpapi-idempotency-key-header-06.html

6.5 Outbox + Inbox Pattern

For event-driven pipelines, the consumer side needs an inbox table:

CREATE TABLE inbox (
    event_id TEXT PRIMARY KEY,    -- producer-set unique id
    processed_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

In the same transaction that performs the side effect, INSERT into inbox. The PRIMARY KEY constraint dedupes — if event already processed, INSERT fails, transaction rolls back.

def handle_event(evt):
    with db.transaction():
        try:
            db.execute("INSERT INTO inbox(event_id) VALUES (%s)", evt.id)
        except UniqueViolation:
            return  # already processed
        # idempotent action goes here, committed atomically with the inbox row
        update_state(evt)

6.6 Kafka Exactly-Once Semantics (EOS)

   producer.initTransactions()
   producer.beginTransaction()
   producer.send(...)           # to topic A
   producer.send(...)           # to topic B
   producer.sendOffsetsToTransaction(consumed_offsets, "consumer-group")
   producer.commitTransaction()

With enable.idempotence=true + transactional.id + consumer's isolation.level=read_committed, you get end-to-end EOS for the read-process-write loop in Kafka.

Reference: https://www.confluent.io/blog/exactly-once-semantics-are-possible-heres-how-apache-kafka-does-it/

6.7 Saga Pattern + Compensation

Long-running operations across multiple services use the Saga pattern: a sequence of local transactions, each with a compensating action. On failure, run compensations in reverse.

Tools: Temporal, AWS Step Functions, Camunda.

6.9 Detailed State Machine for Stripe-Style Idempotency

The "5-state machine" model:

   [received] ──HTTP 200 if existing────▶ replay response
       │
       │ first time seeing this key (lock acquired)
       ▼
   [processing] ── crash ─▶ resume on retry ─▶ continue from recovery_point
       │
       │ side effect 1 done; recovery_point = 'after_charge'
       ▼
   [partial_a] ─ crash ─▶ resume from after_charge
       │
       │ side effect 2 done; recovery_point = 'after_record'
       ▼
   [partial_b] ─ crash ─▶ resume from after_record
       │
       │ side effect 3 done; response built
       ▼
   [finished] ──HTTP 200 with response──▶ done

Each transition writes its own row update in the same database transaction as the side effect (when the side effect is in the same DB) or in a follow-up confirmation step (when the side effect is external — see two-phase variant below).

6.10 Two-Phase Idempotency for External Side Effects

When a side effect is outside the DB (charge a credit card via Stripe API), you can't atomically commit "side effect done" with "DB updated". The pattern:

  1. Phase 1 (intent): in DB transaction, mark request as processing, store recovery_point=pending_charge.
  2. Phase 2 (call external): invoke the external API with an idempotency key derived from your internal ID. Wait for response.
  3. Phase 3 (commit): in a new DB transaction, store the external response, advance recovery_point=charged.

If we crash between phase 2 and 3: on retry, look up by idempotency key at the external service first. Stripe's API supports this — retrying with the same key returns the original result, not a duplicate.

This is "transactional outbox + external idempotency key" combined.

6.11 Race Conditions in Distributed Idempotency

Two concurrent requests with the same key arrive at different servers:

   Server A: SELECT ... WHERE key = 'X'  → not found
   Server B: SELECT ... WHERE key = 'X'  → not found
   Server A: INSERT (key='X', ...)        → succeeds
   Server B: INSERT (key='X', ...)        → fails (UNIQUE violation)

The primary key constraint serializes. Server B can either: - Return 409 Conflict ("already in progress"). - Wait + retry, then read the response.

The IETF draft recommends 409. Most APIs return 409 with Retry-After.

6.12 Idempotency Window vs Retention

  • Window: how long after submission a retry with the same key is honored.
  • Retention: how long the response is stored.

Stripe: 24h. Crunchy: 1h. PayPal: forever (per their docs). Different SLAs reflect different fraud / replay risk profiles.

6.13 Compound Idempotency Keys

For deeply nested operations (e.g., a multi-line invoice), key = hash of (user_id, idempotency_key, request_body_hash). This protects against: - Same key + different body → 422. - Same key + same body → replay response.

Brandur's reference includes a request_params_hash column for this exact reason.

6.14 Outbox + Inbox: The Complete Pattern

   ┌──────────── Service A ──────────────┐
   │  BEGIN TX                           │
   │   UPDATE business_state ...         │
   │   INSERT outbox(event_id, payload)  │
   │  COMMIT TX                          │
   │       │                             │
   │       ▼ (CDC)                       │
   └──────────────────────────────────────┘
              │ Kafka / SQS / etc.
              ▼
   ┌──────────── Service B ──────────────┐
   │  BEGIN TX                           │
   │   INSERT inbox(event_id) → unique   │
   │   UPDATE business_state ...         │
   │  COMMIT TX                          │
   │  (on UNIQUE violation: skip)        │
   └──────────────────────────────────────┘

This gives end-to-end exactly-once semantics across services without distributed transactions.

6.15 Kafka Transactions Deep Dive

The Kafka EOS pattern in detail:

# Producer with idempotence
producer = KafkaProducer(
    bootstrap_servers='...',
    enable_idempotence=True,
    transactional_id='unique-tx-id-per-producer',
    acks='all',
    max_in_flight_requests_per_connection=5,  # max for idempotence
)
producer.init_transactions()

while True:
    records = consumer.poll(...)
    producer.begin_transaction()
    for r in records:
        # process
        producer.send('output_topic', transformed(r))
    producer.send_offsets_to_transaction(
        offsets={tp: r.offset+1 for tp, r in last_records.items()},
        consumer_group_metadata=consumer.group_metadata())
    producer.commit_transaction()

What guarantees Kafka EOS: 1. enable.idempotence=true: each producer has a producer ID and sequence number; broker dedupes within a session. 2. transactional.id: identifies the producer across restarts; broker fences old instances. 3. send_offsets_to_transaction: commits consumer offsets as part of the same transaction. 4. Consumers with isolation.level=read_committed: only see committed transaction outputs.

Reference: https://www.confluent.io/blog/exactly-once-semantics-are-possible-heres-how-apache-kafka-does-it/

6.16 Common Pitfalls

  • Storing only the response, not the recovery_point: on crash mid-processing, retry would re-process from scratch — duplicates if side effects are non-idempotent.
  • Forgetting to lock during processing: two concurrent retries both run side effects.
  • Using request body as the key: clients can't reliably retry exactly the same body (timestamps, retry counters change).
  • Letting client choose key freely without validation: client could collide with another client. Scope key per user.
  • Garbage-collecting too aggressively: client retries after retention window expires → duplicate processing.
  • Using non-monotonic generation: in distributed systems, ensure key uniqueness via UUID or per-client counter.

6.17 Saga Compensation Patterns

When you can't have idempotent commit, use compensating actions:

   ┌─ Charge → ─ Reserve Inventory → ─ Ship → ─ Notify ─┐
   │                                                    │
   │ on error:                                          │
   │   undo Notify ← undo Ship ← undo Reserve ← Refund │
   └────────────────────────────────────────────────────┘

Each forward action has an inverse. Failures roll back via inverses. State machine tracks progress; on crash, resume forward or backward.

Implementations: Temporal, AWS Step Functions, Camunda, Eventuate.

6.18 Finer Points

  • Idempotency vs. exactly-once: idempotency is at the application protocol level; exactly-once is a stronger consistency property. Idempotency + at-least-once delivery ≈ exactly-once effects.
  • Conditional writes (DynamoDB, S3, Postgres WHERE version = X): a form of optimistic-concurrency that gives idempotency for in-place updates.
  • HTTP methods: PUT and DELETE are defined idempotent in HTTP; POST is not. But idempotency-key headers turn POST idempotent.
