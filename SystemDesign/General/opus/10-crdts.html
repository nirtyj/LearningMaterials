10.1 What CRDTs Are

A Conflict-Free Replicated Data Type is a data type whose state can be replicated across nodes and updated independently, where merging always converges to the same value (Strong Eventual Consistency, SEC).

Two flavors: - State-based (CvRDT): replicas exchange full state; merge is associative, commutative, idempotent (a join-semilattice). - Op-based (CmRDT): replicas exchange operations; messages are commutative; need reliable causal broadcast.

Foundational paper: Shapiro et al., A comprehensive study of CRDTs (INRIA RR-7506, 2011). https://hal.inria.fr/inria-00555588/document

10.3 RGA — Replicated Growable Array (the canonical text CRDT)

Each character has a unique id = (timestamp, replica_id). Inserting at position p means "insert after the character whose id is at position p-1". Concurrent inserts at the same anchor are ordered by id (e.g., higher timestamp wins).

   Initial:  [A]
   Replica 1 inserts B after A, id=(2, R1) → [A, B(2,R1)]
   Replica 2 inserts C after A, id=(2, R2) → [A, C(2,R2)]
   Merge:    sort children of A by id desc → [A, C(2,R2), B(2,R1)]
             (deterministic — both replicas converge)

Tombstones: deleted characters become tombstones (kept for causal context). Garbage collection requires consensus on which tombstones can be removed.

10.4 Yjs YATA

YATA is RGA-flavored but simpler integration: - Each block is (id, origin_left, origin_right, content). - Insertion finds its place between origin_left and origin_right. - Concurrent inserts at the same place use lexicographic id ordering.

Yjs encodes blocks compactly; runs of consecutive characters from the same client are merged into one block. This is why Yjs is so much smaller and faster than naïve RGA.

10.5 Automerge JSON

Automerge supports arbitrary JSON. Each map field is an MV-register; each list is RGA; nested objects are independent CRDTs. v2 changed the storage to a binary format with offset compression.

Comparison with Yjs (Kleppmann's B4 benchmark):

Library Time Memory
Yjs 970 ms 3 MB
Automerge v1 291 s 880 MB
Automerge v2 < 5 s 30 MB

10.6 The Tree-Move Problem

Moving an item to a new parent has a subtle issue: if A moves item X under Y, and concurrently B moves Y under X, you get a cycle. Kleppmann's paper A Highly Available Move Operation for Replicated Trees (TPDS 2022) presents a deterministic resolution: order operations by timestamp, undo conflicting moves on reception.

Reference: https://martin.kleppmann.com/2021/10/07/crdt-tree-move-operation.html

10.7 OT vs CRDT (Joseph Gentle: "I was wrong")

OT was the dominant approach 2000s-2010s (Google Docs, Etherpad). CRDTs took over from ~2018 onward. Joseph Gentle (ShareJS author) wrote I was wrong, CRDTs are the future: https://josephg.com/blog/crdts-are-the-future/

Key arguments: - CRDTs work without a central server; OT requires one. - CRDTs can be authored offline and synced. - Modern CRDTs (Yjs, Loro) match OT performance.

But: server-mediated apps (Figma, Notion) often choose hybrid because CRDT semantics don't always match user intent (e.g., "I edited this, the other person's old edit shouldn't reappear" needs causal context).

10.8 Production CRDT Usage

  • Yjs: TipTap, BlockNote, Liveblocks, Hocuspocus, JupyterLab.
  • Automerge: Ink+Switch demos, some local-first apps.
  • Loro: emerging; used in Affine.
  • Riak: ORSWOT (subset of OR-Set) for distributed key-value semantics.
  • Redis Enterprise: CRDB feature; geo-replicated CRDTs.
  • Figma: LWW-per-property with tree-CRDT for hierarchy.

10.9 Limitations / Tradeoffs

  • Metadata overhead: tombstones, vector clocks, IDs per element.
  • Garbage collection is hard — when can we forget a tombstone?
  • Causal broadcast — op-based CRDTs need it; over WAN this means buffering.
  • No invariants across keys: CRDTs guarantee per-key convergence, not "balance ≥ 0" across an account.

For invariants, you need either consensus or compensating actions.

10.11 State-Based vs Op-Based — Concrete Example

G-Counter (state-based)

State at each replica: vector<int> indexed by replica ID.

class GCounter:
    def __init__(self, replica_id, num_replicas):
        self.id = replica_id
        self.counts = [0] * num_replicas

    def increment(self, n=1):
        self.counts[self.id] += n

    def value(self):
        return sum(self.counts)

    def merge(self, other):
        # Element-wise max; commutative, associative, idempotent
        for i in range(len(self.counts)):
            self.counts[i] = max(self.counts[i], other.counts[i])

The merge is a join in a semilattice (max is a least-upper-bound operation). All replicas converge to the same value regardless of merge order or message duplication.

G-Counter (op-based)

class GCounterOp:
    def __init__(self):
        self.value = 0

    def apply(self, op):
        # op = ("inc", n)
        if op[0] == "inc":
            self.value += op[1]

Op-based: just broadcast ("inc", 1). Increment is commutative. But: needs reliable, exactly-once delivery (else under/over-count). State-based is more robust to network shenanigans.

10.12 OR-Set Construction

The naive set CRDT (just union) doesn't allow removal. The 2P-Set adds a remove-set, but you can't re-add an element after removing.

OR-Set (Observed-Remove): tag each add with a unique id. On remove, mark all currently-known ids as removed.

class ORSet:
    def __init__(self):
        self.adds = {}    # element → set of unique ids
        self.tomb = {}    # element → set of removed ids

    def add(self, x):
        uid = uuid.uuid4().hex
        self.adds.setdefault(x, set()).add(uid)
        return uid

    def remove(self, x):
        ids = self.adds.get(x, set())
        self.tomb.setdefault(x, set()).update(ids)

    def contains(self, x):
        live = self.adds.get(x, set()) - self.tomb.get(x, set())
        return bool(live)

    def merge(self, other):
        for x, ids in other.adds.items():
            self.adds.setdefault(x, set()).update(ids)
        for x, ids in other.tomb.items():
            self.tomb.setdefault(x, set()).update(ids)

Concurrent add and remove: add wins (the add's uid wasn't in the remove-set when remove ran). This matches user intuition for "I just added, don't remove based on stale state".

10.13 RGA Insertion Algorithm Detail

class RGANode:
    def __init__(self, char, id, parent_id):
        self.char = char
        self.id = id              # (timestamp, replica)
        self.parent = parent_id   # the node this was inserted after
        self.children = []        # child nodes; sorted by id desc
        self.tombstone = False

class RGA:
    def __init__(self, replica_id):
        self.replica_id = replica_id
        self.clock = 0
        self.root = RGANode(None, (0, 0), None)
        self.index = {(0, 0): self.root}

    def _next_id(self):
        self.clock += 1
        return (self.clock, self.replica_id)

    def insert_after(self, parent_id, char):
        new_id = self._next_id()
        node = RGANode(char, new_id, parent_id)
        parent = self.index[parent_id]
        # Insert into children sorted by id descending
        # so concurrent inserts at same parent are deterministically ordered
        bisect.insort(parent.children, node, key=lambda n: -n.id)
        self.index[new_id] = node
        return new_id

    def delete(self, node_id):
        if node_id in self.index:
            self.index[node_id].tombstone = True

    def render(self):
        out = []
        def walk(node):
            for c in node.children:
                if not c.tombstone:
                    out.append(c.char)
                walk(c)
        walk(self.root)
        return "".join(out)

The id sort gives determinism under concurrency: replicas that see the same set of operations produce the same render order, regardless of order of receipt.

10.14 YATA (Yjs) Block Encoding

Yjs encodes runs of consecutive characters from the same client into a "block":

   block = {
     id: (client_id, clock_start, length),
     content: "hello world",
     origin_left: <id of left neighbor>,
     origin_right: <id of right neighbor>,
   }

This dramatically compresses storage when one user types a long sequence. A 10000-character paste becomes one block, not 10000 nodes.

Splits happen on: - A different client's insertion in the middle. - A delete in the middle (creates two blocks with a tombstone gap).

This is why Yjs's memory and disk usage are 1-2 orders of magnitude smaller than naive RGA.

10.15 Automerge's Append-Only Op Log

Automerge stores all operations in a log; the document state is derived by applying them in causal order.

   ops = [
     {action: "set", obj: ROOT, key: "title", value: "Doc",   pred: []},
     {action: "set", obj: ROOT, key: "title", value: "Notes", pred: [op1.id]},
     ...
   ]

Each op cites which prior ops it supersedes (pred). Concurrent ops are ones not in each other's pred. Automerge resolves concurrent register writes via MVR (kept as multi-value).

Versioning: any state can be reconstructed by applying any prefix of the log. Branching: hand someone a set of ops; they can replay in their own causal order.

Reference: https://arxiv.org/pdf/1608.03960

10.16 Loro's Eg-Walker

Eg-walker (Event Graph Walker) by Loro re-thinks RGA. Instead of maintaining the full tree, store ops in a DAG ordered by causal dependencies. To compute current state, walk the DAG.

Key advantage: most operations don't need to materialize the full tree; they walk only the paths that are causally ordered relative to themselves. This sidesteps the O(N²) cost of OT and the heavy metadata cost of classical RGA.

Reports on Kleppmann's B4 trace are competitive with Yjs (often within 2×) while supporting time-travel via DAG checkpoints.

Reference: https://arxiv.org/abs/2409.14252

10.17 The Tree-Move Problem (Kleppmann 2022)

For a filesystem-like CRDT (folders containing items), naive concurrent moves can create cycles:

   Initial: A → B → C
   User 1: Move B inside C  → A, C → B
   User 2: Move C inside B  → A, B → C

   Naive merge: B → C and C → B → cycle

Solution: each move op has timestamp; on receive, replicas check if applying would create a cycle; if so, undo the op and re-apply later in different order. The deterministic algorithm guarantees all replicas converge.

Reference: https://martin.kleppmann.com/2021/10/07/crdt-tree-move-operation.html

10.18 Garbage Collection in CRDTs

Tombstones, op logs, and version vectors grow unboundedly. GC strategies: - Stable point detection: an op is stable when all replicas have observed it. Stable ops can be compacted into "snapshots". - Causal-stability protocol: each replica reports its observed-version vector; intersection is the stable frontier. - Lossy GC: drop history older than X days at the cost of losing time-travel.

Yjs supports incremental GC; Automerge has snapshot+log compaction; Loro's DAG checkpoints define stable boundaries.

10.19 OT vs CRDT — When Each Wins

Property OT CRDT
Server required Yes (single point of order) No (peer-to-peer OK)
Per-op overhead Low (just position) Higher (id, refs)
Offline editing Hard Native
Algorithmic complexity T() function per op-pair Merge / Apply
Convergence proof TP1 / TP2 conditions Semi-lattice math
Modern usage Google Docs, Etherpad Yjs, Automerge, Loro

Today's consensus (Joseph Gentle's blog): CRDTs win for most new builds. OT remains for systems already invested in server-mediated single-source-of-truth.
