Staff-Level System Design on AWS — Architecture Diagrams, Component Deep Dives, Technologies & Patterns

How to Use This Guide

This guide covers the complete design of a Conversational Ad Experiences (CAX) platform — the system that powers sponsored content inside AI shopping conversations on Amazon. It's organized in layers, from big picture to real AWS API calls.

Key Insight: Three-Plane Separation

The Consumer Plane is latency-critical (p99 < 2s end-to-end for full conversational response). The Ad Decisioning Plane is throughput-critical (billions of auctions/day). The Advertiser Plane is consistency-critical (budget accuracy, attribution correctness). Each plane scales independently with different AWS primitives.

Critical Design Decision: Parallel Retrieval + LLM Prefill

The auction pipeline (retrieval → scoring → settlement) runs in parallel with LLM prefill (loading system prompt + conversation history into KV cache). This overlaps the ~100ms auction with the ~200ms prefill, hiding auction latency entirely. Winners are injected into the generation context before the first output token. This is the key architectural innovation that makes conversational ads feasible at ad-system latency.

DynamoDB Schema Design

The single-table design uses user_id as PK and session_id#turn_num as SK for efficient range queries ("get last 10 turns"). A GSI on session_id alone enables session-level queries. Ads metadata is denormalized into each turn for attribution — no joins needed. DAX provides microsecond reads for the hot conversation path.

API Gateway Layer: Explained From Scratch

Before we get into the technical details, let's understand what problem this layer solves. When Nirty types "I need a dishwasher installer" into Amazon's shopping assistant, that message needs to travel from their browser, through the internet, through Amazon's network, arrive at the right server, get authenticated ("is this a real Amazon customer?"), and get routed to the right service. That's what the API Gateway Layer does — it's the front door of the entire system.

Why NOT Use API Gateway (the AWS Service)?

AWS has a service literally called "API Gateway" — so why don't we use it? This is a common interview question:

API Gateway WebSocket Limitations (Why We Avoid It)

Connection limit: API Gateway WebSocket maxes out at 500 new connections per second per account. For Amazon's traffic (millions of concurrent shoppers), this is a hard blocker.

Message size: 128KB max per WebSocket frame. LLM responses with product data can exceed this.

Timeout: 10-minute idle timeout, 2-hour max connection. Shopping sessions can last longer.

No streaming control: You can't easily implement backpressure (slowing down if the client can't keep up).

Cost: API Gateway charges per message ($1 per million). At billions of token-level messages per day, this adds up fast. ECS Fargate is flat-rate compute.

WebSocket vs SSE vs HTTP — Why WebSocket?

The LLM generates tokens one by one, and we want to show them to the user as they arrive (the "typing" effect you see in ChatGPT). There are three ways to stream data from server to browser:

ProtocolDirectionHow It WorksGood For CAX?
Regular HTTPRequest → Response (one-shot)Client sends request, waits for ENTIRE response, then displays it❌ User waits 2 seconds staring at a spinner, then sees everything at once. Terrible UX.
SSE (Server-Sent Events)Server → Client onlyServer pushes events to client over a long-lived HTTP connection. Client can't send messages back on the same connection.⚠️ Works for web browsers (token streaming). But the user can't send a new message while receiving tokens without opening a second connection. Also doesn't work for Alexa/voice.
WebSocketBoth directions, any timeFull-duplex: both sides can send messages at any time. Stays open indefinitely.✅ User sees tokens streaming AND can interrupt ("actually, never mind"). Works for all surfaces. Single connection for the whole session.

Semantic Cache: Explained From Scratch

Before the system calls the LLM (which costs money and takes time), it asks: "Have I seen a question like this before?" If yes, it can return a previous answer instantly. This is the semantic cache.

Why Two Layers? (ElastiCache → OpenSearch)

  SEMANTIC CACHE — THE TWO-LAYER DESIGN:

  Nirty asks: "I need someone to install my dishwasher"
     │
     ▼
  ┌──────────────────────────────────────────────────────────────┐
  │  LAYER 1: EXACT MATCH (ElastiCache Redis, <1ms)             │
  │                                                              │
  │  How: Hash the query text → look up in Redis                 │
  │                                                              │
  │  hash("I need someone to install my dishwasher")             │

  │    GET cache:a7f3b2e1c9d8...                                 │
  │                                                              │

  │  If not → MISS (try Layer 2)                                 │
  │                                                              │
  │  This catches: Nirty asking the same thing twice, or two     │
  │  different users typing the EXACT same question.             │
  │                                                              │
  │  Hit rate: ~10% (most people word things differently)        │
  └──────────────────────────────────────────────────────────────┘
     │ MISS
     ▼

  │                                                              │

The Ad Freshness Problem and Skeleton Caching

There's a catch: cached responses contain specific ad results. But ads change constantly — an advertiser's budget might run out 2 minutes after we cached a response. If we serve a stale cached response, we might show an ad for an installer who's no longer running their campaign.

The solution is skeleton caching — we cache the response structure and organic content separately from the ads:

  SKELETON CACHING EXAMPLE:

  WHAT WE CACHE (the "skeleton"):
  ┌──────────────────────────────────────────────────────────────┐
  │  {                                                           │
  │    "query_embedding": [0.23, -0.87, ...],                    │
  │    "intent": "SERVICE_SEARCH",                               │
  │    "organic_results": [                                      │
  │      {"name": "HandyHub", "price": 189, "rating": 4.7}      │
  │    ],                                                        │
  │    "response_skeleton": "Great choice on the {product}!      │
  │      I found several installers in {location}. Here are      │
  │      your best options:\n\n{AD_SLOT_1}\n\n{AD_SLOT_2}\n\n    │
  │      {ORGANIC_1}\n\nWould you like to compare these?",      │
  │    "ad_slot_count": 2,                                       │
  │    "cached_at": "2026-03-29T10:30:00Z",                     │
  │    "ttl": 900  // 15 minutes                                 │
  │  }                                                           │
  └──────────────────────────────────────────────────────────────┘

  WHAT WE DON'T CACHE: the specific winning ads.

  ON CACHE HIT:
  1. Retrieve the skeleton (organic text + slot positions)  → 5ms
  2. Run ONLY the auction (skip retrieval + LLM entirely)   → 15ms
  3. Fill {AD_SLOT_1} and {AD_SLOT_2} with fresh winners   → 1ms
  4. Return the assembled response                          → <1ms
  ────────────────────────────────────────────────────────
  Total: ~22ms instead of ~1000ms — a 45× speedup!

  We skip the most expensive parts (RAG retrieval + LLM generation)
  while still serving fresh, currently-valid ads.

  # Real Redis + OpenSearch implementation for semantic cache

  import hashlib
  import json
  import numpy as np

  async def check_semantic_cache(query, session_context):
      """
      Two-layer cache check. Returns cached response or None.
      """
      
      # ── LAYER 1: Exact match in Redis ──────────────────────
      cache_key = hashlib.sha256(
          f"{query}:{session_context['location']}".encode()
      ).hexdigest()
      
      cached = await redis.get(f"cache:exact:{cache_key}")
      if cached:
          skeleton = json.loads(cached)
          # Re-run ONLY the auction with fresh ad data
          fresh_ads = await run_auction_only(skeleton['intent'])
          return fill_skeleton(skeleton, fresh_ads)
      
      # ── LAYER 2: Semantic match in OpenSearch ──────────────
      query_embedding = await get_embedding(query)  # Titan, ~3ms
      
      os_response = await opensearch.search(
          index="semantic-cache",
          body={
              "size": 1,
              "query": {
                  "knn": {
                      "query_embedding": {
                          "vector": query_embedding,
                          "k": 1
                      }
                  }
              },
              "min_score": 0.8  # cosine similarity threshold
          }
      )
      
      if os_response['hits']['total']['value'] > 0:
          hit = os_response['hits']['hits'][0]
          similarity = hit['_score']
          
          if similarity >= 0.8:
              skeleton = hit['_source']
              fresh_ads = await run_auction_only(skeleton['intent'])
              return fill_skeleton(skeleton, fresh_ads)
      
      # ── BOTH LAYERS MISSED ─────────────────────────────────
      return None  # proceed with full pipeline


  async def cache_response(query, intent, organic_results,
                            response_text, ad_slot_positions):
      """
      After generating a response, cache it for future queries.
      """
      query_embedding = await get_embedding(query)
      cache_key = hashlib.sha256(query.encode()).hexdigest()
      
      skeleton = {
          "intent": intent,
          "organic_results": organic_results,
          "response_skeleton": extract_skeleton(
              response_text, ad_slot_positions
          ),
          "ad_slot_count": len(ad_slot_positions),
          "cached_at": datetime.utcnow().isoformat()
      }
      
      # Layer 1: exact match (5-minute TTL)
      await redis.setex(
          f"cache:exact:{cache_key}",
          300,  # 5 minutes
          json.dumps(skeleton)
      )
      
      # Layer 2: semantic match (15-minute TTL via OpenSearch)
      await opensearch.index(
          index="semantic-cache",
          body={
              "query_text": query,
              "query_embedding": query_embedding,
              **skeleton,
              "ttl_timestamp": (datetime.utcnow()
                  + timedelta(minutes=15)).isoformat()
          }
      )

The Architecture Pattern: Adapter + Canonical Message

This is a classic Adapter Pattern from software engineering. Each surface (browser, mobile, voice, API) has its own adapter that translates between the surface's protocol/format and a single internal canonical format (InternalConversationMessage). The entire backend pipeline processes only the canonical format. On the way out, the adapter translates the response back to the surface's format.

This means adding a new surface (say, a car dashboard, or AR glasses) requires ONLY writing a new adapter. Zero changes to the Intent Agent, Retrieval, Auction, LLM, or Guardrails pipeline. That's the power of clean separation.

Auction Pipeline Explained From Scratch

Imagine you're selling a house. You put it on the market and multiple buyers make offers. Your real estate agent doesn't just pick the highest bidder — they also consider who's most likely to close (financing approved? serious buyer?) and who's the best fit. That's essentially what the Auction Engine does, except it happens in 15 milliseconds instead of 15 days, and it's happening for every single conversation turn on Amazon.

The Setup: Who's Bidding?

Before any user ever types a message, thousands of businesses have already set up advertising campaigns on Amazon. Let's say there are 200 appliance installation services in California. Each one has configured:

Example: HomeServe Pro's Campaign Setup (done once by the business)

Campaign name: "Bay Area Dishwasher Installation"

Daily budget: $500 (once this runs out, stop showing my ad today)

Max bid per click: $2.50 (the most I'm willing to pay when someone engages with my ad)

Targeting: Categories: "home_services", "appliance_installation". Locations: San Francisco Bay Area. Keywords: "dishwasher install", "appliance setup"

Discount rules: Allow up to 10% off for users who express price sensitivity

HomeServe Pro set this up on Amazon's advertiser dashboard (like Google Ads). They don't know when their ad will show — the system decides that automatically every time a relevant conversation happens.

Stage 1: Candidate Enrichment — "Who's still in the game?"

The Retrieval Agent (explained in section 3c) just handed us 50 installer businesses that are relevant to Nirty's query. But not all 50 can actually show ads right now. Stage 1 checks each one against reality:

  50 candidates from Retrieval Agent
     │
     ▼  For EACH candidate, check (all from Redis, <1ms per lookup):
     │
     │  HomeServe Pro:
     │    Budget remaining today? $487 ✅ (has money left)
     │    Daily impression cap hit? 234/1000 ✅ (under cap)
     │    Serves Livermore, CA? ✅ (in their geo-target)
     │    Campaign active? ✅ (not paused)
     │    → PASSES, stays in auction
     │
     │  QuickFix Larry:
     │    Budget remaining today? $0.03 ❌ (basically broke!)
     │    → ELIMINATED (can't afford even one more click)
     │
     │  Premium Install Co:
     │    Serves Livermore, CA? ❌ (only SF proper)
     │    → ELIMINATED (wrong location)
     │
     │  50 → 20 candidates survive this filter

This is just a data lookup, not AI. It's checking a fast in-memory database (Redis) for each candidate's current state. Think of it like checking "is this restaurant currently open and accepting orders?" before showing it in a food delivery app.

Stage 2: Scoring — "Who DESERVES to win?"

This is the heart of the auction. We have 20 eligible candidates, but only 2–3 ad slots in Nirty's conversation. Who gets in?

The naive approach would be: whoever bids the most money wins. But that would be terrible for users — it would mean the richest companies always dominate, even if their service is bad. Instead, the system calculates an AdRank that balances money with quality:

The AdRank Formula

AdRank = Bid × pCTR × pCVR × QualityFactor

Bid = How much the advertiser is willing to pay per click ($2.50 for HomeServe)

pCTR (predicted Click-Through Rate) = "What's the probability this user will actually click/engage with this ad?" This is a machine learning model (not an LLM — it's a fast, lightweight model called LightGBM) that predicts: "Given that Nirty is searching for dishwasher installation in Livermore with a $200 budget, and HomeServe is a 4.8★ installer at $159... there's an 8% chance Nirty clicks." This model was trained on millions of past interactions.

pCVR (predicted Conversion Rate) = "If Nirty clicks, what's the probability they actually book?" Another ML model. It knows that users who state a specific budget and location convert at higher rates than users who are just browsing.

QualityFactor = A score from 0 to 1 based on the advertiser's review score, content quality, and how relevant they are to this specific conversation. HomeServe with 4.8★ and 2,340 reviews gets a 0.95. A new installer with 3.2★ and 12 reviews gets a 0.45.

Let's walk through the actual math for our dishwasher example with 5 candidates:

  SCORING EXAMPLE (for Nirty's "install my dishwasher" query):

  Candidate             Bid     pCTR   pCVR   Quality  AdRank    Rank
  ─────────────────     ─────   ────   ────   ───────  ────────  ────
  HomeServe Pro         $2.50   0.08   0.30   0.95     0.0570    🏆 1st
  Bay Area Experts      $3.00   0.06   0.25   0.88     0.0396    🥈 2nd
  InstallRight          $1.00   0.09   0.35   0.92     0.0290    🥉 3rd
  PremiumFix            $5.00   0.03   0.15   0.70     0.0158       4th
  BudgetInstall         $0.50   0.10   0.40   0.60     0.0120       5th

  WHAT'S HAPPENING HERE:

  ► PremiumFix bid the MOST ($5.00!) but ranks 4th.
    Why? Their pCTR is only 3% (users rarely click them — maybe
    their listing isn't compelling) and their quality score is
    low (0.70 — maybe bad reviews). Throwing money at the
    problem doesn't help if your service isn't good.

  ► HomeServe Pro wins DESPITE bidding only $2.50.
    Why? Users love them (8% click rate, 4.8★ reviews = 0.95
    quality). The system rewards businesses that users actually
    want to interact with.

  ► BudgetInstall has the BEST click rate (10%) but ranks last.
    Why? Their bid is only $0.50 — they're not willing to pay
    much. The system respects that but can't give them top
    placement over others who value the slot more.

  Result: HomeServe (#1) and Bay Area (#2) win the 2 ad slots.
  InstallRight is the first runner-up (matters for pricing).

The pCTR and pCVR models deserve emphasis: these are NOT LLMs. They're lightweight, fast machine learning models (LightGBM, small neural networks) that run on specialized chips (AWS Inferentia2) in under 2 milliseconds. They were trained offline on billions of past ad interactions. Their job is purely numerical prediction — "given these 20 input features, predict a probability between 0 and 1." They don't generate text, don't understand language, and don't reason. They're more like a very sophisticated calculator.

Stage 3: Settlement — "What does the winner actually pay?"

This is the clever part. HomeServe bid $2.50, but they don't pay $2.50. They use a second-price auction — they pay just barely enough to beat the person behind them.

Second-Price Auction: Why It's Genius

Imagine you're at an auction for a painting. You'd pay up to $1,000 for it. In a first-price auction (you pay what you bid), you'd try to game it — bid $600, hoping that's enough to win without overpaying. But what if someone else bid $601? You lose something you valued at $1,000.

In a second-price auction, you bid your true value ($1,000). If the next highest bid is $400, you only pay $401 (second price + $1). There's no reason to bid less than your true value because you'll never overpay. This encourages honest bidding.

Amazon uses this because it simplifies advertiser behavior — just set your true willingness to pay, and the system handles the rest.

  SETTLEMENT CALCULATION:

  Winner 1: HomeServe Pro (AdRank = 0.0570)
  ─────────────────────────────────────────
  Second-highest AdRank = 0.0396 (Bay Area Experts)
  HomeServe's QualityScore = pCTR × pCVR × Quality = 0.08 × 0.30 × 0.95 = 0.0228

  CPC = (Second_AdRank / Winner_QualityScore) + $0.01
      = (0.0396 / 0.0228) + $0.01
      = $1.74 + $0.01
      = $1.75 per click

  HomeServe bid $2.50 but pays only $1.75. They saved $0.75!

  Winner 2: Bay Area Experts (AdRank = 0.0396)
  ─────────────────────────────────────────────
  Third-highest AdRank = 0.0290 (InstallRight — the runner-up)
  Bay Area's QualityScore = 0.06 × 0.25 × 0.88 = 0.0132

  CPC = (0.0290 / 0.0132) + $0.01
      = $2.20 + $0.01
      = $2.21 per click

  Bay Area bid $3.00 but pays only $2.21. They saved $0.79!

Stage 4: Budget Decrement — "Deduct the charge"

After settlement, the system immediately deducts the CPC from each winner's budget. This uses a Redis Lua script — a tiny program that runs atomically (meaning no other operation can interrupt it) inside the Redis cache. It's like a bank transaction: debit the account and record the transaction as one unbreakable action.

Why Redis and not a regular database? Because this needs to happen in under 1 millisecond. DynamoDB would take 5–10ms. At billions of auctions per day, every millisecond counts. Redis stores all budget data in memory (RAM), not on disk, making it essentially instant. The data is written back to DynamoDB every 5 seconds as a permanent backup.

RAG Explained From Scratch: What, Why, and How

First — What IS "RAG"?

RAG stands for Retrieval-Augmented Generation. Despite the fancy name, it's a simple idea. Let's build up to it:

An LLM (like ChatGPT or Amazon's Nova) is a text-generating AI. It learned from billions of web pages and books. But it has two big problems: (1) It doesn't know about your specific data — it has no idea what installers are available in Livermore today, or what their current prices are. (2) It makes things up (hallucinations) — if you ask "what's HomeServe Pro's price?", it might confidently say "$129" when the real price is $159.

RAG solves both problems by adding a "retrieval" step before the LLM generates its response. Instead of asking the LLM to answer from memory, you:

RAG in Plain English

Step 1 (Retrieve): Search a database of real, current data to find relevant information

Step 2 (Augment): Paste that real data into the LLM's prompt as context

Step 3 (Generate): Ask the LLM to write a response using ONLY the provided data

Think of it like this: if you ask a random person on the street "what's a good dishwasher installer in Livermore?", they'd guess. But if you hand them a printed list of all installers with their prices and reviews, and THEN ask them to recommend one, they'd give you a useful, accurate answer. RAG is handing the LLM that printed list.

The term "RAG" is a specific technical concept from AI research (coined by Meta/Facebook in 2020), not just a flow name. It's now an industry-standard pattern used by virtually every LLM-powered application that needs real data. Amazon uses it in Rufus, Alexa+, and the CAX system we're designing.

Hybrid Search — Why Two Different Search Methods?

We now run two searches in parallel (meaning at the same time, not one after the other). Here's why we need both:

Search Method 1: Keyword Search (BM25)

This is classic text search — the same technology Google used in the early 2000s. It works by matching exact words. If you search "dishwasher installation", it finds documents containing those words and ranks them by how frequently and prominently those words appear (the BM25 algorithm).

Search Method 2: Vector (Semantic) Search

This is the newer, AI-powered approach. Instead of matching exact words, it matches meaning. Here's how it works:

Every piece of text can be converted into a list of 768 numbers (called a vector or embedding). These numbers represent the meaning of the text, not the words themselves. Texts with similar meanings have similar numbers. This conversion is done by a specialized model (Amazon Titan Text Embeddings) — not an LLM, but a smaller, faster model designed just for this purpose.

  VECTOR SEARCH EXAMPLE:

  Step 1: Convert the search query into numbers (embedding):

  "appliance installer plumber home service under $200"
     ↓ (Titan Embeddings model, ~3ms)
  [0.23, -0.87, 0.44, 0.11, -0.33, 0.78, ... ] ← 768 numbers

  Step 2: Compare against pre-computed embeddings of all ads:

  HomeServe Pro's embedding:
  [0.25, -0.85, 0.42, 0.13, -0.31, 0.76, ... ]
  Similarity to query: 0.94 (very similar meaning!) ✅

  Mike's Kitchen Fitters embedding:
  [0.21, -0.82, 0.39, 0.15, -0.29, 0.72, ... ]
  Similarity to query: 0.87 (similar meaning!) ✅ ← FOUND IT!
  
  Even though Mike's listing says "mount and connect kitchen
  appliances" and the query says "appliance installer" — the
  MEANING is the same, so the embeddings are similar.

  Random Plumber (only does pipes, not appliances):
  [0.56, -0.12, 0.88, -0.45, 0.67, -0.23, ... ]
  Similarity to query: 0.31 (very different meaning) ❌

  "Cosine similarity" is the math that measures how similar
  two lists of numbers are. It outputs a score from 0 (totally
  different) to 1 (identical meaning). We require ≥ 0.7.

Why do we need BOTH? Because each method catches things the other misses:

ScenarioKeyword SearchVector Search

Running both in parallel (at the same time) and merging results gives us the best of both worlds. Keyword search returns ~100 results, vector search returns ~100 results, with overlap.

RRF (Reciprocal Rank Fusion) — Merging Two Different Rankings

Now we have two separate ranked lists. How do we combine them into a single ranking? We can't just average the scores because the two scoring systems use completely different scales (BM25 scores might be 0–15, cosine similarity is 0–1).

RRF solves this elegantly by ignoring scores entirely and only looking at rank positions:

  RRF (Reciprocal Rank Fusion) WORKED EXAMPLE:

  Formula: RRF_score(doc) = Σ  1 / (k + rank_in_list)
           where k = 60 (a constant that prevents top ranks
           from being TOO dominant)

  KEYWORD search ranked:              VECTOR search ranked:
  ┌────┬──────────────────────┐      ┌────┬──────────────────────┐
  │ #1 │ HomeServe Pro         │      │ #1 │ HomeServe Pro         │
  │ #2 │ Bay Area Experts      │      │ #2 │ Mike's Kitchen Fitters│
  │ #3 │ QuickFix Larry        │      │ #3 │ Bay Area Experts      │
  │ #4 │ PremiumFix            │      │ #4 │ HandyHub              │
  │ #5 │ BudgetInstall         │      │ #5 │ InstallRight          │
  └────┴──────────────────────┘      └────┴──────────────────────┘

  Now calculate RRF score for each:

  HomeServe Pro:
    Keyword rank = 1  →  1/(60+1) = 0.01639
    Vector rank = 1   →  1/(60+1) = 0.01639
    RRF = 0.01639 + 0.01639 = 0.03279  🏆 (appears in both!)

  Bay Area Experts:
    Keyword rank = 2  →  1/(60+2) = 0.01613
    Vector rank = 3   →  1/(60+3) = 0.01587
    RRF = 0.01613 + 0.01587 = 0.03200  🥈

  Mike's Kitchen Fitters:
    Keyword rank = ∞   →  0 (not in keyword results at all!)
    Vector rank = 2    →  1/(60+2) = 0.01613
    RRF = 0 + 0.01613 = 0.01613        🥉

    ↑ KEY: Mike's was INVISIBLE to keyword search but vector
    search found them at rank 2. RRF gives them credit for
    that. Without hybrid search, they'd be missed entirely.

  QuickFix Larry:
    Keyword rank = 3  →  1/(60+3) = 0.01587
    Vector rank = ∞   →  0 (not in vector results)
    RRF = 0.01587 + 0 = 0.01587

  FINAL MERGED RANKING (by RRF score):
  1. HomeServe Pro      0.03279  ← Both searches agreed
  2. Bay Area Experts   0.03200  ← Both searches agreed
  3. Mike's Fitters     0.01613  ← Only vector search found
  4. QuickFix Larry     0.01587  ← Only keyword search found
  ...and so on for all ~143 unique results

Why k=60? This is a tunable constant. A higher k makes all positions more equal (rank 1 vs rank 10 matters less). k=60 is the standard value from the original RRF paper and works well in practice. It prevents the #1 result from having an overwhelmingly dominant score while still rewarding higher rankings.

Re-Ranking — The Final Quality Check

After RRF gives us a merged list, we run one more model called a cross-encoder re-ranker. Why? Because both keyword search and vector search have a limitation: they compare the query against each ad independently. The re-ranker compares them together — it reads the full conversation context AND the full ad description side-by-side and asks: "How relevant is this ad to what this user actually needs?"

  RE-RANKER EXAMPLE:

  The re-ranker (a specialized model, NOT a full LLM) takes pairs:

  Pair 1: (conversation, ad)
  ┌──────────────────────────────────────────────────────────┐

  │  installation starting at $159. Serving Livermore,        │
  │  Dublin, Pleasanton. Same-day available."                │
  │                                                          │
  │  Relevance score: 0.96 ← Extremely relevant!             │
  │  (Right service, right area, under budget, available)    │
  └──────────────────────────────────────────────────────────┘

  Pair 2: (conversation, ad)
  ┌──────────────────────────────────────────────────────────┐
  │  CONVERSATION: (same as above)                           │
  │                                                          │
  │  AD: "Discount Dishwasher Parts — Bosch compatible       │
  │  racks, filters, and accessories. Free shipping!"        │

It was trained on millions of (query, document, relevant/not-relevant) triplets. Think of it as a very sophisticated relevance detector — it reads two pieces of text and outputs a single number: how well do they match?

The Parallel Trick: Using "Wasted" Prefill Time

Here's the key insight that makes CAX possible at low latency. During the prefill phase (150-250ms), the LLM is "reading" the input and doing nothing visible to the user. The user is just waiting. That's 200ms of dead time.

Meanwhile, the ad pipeline (retrieval + auction) takes about 100ms total. So we start both at the same time:

  TIME →→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→

  ❌ NAIVE APPROACH (sequential — one after the other):

  0ms                     100ms                    300ms         1100ms
  │                         │                        │              │
  │◄── Ad Pipeline (100ms)─►│                        │              │

  │                         │                        │  LLM writes  │
  │                                                                 │

  0ms           100ms        200ms                              1000ms
  │               │            │                                   │
  │◄─ Ad Pipeline (100ms) ─►│  │                                   │
  │  Retrieval → Auction     │  │                                   │
  │                          │  │                                   │

  │  LLM reads system prompt │  │                                   │

  │                          │  ▼                                   │
  │                          │  INJECT ad winners into LLM context  │
  │                          │  (append to the KV cache, ~5ms)      │
  │                          │                                      │
  │                          │◄──────── Decode (800ms) ────────────►│
  │                          │  LLM generates response with ads     │
  │                                                                 │
  Total: max(100, 200) + 800 = 1,000ms to completion  ← 100ms SAVED!
  TTFT: 200ms + 5ms = 205ms  ← under 500ms target ✅

Why this works: The ad pipeline and the LLM prefill are completely independent operations that don't need each other's results. The ad pipeline searches a database and runs an auction. The LLM prefill reads the system prompt and conversation history. Neither depends on the other. They only need to come together at the end — when the ad winners are injected into the LLM's context right before it starts generating output.

But Wait — How Can the LLM Start Before We Know the Ads?

The LLM begins prefilling with everything it already knows: the system prompt ("You are Amazon's shopping assistant..."), the conversation history ("Turn 1: User asked about dishwashers..."), and the user's current message. This is ~80% of the input tokens.

When the ad auction finishes (~100ms), the winners are added as a small appendix to the context — maybe 50 extra tokens like "Sponsored result 1: HomeServe Pro, $159, 4.8★". The LLM processes this tiny addition in ~5ms.

The LLM doesn't start generating until it has the ad data. It just starts reading the parts it already knows. By the time it finishes reading, the ads are ready. This is the same principle as preheating an oven while you prepare the ingredients — the oven doesn't need to know what you're cooking to start heating up.

What "Prefill" Is NOT

Let's clear up a common confusion: "prefill" is NOT a name we invented. It's a real technical term used by every LLM serving framework (vLLM, TensorRT-LLM, etc.). It refers to the specific GPU computation of processing input tokens and building the KV (key-value) cache. Here's the terminology hierarchy:

  LLM INFERENCE = PREFILL + DECODE

  PREFILL (a.k.a. "prompt processing", "context encoding"):
  ├── Reads all input tokens in parallel (GPU-efficient)
  ├── Builds the KV cache (the model's "working memory")
  ├── Duration depends on input length (more tokens → longer)
  ├── Measured as "Time to First Token" (TTFT)
  └── This is a REAL term used by vLLM, NVIDIA TensorRT-LLM,
      AWS Neuron SDK, etc.

  DECODE (a.k.a. "autoregressive generation", "token generation"):
  ├── Generates tokens one at a time (sequential, GPU-limited)
  ├── Each token reads the KV cache from prefill
  ├── Duration depends on output length
  ├── Measured as "tokens per second" throughput
  └── This is also a REAL term.

  ════════════════════════════════════════════════════════════

  THINGS THAT ARE NOT LLM TERMS (just flow names we chose):
  ├── "Retrieval Agent" — our name for the RAG search step
  ├── "Auction Engine" — standard ad-tech term, not LLM-specific
  ├── "Response Generation Agent" — our name for the LLM call
  ├── "Guardrails Pipeline" — our name for safety checks
  └── "Temporal Decoupling" — our design pattern name for the
      parallel execution trick

Key Takeaway for Interviews

When asked "how do you handle the latency of adding ads to an LLM response?", the answer is: "We don't add latency — we hide it." 3c¾. How This ACTUALLY Works on AWS — The Real API Calls

The Honest Truth: You Can't Literally Pause Prefill

Let me be upfront about something. When I drew the diagram showing "LLM prefill runs in parallel with the auction, then we inject ads into the middle of prefill" — that's a conceptual simplification. In reality, you cannot call the Bedrock API, pause it mid-prefill, shove in extra tokens, and resume. The Bedrock API is a single request/response: you send a complete prompt, and you get back a stream of tokens. There's no "pause and inject" mechanism.

So how do we actually achieve the parallel effect? There are three real implementation approaches on AWS, from simplest to most advanced. Let me show each one with actual code.

Approach 1: The Practical Approach — Parallel Prep + Single Call

Difficulty: Easy. This is what you'd build first.

The insight: the auction (100ms) and the prompt construction (fetching conversation history, building system prompt — ~20ms) can run in parallel. Then you make ONE Bedrock call with the complete prompt including ad results. There's no fancy mid-stream injection — you just overlap the preparation work.

  async def call_bedrock_streaming(messages):

      This is THE call that triggers prefill + decode.
      
      What happens inside Bedrock when we call this:
      1. Bedrock receives our full prompt (system + history + ads + query)
      2. PREFILL: The model reads all input tokens, builds KV cache (~200ms)
      3. DECODE: The model generates tokens one at a time, streaming each back
      
      We have NO control over steps 2 and 3 — Bedrock handles them
      internally. We just receive a stream of output tokens.
      """
      
      response = bedrock.converse_stream(
          modelId='amazon.nova-pro-v1:0',  # or anthropic.claude-3-sonnet
          messages=messages,
          system=[{
              'text': """You are Amazon's shopping assistant. 
              When presenting sponsored results, always prefix 
              with [Sponsored]. Present options fairly. Use only 
              the provided product data — never make up prices 
              or features."""
          }],
          inferenceConfig={
              'maxTokens': 512,
              'temperature': 0.3,   # low = more predictable
              'topP': 0.9
          }
      )

      # Stream tokens back to the user's WebSocket connection
      stream = response['stream']
      full_response = ""
      
      for event in stream:
          if 'contentBlockDelta' in event:
              token = event['contentBlockDelta']['delta']['text']
              full_response += token
              
              # Send each token to the user immediately
              await websocket.send(json.dumps({
                  'type': 'token',
                  'content': token
              }))
          
          elif 'messageStop' in event:
              # Generation complete
              break
      
      return full_response

What's Really Happening Inside Bedrock

When you call converse_stream(), you hand Bedrock a complete prompt and get back a stream. You have zero control over what happens between sending the request and receiving the first token. Bedrock internally does:

1. Receives your full messages array over HTTPS

2. Tokenizes the text (converts words to numbers the model understands)

3. Prefill (processes all input tokens through the transformer layers, builds KV cache)

4. Decode (generates tokens one by one, streaming each back via HTTP chunked transfer)

Steps 2–3 are the "black box" you can't control. The TTFT you measure (time from API call to first token received) includes network latency + tokenization + prefill. You can't split them apart or inject anything mid-flight.

Approach 2: Prompt Caching — The Real Parallel Trick

Difficulty: Medium. This is where the latency savings actually come from.

Here's the approach that actually delivers the "parallel prefill" effect. Bedrock Prompt Caching (and SageMaker vLLM prefix caching) store the KV cache from previous calls. If your next call starts with the same prefix, the prefill for that prefix is skipped entirely — it's already computed and cached in GPU memory.

  Then decodes:                      ░░░░░░░░░░░░░░░░░░░  800ms

  Every call re-reads the system prompt and entire history!

  ═══ WITH PROMPT CACHING: ═══════════════════════════════════

  Call 1 (Turn 1): "System prompt + history + ads + query"
                    ├── cached prefix ──┤├─ new ─┤
  Bedrock prefills ALL tokens: ████████████████████  200ms
  Then decodes:                ░░░░░░░░░░░░░░░░░░░  800ms

Here's how you enable prompt caching in the Bedrock API:

  # Bedrock Prompt Caching — actual API usage

  async def call_bedrock_with_caching(messages, session_id):
      """
      Bedrock prompt caching works by marking which parts of
      the prompt should be cached with a 'cachePoint' block.
      
      Everything BEFORE the cachePoint is cached between calls.
      Everything AFTER it is processed fresh each time.
      """
      
      # The system prompt is identical for every call — perfect
      # for caching. We add a cachePoint after it.
      system_blocks = [
          {
              'text': """You are Amazon's shopping assistant.
              Rules: always label sponsored content, present
              options fairly, use only provided data..."""
          },
          {'cachePoint': {'type': 'default'}}  # ← CACHE MARKER
          #
          # Everything above this line gets cached in GPU memory.
          # On subsequent calls with the same prefix, Bedrock
          # skips the prefill for these tokens entirely.
      ]
      
      # For even more caching, we can also cache the conversation
      # history. The pattern is: put a cachePoint after the last
      # message that won't change.
      
      messages_with_cache = []
      
      # Add all previous turns (these won't change)
      for turn in conversation_history[:-1]:
          messages_with_cache.append(turn)
      
      # Add a cache point after the stable history
      messages_with_cache.append({
          'role': 'user',
          'content': [
              {'text': conversation_history[-2]['content']},
              {'cachePoint': {'type': 'default'}}  # ← CACHE HERE
          ]
      })
      
      # Add the NEW content (ads + current message) — NOT cached
      # This is the only part that needs fresh prefill
      messages_with_cache.append({
          'role': 'user',
          'content': [{'text': augmented_message_with_ads}]
      })
      
      response = bedrock.converse_stream(
          modelId='amazon.nova-pro-v1:0',
          messages=messages_with_cache,
          system=system_blocks,
          inferenceConfig={
              'maxTokens': 512,
              'temperature': 0.3
          }
      )
      
      # Check cache performance in the response metadata
      for event in response['stream']:
          if 'metadata' in event:
              usage = event['metadata'].get('usage', {})
              cache_read = usage.get('cacheReadInputTokens', 0)
              cache_write = usage.get('cacheWriteInputTokens', 0)
              fresh = usage.get('inputTokens', 0)
              print(f"Cache hit: {cache_read} tokens (FREE prefill)")
              print(f"Cache write: {cache_write} tokens (cached for next time)")
              print(f"Fresh prefill: {fresh} tokens (only these cost time)")
              #
              # Example output:
              # Cache hit: 1,200 tokens (FREE prefill)  ← system + history
              # Cache write: 0 tokens
              # Fresh prefill: 150 tokens (only these cost time) ← ads + msg
              #
              # Only 150 tokens needed fresh prefill instead of 1,350!
              # That's ~30ms instead of ~200ms.
      
      # ... stream tokens back to user as before
What Bedrock Prompt Caching Actually Does Inside the GPU

During prefill, the LLM processes each input token through ~80 transformer layers, computing "attention" between every pair of tokens. The result is a KV (Key-Value) cache — a large matrix stored in GPU memory that the model references during decode to "remember" the input.

Without caching: every API call recomputes this entire KV cache from scratch. For a 1,200-token prompt, that's 1,200 × 80 layers of matrix multiplication. Takes ~200ms.

With caching: Bedrock keeps the KV cache in GPU memory between calls (for a time window, typically 5 minutes). If your next call starts with the same 1,200-token prefix, it skips recomputation and just loads the cached KV values. Only the NEW tokens (the ad results + current message, ~150 tokens) need fresh computation. That's ~30ms.

Cost implication: Cached input tokens are billed at a 90% discount on Bedrock (you pay 10% of normal input token price). So caching saves both latency AND money.

Approach 3: SageMaker + vLLM — Full Control

Difficulty: Hard. Maximum flexibility, maximum operational burden.

If you need even more control (which Amazon internally likely has), you can host the model yourself on SageMaker with vLLM. This gives you access to prefix caching, chunked prefill, and direct control over the inference pipeline.

  SAGEMAKER + vLLM APPROACH:

  vLLM (the open-source LLM serving engine) has features that
  Bedrock doesn't expose:

  1. PREFIX CACHING (automatic):
     vLLM automatically detects shared prefixes across requests
     and caches their KV values. No explicit cachePoint needed.
     
     Request A: "System prompt + history + ads_v1 + query_A"
     Request B: "System prompt + history + ads_v2 + query_B"
     
     vLLM sees that "System prompt + history" is shared and
     reuses the KV cache. Only "ads_v2 + query_B" gets fresh
     prefill.

  2. CHUNKED PREFILL:
     vLLM can process prefill in chunks, interleaving it with
     decode operations from OTHER requests. This means the
     GPU stays busy even during long prefills.
     
     This doesn't help OUR latency, but it helps THROUGHPUT
     across all concurrent users.

  3. CONTINUOUS BATCHING:
     Multiple user requests are batched together dynamically.
     While user A is in prefill, user B might be in decode.
     The GPU processes both simultaneously.

The Interview Answer

When asked "how does the parallel prefill + auction work in practice?", the honest answer is:

"In the conceptual architecture, we describe running the LLM prefill in parallel with the auction. The practical implementation uses Bedrock Prompt Caching. The system prompt and conversation history — which make up ~85% of the input tokens — are cached in GPU memory between turns. This means prefill for those tokens is essentially zero. When the auction completes (~100ms), we append the winning ads as the only new tokens, and call Bedrock. The fresh prefill for just those ~150 new tokens takes ~30ms. So the effective TTFT from the user's perspective is auction time (100ms) + fresh prefill (30ms) + network overhead (5ms) = ~135ms on Turn 2+."

"On Turn 1 there's no cache, so TTFT is higher (~300ms). But Turn 1 of a conversation often doesn't have ads anyway — the system is still understanding intent. By Turn 2 when ads appear, the cache is warm."

First, What is an "Ad Server" in Plain English?

Think of it like this: when you search Google for "plumber near me," the results page shows two kinds of results. Organic results are free — Google shows them because they're relevant. Sponsored results (the ones marked "Ad" at the top) are there because a plumber paid to appear when someone searches for that term. The plumber set a budget ("I'll spend up to $50/day") and a bid ("I'll pay up to $3 every time someone clicks my ad"). Google runs a lightning-fast auction — every single time you search — to decide which plumber's ad wins the top spot.

An "ad server" is just the system that runs this process: it receives the search, finds relevant advertisers who want to show up, runs the auction, picks the winners, and inserts their ads into the results. Amazon does the exact same thing for its Sponsored Products — when you search "dishwasher" on Amazon, the first few results marked "Sponsored" are there because brands paid to be shown.

Now, CAX changes the game: instead of showing ads as product tiles on a search results page, the ads appear inside a conversation. The AI assistant recommends installers, compares their prices, and even helps negotiate — and some of those recommendations are paid placements by the installers. That's the product this team is building.

Deep Dive: Agent Orchestrator Options

For the full orchestrator architecture diagram and execution flow, see Section 3d. This section covers implementation trade-offs.

Why Supervisor, Not Independent Agents

Google DeepMind's research (December 2025) showed independent multi-agent systems amplify errors 17.2× while centralized supervisor architectures contain amplification to 4.4×. For ad systems where a single bad response can erode trust, the supervisor pattern is the only viable choice. The orchestrator uses a plan-then-execute model: an LLM call generates an execution plan, then Step Functions Express Workflows execute it with deterministic parallelism.

Implementation Options

Option A: Bedrock Agents (Managed)

Pros: Built-in action groups, knowledge base integration, session management, trace logging. Cons: Limited control over execution order, harder to implement the parallel retrieval+prefill pattern, agent invocation adds ~100ms overhead. Best for: prototyping and simpler orchestration flows.

Option B: Custom Orchestrator on ECS + Step Functions Express

Pros: Full control over parallel execution, sub-millisecond state transitions, custom circuit breakers, direct Bedrock API calls without agent overhead. Cons: More operational burden, must build session/trace management. Best for: production at Amazon scale where latency control is paramount. Step Functions Express Workflows support 100K+ state transitions/second with <50ms overhead per transition.

Circuit Breaker Policy

If any agent fails or times out (timeout budget: Intent 100ms, Retrieval 150ms, Auction 20ms, Generation 3s), the supervisor falls back to an organic (non-ad) response. The user never sees an error. Failed ad opportunities are logged to EventBridge for budget reconciliation. The circuit opens after 5 consecutive failures per agent type, with a 30-second half-open probe.

Deep Dive: Conversational Auction Signals

For the full auction pipeline (4 stages, scoring math, second-price settlement), see Section 3b. This section covers what makes conversational auctions different from traditional search auctions.

Why Enhanced Second-Price

Amazon's existing Sponsored Products auction uses second-price mechanics, and CAX should inherit this for two reasons: (1) truthful bidding — advertisers bid their true value since they pay less if they win, reducing strategic complexity; (2) ecosystem consistency — advertisers already understand the mechanism. The "enhanced" part is that Ad Rank incorporates quality and relevance, not just bid amount.

Conversation-Specific Scoring Signals

Unlike traditional search auctions, CAX scores include: conversation turn count (deeper in conversation = higher purchase intent), expressed budget (user said "$200"), comparison intent (user is actively comparing options), sentiment (frustrated users get no ads), and frequency cap (same ad shown 3x → suppress). These are computed by the Intent Agent and passed as features to the pCTR/pCVR models.

Ad Slot Design in Conversations

Unlike a search page with fixed ad positions, a conversational turn has variable ad slot count (0–3). The orchestrator determines slot count based on intent: a product search gets 2–3 ads, a comparison gets 1–2 (the compared options), an information query gets 0–1. Each slot has a position discount factor (position 2 = 0.8× CTR of position 1), applied to the pCTR estimate before ranking.

Regulatory Requirements

FTC: "Sponsored" or "Ad" label before ad content; clear visual differentiation; no exploitation of consumer-chatbot relationships. EU AI Act Article 50 (August 2026): AI-generated content must be clearly identifiable; penalties up to €35M or 7% of worldwide turnover. Bedrock Guardrails Configuration

Six policy types configured for CAX: Content Filters (hate, insults, sexual, violence — HIGH threshold for ad content), Denied Topics (competitor disparagement, regulated health/financial claims, political content), Word Filters (profanity + blocked brand terms), Sensitive Info Filters (PII detection and redaction), Contextual Grounding (verify claims against product data, threshold ≥ 0.85), Automated Reasoning (formal verification of factual claims, 99% accuracy).

Latency Impact

The ApplyGuardrail API can run in parallel with model inference via the independent API. Input guardrails add ~5ms. Output guardrails run on the complete response (~10ms) but can be pipelined with the last streaming chunk. Total guardrails overhead: ~15ms, well within budget.

Deep Dive: Budget Pacing

Budget decrement happens in Stage 4 of the Auction Pipeline (3b). This section explains how pacing distributes that spend across the day so advertisers don't exhaust their budgets in minutes.

Problem

An advertiser with a $1,000 daily budget could exhaust it in minutes during peak traffic without pacing. Pacing distributes spend across the day to maximize exposure.

Architecture

Redis cluster holds per-campaign budget counters (budget_remaining, daily_cap, hourly_target). A Lua script atomically decrements budget and checks against pacing targets. The pacing controller runs as a Lambda function triggered every 60 seconds that adjusts the probability-of-serving (throttle rate) per campaign based on: budget consumed vs. time elapsed, predicted remaining traffic (from historical patterns), and bid landscape changes.

Adaptive Waterlevel Pacing

Rather than even pacing (spend 1/24th per hour), waterlevel pacing spends proportionally to traffic volume. A campaign that gets 60% of its traffic between 6–10 PM allocates 60% of budget to that window. The "waterlevel" is the minimum effective bid that, if applied all day, would exactly exhaust the budget — campaigns bid normally when market prices are below the waterlevel, and throttle when above.

Consistency

Redis writeback to DynamoDB every 5 seconds. If Redis fails, DynamoDB is the source of truth (slightly stale but consistent). Cross-region campaigns use regional budget slicing: a $1,000 global budget is pre-allocated $600 US-EAST, $300 US-WEST, $100 EU-WEST based on historical traffic distribution, with a 10% reserve for rebalancing.

Deep Dive: Attribution & Analytics

Event Flow

Every ad interaction emits events to Kinesis Data Streams: impression (ad shown in response), engagement (user asks follow-up about ad), click (user navigates to product), and conversion (purchase within attribution window). Conversational attribution is harder than search attribution because a single session may expose the user to an ad across multiple turns. Kinesis → Firehose → S3 (raw events, Parquet format) → Redshift Serverless (aggregation, campaign reporting). QuickSight dashboards surface: CPC, CTR, CVR, ROAS, budget utilization, guardrail block rate, and conversation depth metrics. Latency from event to dashboard: <15 minutes.

Design Patterns to Know

Patterns that will surface in staff-level discussions, mapped to where they apply in CAX.

Agent & Orchestration Patterns

Supervisor Agent Pattern
A central LLM orchestrator plans and delegates to specialist sub-agents, consolidating results. Contains error amplification to 4.4× (vs 17.2× for independent agents). The orchestrator never generates end-user content — it only routes.
Used: Agent Orchestrator. Why: Latency control, error containment, clear separation of concerns.
Plan-Then-Execute
The supervisor generates a structured execution plan (which agents, what order, what parallelism) before invoking any agent. The plan is deterministic and cacheable. Separates reasoning from execution.
Used: Orchestrator planning phase. Why: Enables static analysis of execution plans for optimization and debugging.
Generator-Critic (Dual-LLM)
One LLM generates the response; a separate LLM (or guardrail system) validates it. The generator optimizes for helpfulness; the critic optimizes for compliance. Neither can override the other — both must pass.
Used: Response Generation + Guardrails Pipeline. Why: Ad content requires both quality AND compliance.

Data & Retrieval Patterns

Hybrid Search with RRF
Run keyword (BM25) and vector (HNSW) searches in parallel, merge results via Reciprocal Rank Fusion. No score normalization needed. The alpha (keyword vs. semantic weight) is tuned per query distribution.
Used: Ad Retrieval. Why: Captures both exact matches (brand names, ASINs) and semantic intent.
Semantic Caching with Skeleton Separation
Cache organic response skeletons separately from ad placements. On cache hit, re-run only the auction (fast) to fill ad slots with current winners. Preserves caching benefits while ensuring ad freshness.
Used: Semantic Cache. Why: Ad inventory changes faster than conversation patterns, requiring split invalidation.
CQRS (Command Query Responsibility Segregation)
Write path (budget decrement, impression logging) is separated from read path (campaign reporting, analytics). Writes go to Redis/DynamoDB; reads go to Redshift/QuickSight. Eventually consistent across paths.
Used: Budget Pacing + Attribution. Why: Write path needs sub-ms latency; read path needs complex aggregation.

Reliability Patterns

Circuit Breaker with Graceful Degradation
If the ad pipeline fails (retrieval timeout, auction error, guardrail block), fall back to an organic response. The user gets a helpful answer without ads rather than an error. Failed ad opportunities are logged for budget reconciliation.
Used: Every agent boundary. Why: Ad failure should never degrade user experience.
Parallel Prefill + Auction
Overlap the LLM's KV-cache prefill (loading system prompt + conversation history) with the ad retrieval and auction pipeline. Inject auction winners into context before the first output token. Hides ~100ms of auction latency behind ~200ms of prefill.
Used: Core pipeline. Why: Without this, adding ads would add 100ms+ to every response — unacceptable at scale.
Adaptive Waterlevel Budget Pacing
Instead of even spend over time, allocate budget proportional to predicted traffic volume. A "waterlevel" bid threshold ensures budget exhaustion exactly at day's end. Adjusts every 60 seconds based on actual vs. predicted spend.
Used: Budget Pacing. Why: Conversational traffic is burstier than search; even pacing wastes budget in low-traffic periods.

Auction & Economics Patterns

Enhanced Second-Price Auction
Winner pays $0.01 above the second-highest effective bid. AdRank = Bid × Relevance × Quality. High-relevance ads win at lower bids. Preserves truthful bidding incentive (bid your true value).
Used: Auction Engine. Why: Consistent with existing Amazon Ads mechanism; reduces advertiser strategic complexity.
Variable Slot Allocation
Unlike search (fixed N ad positions per page), conversational turns have dynamic slot counts (0–3) determined by intent type and conversation context. More commercial intent → more slots. Information-seeking → fewer or none.
Used: Orchestrator + Auction. Why: Jamming ads into every turn erodes trust (Perplexity's lesson).

Staff-Level Interview Q&A

Q: How do you ensure ads don't degrade the conversational experience?

Answer framework: Three mechanisms. (1) Intent-gated ad injection — only 60% of turns are ad-eligible; complaints, support, and chit-chat never show ads. (2) Variable slot allocation — commercial queries get 2–3 ads, information queries get 0–1, never forcing ads where they don't fit. (3) Trust metrics — track "trust erosion" via conversation abandonment rate after ad exposure; A/B test ad density with guardrails that automatically reduce frequency if abandonment exceeds baseline by >5%. Reference Perplexity's failure: they showed ads everywhere and lost users. Amazon's approach with Rufus is selective and contextual.

Q: Walk me through the auction for a conversational turn

Step-by-step: (1) Intent Agent classifies the turn as "product search" with entities {category: home_services, budget: $200, product: dishwasher_installation}. (2) Retrieval Agent rewrites the query, runs hybrid search, returns 50 candidates. (3) Auction Engine enriches candidates with campaign state from Redis (budget remaining, bid strategy), filters to 20 eligible candidates, scores each with AdRank = Bid × pCTR × pCVR × QualityFactor using conversation-specific features. (4) Sort by AdRank, select top-3 winners. (5) Second-price settlement: winner 1 pays (AdRank_2nd / QS_winner1) + $0.01. (6) Budget decremented atomically in Redis. (7) Winners injected into generation context with [Sponsored] labels. All of this happens in ~15ms p99, overlapped with LLM prefill.

Q: How do you handle the latency challenge of adding LLM-generated ads?

The parallel prefill + auction pattern is the key insight. While the LLM is loading its KV cache (system prompt + conversation history = ~200ms), the retrieval and auction pipeline runs in parallel (~100ms total). By the time the LLM is ready to generate output tokens, the auction has already completed and winners are injected into context. Net latency addition from ads: effectively zero in the common case. Additionally: speculative decoding (2–3× throughput), semantic caching (40% hit rate, zero LLM latency), and Inferentia2 for pCTR/pCVR models (sub-ms inference) keep total latency within the 2s budget.

Q: How would you design the system for multi-turn negotiation?

Negotiation state machine: The conversation state (DynamoDB) tracks a negotiation_state per advertiser: {initial_offer, counter_offers[], current_price, user_budget, max_discount_pct}. The advertiser pre-configures discount rules in their campaign: "allow up to 15% off if user expresses price sensitivity." The Generation Agent uses these rules as tool parameters — it can offer discounts within the authorized range. Each counter-offer triggers a budget re-check and a new attribution event. The Auction Engine re-runs on each negotiation turn because the effective bid changes (discounted price = lower margin = lower bid). This is analogous to Pactum AI's approach: 2,000+ concurrent negotiations, automated within policy bounds, human escalation for out-of-bounds requests.

Q: What are the failure modes and how do you handle them?

Five critical failure modes: (1) LLM timeout/error → Circuit breaker, serve a templated response with no ads. (2) Auction engine failure → Serve organic response (no ads), log missed revenue opportunity. (3) Guardrail false positive → Block response, re-generate without ads. Track false positive rate; if >2%, alert for guardrail tuning. (4) Budget data inconsistency → Redis is authoritative for real-time; DynamoDB reconciles every 5s. Overspend tolerance: 0.1% of daily budget. (5) Catalog stale data → LLM makes claims about a product that's changed. Mitigation: Contextual Grounding Check in guardrails validates claims against latest catalog; TTL on cached product data is 5 minutes. In all cases: the user never sees an error, and ad failures are invisible.

Q: How would you measure success for this system?

Four metric categories: (1) User experience — conversation completion rate (target: no degradation vs. non-ad baseline), NPS for ad-containing conversations, trust score (survey-based). (2) Ad performance — conversational CTR (target: >3%, vs. ~1.5% traditional search ads), CPA, ROAS per campaign, ad fill rate. (3) System health — e2e latency p99 <2s, time-to-first-token p99 <500ms, guardrail block rate <5%, circuit breaker trigger rate <0.1%. (4) Business impact — incremental revenue per conversation, ad revenue as % of total Rufus-driven GMV, advertiser adoption rate, repeat advertiser spend. The north-star metric is incremental GMV per ad dollar spent — proving that conversational ads drive purchases that wouldn't have happened otherwise.
