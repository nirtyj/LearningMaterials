What you actually need for interviews

  • CPython is the default runtime. When you write Python on LeetCode, you're running CPython — a bytecode interpreter written in C.
  • Dynamic typing — type checking happens at runtime. There's no compile step that catches obj.foo() on a wrong-typed object; you find out when it runs.
  • Bytecode is cached in __pycache__/*.pyc. First import is slow (compile); subsequent imports skip compilation if the source hasn't changed.
  • The GIL (Global Interpreter Lock) — only one Python thread executes bytecode at a time. CPU-bound threading doesn't speed up pure-Python code. Multiprocessing or C extensions get around it. Rarely tested directly, but useful context.
  • Reference counting + cycle GC — objects are freed as soon as their refcount hits zero, with a periodic cycle detector for circular references.
  • PyPy is an alternate runtime with a JIT compiler — 5–10× faster on tight loops. Not available on LeetCode but worth knowing if your contest judge offers it.

Performance reality on LeetCode

  • A pure-Python solution is typically 3–10× slower than the same algorithm in a compiled language. LeetCode time limits account for this — your solution doesn't need to be fastest, just within Python's bucket.
  • When you see TLE on a correct algorithm: switch to built-in/C-backed operations (sum(), min(), sorted(), list slicing, collections.Counter) instead of hand-rolled loops. The hot loop is C, not Python.
  • Avoid repeated string concatenation in a loop (s += c) — use "".join(parts) instead. Strings are immutable, so += rebuilds the whole string each time.
